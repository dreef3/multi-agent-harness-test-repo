# Implementation Plan: Codebase Structure Analysis for E2E Test Repo

Repository: E2E Test Repo
Repository ID: 93a8bf56-e9c8-4b1e-8eb7-291ecbfe799f

Overview
- The implementation will be executed by sub-agents (or a CI job) that perform read-only inspection of the repository to produce a Markdown report and a JSON summary. Each task is self-contained and lists the exact files/commands to run.

Deliverables
- reports/codebase-structure-report.md (human-readable analysis)
- reports/codebase-structure-summary.json (machine-readable summary)

Tasks (task names suitable for dispatching)

1) Inspect top-level layout
- Purpose: list top-level files and directories, detect workspace/monorepo markers.
- Commands to run in repo root:
  - ls -la
  - tree -L 2 || find . -maxdepth 2 -type d -print
- Files to read (if present): README.md, CONTRIBUTING.md, LICENSE, CODE_OF_CONDUCT.md
- Expected outputs: top_level_dirs.txt

2) Detect languages and package managers
- Purpose: find manifest files and common package manager lockfiles.
- Files/paths to check (read if present):
  - package.json, package-lock.json, yarn.lock, pnpm-lock.yaml, .npmrc
  - tsconfig.json, babel.config.js
  - pyproject.toml, setup.py, requirements.txt, Pipfile
  - go.mod, go.work
  - Cargo.toml
  - Gemfile, Gemfile.lock
  - composer.json
  - build.gradle, settings.gradle, gradlew
  - mix.exs
  - Makefile
- Commands:
  - for f in package.json pyproject.toml go.mod Cargo.toml Gemfile requirements.txt; do [ -f "$f" ] && echo "FOUND:$f"; done
  - rg "(workspaces|lerna|workspace)" package.json || true
- Expected outputs: manifests_found.json

3) Read CI / workflow config
- Purpose: identify CI providers and important workflows (build/test/publish).
- Paths to read (if present):
  - .github/workflows/*.yml
  - .gitlab-ci.yml
  - .circleci/config.yml
  - Jenkinsfile
  - azure-pipelines.yml
- Expected outputs: ci_files_list.txt and quick summary of build/test steps extracted

4) Locate Docker and container config
- Purpose: find Dockerfiles and container orchestration manifests.
- Files to check: Dockerfile, **/Dockerfile, docker-compose.yml, k8s/ or manifests/
- Commands:
  - rg "^FROM" -n --hidden --glob '!**/node_modules' || true
- Expected outputs: dockerfiles_list.txt

5) Find tests and E2E suites
- Purpose: identify directories and frameworks used for tests.
- Typical locations: tests/, test/, e2e/, spec/, __tests__/, cypress/, integration/
- Commands:
  - rg "describe\(|it\(|cy\.|pytest|unittest|mocha|jest" --hidden --glob '!**/node_modules' -S || true
- Expected outputs: tests_overview.txt

6) Dependency and script extraction
- Purpose: extract dependency lists and project scripts (npm/yarn scripts, Make targets).
- Actions:
  - Parse package.json (if exists) and extract name, version, scripts, dependencies, devDependencies, workspaces.
  - Parse pyproject.toml for [tool.poetry] or [project] metadata.
  - Parse go.mod for module name and require.
- Expected outputs: parsed_manifests/ directory with small JSON files per manifest.

7) Synthesize report and JSON summary
- Inputs: outputs from tasks 1–6.
- Create reports/codebase-structure-report.md with sections:
  - Executive summary (languages, package managers, CI, monorepo vs single package)
  - Top-level layout (directories and purpose)
  - Entry points and scripts
  - Tests and CI
  - Docker/containers
  - Recommended next steps (run tests, run linters, dependency scan, security checks)
- Create reports/codebase-structure-summary.json with fields:
  - repo_id
  - top_level_dirs: []
  - languages: []
  - package_managers: []
  - manifests: {path: type}
  - ci_files: []
  - dockerfiles: []
  - test_dirs: []
  - recommended_actions: []

8) Commit report files
- Path: reports/
- Files: codebase-structure-report.md, codebase-structure-summary.json
- Commit message suggestion: "chore(reports): add codebase structure analysis"
- Note: The harness handles PR creation; the task should create a branch and commit the files.

Exact file reads and command snippets (for sub-agent)
- Read: README.md
- Read: package.json (if exists) and run: jq -r '.name,.version' package.json
- Read: pyproject.toml and parse relevant keys
- Read .github/workflows/*.yml files
- Command set:
  - echo "--- top-level ---"; ls -la
  - echo "--- find manifests ---"; for f in package.json pyproject.toml go.mod Cargo.toml Gemfile requirements.txt; do if [ -f "$f" ]; then echo "$f"; fi; done
  - echo "--- find ci ---"; ls -la .github/workflows || true; [ -f .gitlab-ci.yml ] && echo .gitlab-ci.yml
  - rg "describe\(|it\(|pytest|unittest|cy\.|cypress|mocha|jest" --hidden --glob '!**/node_modules' -S || true

Assumptions, risks, and mitigations
- Assumption: repository is reasonable size (not >5GB of node_modules etc.). Use --hidden and ignore node_modules in searches.
- Risk: private submodules or git-lfs large files; mitigation: detect submodule entries in .gitmodules and flag them in the report.

Estimated effort
- Automated sub-agent run: ~5–15 minutes depending on repository size.
- Human review of report: 15–30 minutes.

Acceptance criteria
- reports/codebase-structure-report.md exists and contains the sections listed above.
- reports/codebase-structure-summary.json exists and contains non-empty arrays for top_level_dirs and manifests (or explains why none found).
