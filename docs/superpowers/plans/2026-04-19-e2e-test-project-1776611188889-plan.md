Implementation Plan: Repository Structure Analysis (task-by-task)

Overview
This plan splits the analysis into discrete tasks. Each task is self-contained: it runs a small number of safe commands, writes outputs to the reports directory, and includes expected output examples. Tasks should be run in the repository root. All produced files are placed under reports/codebase-analysis/ with consistent filenames.

Top-level output directory
- reports/codebase-analysis/
  - summary.md           (human-readable summary)
  - structure.json       (file tree + counts)
  - languages.json       (language and manifest detection)
  - dependencies.json    (parsed dependency manifests)
  - ci-summary.md        (CI/workflow summary)
  - tests-discovery.txt  (discovered test commands and test folders)
  - findings.md          (red flags and recommendations)
  - raw-outputs/         (raw command outputs for debugging)

Task 1 — Inventory files
- Purpose: Create an index of top-level files and directories to guide deeper checks.
- Commands (bash):
  - mkdir -p reports/codebase-analysis/raw-outputs
  - ls -la > reports/codebase-analysis/raw-outputs/ls-root.txt
  - find . -maxdepth 2 -mindepth 1 -printf "%P\n" | sort > reports/codebase-analysis/raw-outputs/top-level-list.txt
  - find . -type f -not -path './.git/*' -printf "%P\n" | sed -n '1,10000p' > reports/codebase-analysis/raw-outputs/all-files.txt
- Expected: list of top-level manifest files (README.md, package.json, pyproject.toml, go.mod, etc.).

Task 2 — Language & manifest detection
- Purpose: Identify languages and framework indicators from manifests.
- Files to check (if present): package.json, yarn.lock, package-lock.json, pyproject.toml, requirements.txt, setup.py, Pipfile, go.mod, Cargo.toml, composer.json, Gemfile, mix.exs, build.gradle, pom.xml, Cargo.lock
- Commands (bash):
  - for f in package.json pyproject.toml requirements.txt setup.py Pipfile go.mod Cargo.toml composer.json Gemfile build.gradle pom.xml; do [ -f "$f" ] && echo "$f: FOUND" >> reports/codebase-analysis/raw-outputs/manifests-found.txt || true; done
  - (optional) jq -r '.name, .version, .scripts' package.json > reports/codebase-analysis/raw-outputs/package-json-summary.txt 2>/dev/null || true
  - python - <<'PY'
import toml, json, sys
try:
  with open('pyproject.toml') as f:
    pj=toml.load(f)
    print(json.dumps(pj.get('tool',{})))
except Exception:
  pass
PY
- Outcome: languages.json with entries like {"node": {manifest: "package.json", main_scripts: [...]} }

Task 3 — File-tree and size summary
- Purpose: Provide structure.json with counts and top directories.
- Commands (bash):
  - find . -type d -maxdepth 3 -printf '%P\n' | sed '/^$/d' | sort > reports/codebase-analysis/raw-outputs/dirs-depth3.txt
  - find . -maxdepth 3 -type f -printf '%P\t%s\n' | sort > reports/codebase-analysis/raw-outputs/files-depth3.txt
  - Create a small JSON summarizing counts (use jq or a short Python script)
- Example JSON schema:
  {"total_files": 123, "total_dirs": 23, "top_level": ["src","tests",".github"], "largest_files": [{"path":"path","size":12345}]}

Task 4 — Test discovery
- Purpose: Detect test directories and suggested commands to run tests (do NOT run tests).
- Patterns: tests/, test/, __tests__/, e2e/, spec/, jest.config.js, pytest.ini, tox.ini, mocha, jest in package.json scripts
- Commands (bash):
  - rg -n "\b(test|pytest|jest|mocha|ava|nose|unittest)\b" -S --hidden --glob '!node_modules' > reports/codebase-analysis/raw-outputs/test-grep.txt || true
  - jq -r '.scripts' package.json > reports/codebase-analysis/raw-outputs/package-scripts.txt 2>/dev/null || true
- Output: tests-discovery.txt listing detected test folders and suggested commands like: npm test, pytest, go test ./...

Task 5 — CI and workflow analysis
- Purpose: Summarize CI pipelines and what they run.
- Paths to inspect: .github/workflows/*.yml, .gitlab-ci.yml, .circleci/config.yml, Jenkinsfile, Dockerfile*
- Commands (bash):
  - if [ -d .github/workflows ]; then for f in .github/workflows/*.yml; do echo "== $f =="; sed -n '1,200p' "$f"; done > reports/codebase-analysis/raw-outputs/github-workflows.txt; fi
  - [ -f .gitlab-ci.yml ] && cp .gitlab-ci.yml reports/codebase-analysis/raw-outputs/
  - rg -n "runs-on|uses:|docker|script:" .github/workflows || true
- Output: ci-summary.md with summary per workflow (jobs, triggers, environments, uses of matrix builds, caching, artifact uploads). Also list secrets referenced (names only) and external actions.

Task 6 — Docker and containerization
- Purpose: Extract Dockerfiles and infer base images and build steps.
- Commands:
  - rg -n "^FROM\b" --hidden --no-ignore-vcs > reports/codebase-analysis/raw-outputs/docker-froms.txt || true
  - cp $(rg -l "^FROM\b" --hidden --no-ignore-vcs || true) reports/codebase-analysis/raw-outputs/ 2>/dev/null || true
- Output: In findings.md note base images and whether multi-stage builds exist.

Task 7 — Dependency parsing and security hints
- Purpose: Emit dependencies.json with top-level dependencies (names and pinned versions if available). Flag major/critical version ranges where possible.
- Commands/examples:
  - Node: jq -r '.dependencies,.devDependencies' package.json > reports/codebase-analysis/raw-outputs/node-deps.txt
  - Python: parse requirements.txt and pyproject.toml
  - Go: cat go.mod
- Notes: Do not run remote vulnerability checks automatically in this analysis step; instead recommend commands (npm audit, pip-audit, go list -m -u all, cargo audit, trivy image) in findings.md.

Task 8 — Static analysis / lint config discovery
- Purpose: Find linter configs and recommended tools.
- Files: .eslintrc*, .prettierrc, pylint, flake8, golangci.yml, .rubocop.yml
- Commands: rg -n "eslint|prettier|flake8|pylint|golangci|rubocop" --hidden || true
- Output: findings.md entries mapping linter configs to recommended commands.

Task 9 — Sensitive files scan (non-exfiltrating)
- Purpose: Detect presence of probable secrets or misconfigured credentials by filename and pattern, without printing contents.
- Checks: filenames like *.pem, *.key, secret.env, .env, AWS credentials in .aws/ , or GitHub token names in workflows.
- Commands: rg -n "(?i)password|secret|token|aws|private_key|BEGIN RSA PRIVATE KEY" --hidden -S --max-filesize 1M --no-line-number || true
- Output: findings.md with path list and recommended remediation steps. Do NOT include file contents.

Task 10 — Summarize and produce final artifacts
- Compose reports/codebase-analysis/summary.md with:
  - One-paragraph summary (languages, primary structure, CI present/absent, tests present/absent)
  - Bullet list of top-level directories and their purpose
  - Actionable recommendations (tests to run, missing CI, security scan suggestions)
- Compose structure.json and dependencies.json as described earlier.

Example commands to create outputs (one-liner sequence)
- mkdir -p reports/codebase-analysis/raw-outputs
- (run tasks 1-9 commands)
- python3 tools/produce_json_summary.py --input reports/codebase-analysis/raw-outputs --out reports/codebase-analysis/structure.json

Deliverables
- The files listed under Top-level output directory, plus raw-outputs for traceability.
- A PR-ready branch (the harness will handle PR creation) with the reports added.

Time estimates
- Inventory and manifest detection: 5–10 minutes
- Test & CI analysis: 10–20 minutes depending on workflow complexity
- Dependency parsing and producing reports: 10–15 minutes
- Total: ~30–45 minutes for a single-run automated analysis

Permissions & Safety
- This plan performs read-only operations. It will not execute test suites or build images. The sensitive-file scan reports filenames/paths only and avoids content exposure.

Notes for implementer
- Use small, robust parsing scripts (Python + toml + json packages) to turn manifest contents into structured JSON.
- Normalize paths to be relative to repo root.
- Keep raw command outputs to help debugging if follow-up questions arise.
