Implementation plan: Task-by-task plan to analyze repository structure

Overview
- This plan breaks the work into small, self-contained tasks that can be assigned to agents. Each task only requires reading files and running shell commands (no builds or tests). Each task will produce partial outputs which are then merged into the final deliverables.

Output paths (to be created in the repo):
- docs/codebase-structure-analysis.md (final human-readable report)
- reports/codebase-structure.json (structured machine-readable summary)
- reports/raw-file-list.txt (optional detailed listing)

Task 1 — Inventory top-level files and directories
- Purpose: gather the top-level tree and identify obvious manifests.
- Commands to run:
  - ls -la
  - find . -maxdepth 2 -mindepth 1 -type d -print | sort
  - find . -maxdepth 2 -mindepth 1 -type f -print | sort
- Files to read: README.md, CONTRIBUTING.md, LICENSE
- Expected artifacts:
  - reports/raw-file-list.txt (write command output)
  - Partial JSON entry: top_level: [{name: <path>, type: dir|file}]
- Notes: If README missing, note it.

Task 2 — Detect languages and package manifests
- Purpose: determine languages, package managers and lockfiles.
- Files to check (in order): package.json, yarn.lock, pnpm-lock.yaml, package-lock.json, pyproject.toml, requirements.txt, setup.py, Pipfile.lock, go.mod, Cargo.toml, pom.xml, build.gradle, Gemfile, composer.json, mix.exs
- Commands:
  - for f in package.json pyproject.toml go.mod Cargo.toml pom.xml build.gradle Gemfile composer.json setup.py requirements.txt; do [ -f "$f" ] && echo "$f"; done
  - rg --hidden "^\s*name\s*[:=]" -n package.json pyproject.toml || true
- Expected artifacts: manifests list in reports/codebase-structure.json under manifests and languages detection logic mapping manifest->language.

Task 3 — Identify CI, workflows, and automation
- Purpose: find CI configs and Docker-related files.
- Paths to read: .github/workflows/*.yml, .gitlab-ci.yml, circleci/config.yml, Dockerfile, docker-compose.yml, .dockerignore, Makefile
- Commands:
  - ls -la .github/workflows || true
  - rg "runs-on|docker|services|image:|uses:" .github -S || true
- Expected artifacts: ci list in JSON, copy of detected workflows embedded or summarized in analysis.

Task 4 — Source layout and entry points
- Purpose: map where source code and binaries live.
- Look for directories: src, lib, cmd, pkg, internal, app, server, web, e2e, tests, test
- Commands:
  - find src cmd lib pkg -maxdepth 2 -type f -name "*.js" -o -name "*.ts" -o -name "*.py" -o -name "*.go" -o -name "*.rs" | sed -n '1,200p'
- Read package.json main/scripts, pyproject[tool.poetry.scripts], go.mod module plus cmd/*
- Expected artifacts: entry_points array for JSON and analysis statements about binary vs library.

Task 5 — Tests and E2E locations
- Purpose: find unit tests, e2e tests, harness code.
- Paths to check: tests/, e2e/, integration_tests/, specs/, cypress/, playwright/, nightwatch/
- Commands:
  - rg "describe\(|it\(|test\(" --hidden -g '!node_modules' || true
  - find . -type d -name test* -o -name e2e -maxdepth 4
- Expected artifacts: tests list in JSON with counts by test framework where detectable.

Task 6 — Readme and docs analysis
- Purpose: extract setup / dev run instructions and note mismatches.
- Files: README.md, docs/*
- Commands:
  - sed -n '1,200p' README.md || true
  - rg "install|run|test|build|docker" README.md docs -S || true
- Expected artifacts: extract sample commands and note whether they reference files that exist.

Task 7 — Heuristics for technical debt and hotspots
- Purpose: surface files that may indicate maintenance issues.
- Heuristics to apply:
  - Very large files (>200KB)
  - Many deprecated config files (both package-lock.json & yarn.lock & pnpm-lock)
  - Missing CI workflows
  - Presence of compiled or vendored files in repo (node_modules, vendor)
  - Outdated language versions referenced in manifest
- Commands:
  - find . -type f -size +200k -print | sed -n '1,200p'
  - rg "node_modules|vendor|venv|__pycache__" -S || true
- Expected artifacts: list of hotspots in analysis and JSON.

Task 8 — Produce final artifacts
- Merge collected JSON fragments into reports/codebase-structure.json with schema:
  {
    "top_level": [...],
    "languages": [...],
    "manifests": [...],
    "ci": [...],
    "tests": [...],
    "dockerfiles": [...],
    "notable_files": [...]
  }
- Create docs/codebase-structure-analysis.md containing:
  - Summary (1–2 paragraphs)
  - Top-level inventory
  - Detected languages and manifests
  - CI and Docker overview
  - Source layout and entry points
  - Tests and e2e locations
  - Technical debt hotspots and recommendations (3–6 action items)

Task responsibilities (for sub-agents)
- Agent A: Run Task 1 and Task 2, produce partial JSON and raw-file-list.
- Agent B: Run Task 3 and Task 4, produce CI and entry_points fragments.
- Agent C: Run Task 5 and Task 6, produce tests fragment and README excerpt.
- Agent D: Run Task 7 and Task 8, merge fragments and produce final docs and JSON.

Exact file paths to read (examples to include in each task):
- README.md
- LICENSE
- package.json, package-lock.json, yarn.lock, pnpm-lock.yaml
- pyproject.toml, requirements.txt, setup.py
- go.mod
- Cargo.toml
- .github/workflows/*.yml
- Dockerfile, docker-compose.yml
- Makefile
- src/, lib/, cmd/, pkg/, tests/, e2e/

Example commands and parsing snippets (for implementers)
- List top-level dirs: find . -maxdepth 1 -mindepth 1 -printf "%P\n" | sort
- Detect manifests: for m in package.json pyproject.toml go.mod Cargo.toml pom.xml; do [ -f "$m" ] && jq -r .name "$m" 2>/dev/null || echo "$m present"; done
- Extract package.json main/scripts: jq '{name:.name, main:.main, scripts:.scripts}' package.json

Acceptance criteria
- docs/codebase-structure-analysis.md present and reviewed (contains all sections above)
- reports/codebase-structure.json present and valid JSON with required keys
- raw-file-list.txt present for auditing

Estimated effort
- Single agent running the commands: ~30–60 minutes depending on repo size.

Notes
- The plan assumes the repository is accessible to the agents that will run the commands. No modifications to the code are planned.
