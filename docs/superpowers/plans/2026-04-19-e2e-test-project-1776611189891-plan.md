Implementation plan: Task-by-task breakdown to analyze codebase structure

Overview
- The implementation will be a sequence of small, independent tasks that can be executed in CI or locally. Each task produces an artifact saved under analysis-report/ in the repository (or a specified output directory). The tasks are ordered so earlier outputs feed into later steps.

Artifacts
- analysis-report/SUMMARY.md
- analysis-report/analysis.json
- analysis-report/file-tree.txt
- analysis-report/dependency-list.txt
- analysis-report/ci-files.txt
- analysis-report/lang-stats.txt

Tasks

Task 1 — Top-level file listing
- Purpose: capture the root layout and identify candidates for further inspection.
- Command(s):
  - ls -la
  - find . -maxdepth 2 -mindepth 1 -type d -print | sort
  - find . -maxdepth 2 -type f -print | sort
- Output files: analysis-report/file-tree.txt (trimmed tree for depth 3)
- Notes: Save the full listing and a trimmed tree (tree -L 3 equivalent).

Task 2 — Language and size breakdown
- Purpose: identify primary languages and approximate size/complexity.
- Command(s):
  - cloc --json . > analysis-report/cloc.json (if cloc installed)
  - tokei . -o json > analysis-report/tokei.json (if available)
  - fallback: rg --hidden --no-ignore -n "" --type-add 'project:*.py' . | wc -l per type using built-in mapping
- Output files: analysis-report/lang-stats.txt (human-readable) and analysis-report/cloc.json
- Notes: Use cloc or tokei; if neither available, run wc/rg counting extensions.

Task 3 — Inspect README and top-level docs
- Purpose: extract project description, build and run instructions, and any badges indicating CI or coverage.
- File(s): README.md, docs/ folder, CONTRIBUTING.md
- Command(s): head -n 200 README.md > analysis-report/README.head
- Output: include short summary in SUMMARY.md

Task 4 — Identify dependency manifests
- Purpose: find package manifests for language ecosystems.
- Files to look for: package.json, yarn.lock, pnpm-lock.yaml, Pipfile, requirements.txt, pyproject.toml, setup.py, Gemfile, go.mod, Cargo.toml, build.gradle, composer.json
- Command(s): rg -n "(^|/)package.json$|(^|/)pyproject.toml$|(^|/)requirements.txt$|(^|/)go.mod$|(^|/)Cargo.toml$" --hidden --glob '!.git' || true
- Parse manifests: for JSON manifests, jq to extract name/version/dependencies (jq '.name, .version' etc.)
- Output: analysis-report/dependency-list.txt and analysis-report/manifests/

Task 5 — CI/workflows and automation surface
- Purpose: list CI workflows and scripts that run builds/tests.
- Files to check: .github/workflows/*.yml, .gitlab-ci.yml, Jenkinsfile, magefiles, Makefile, scripts/, ci/
- Command(s): ls .github/workflows || true; rg -n "workflow|jobs|steps|name:" .github -S || true
- Output: analysis-report/ci-files.txt; extract badge links found in README

Task 6 — Dockerfiles and containerization
- Purpose: find Dockerfiles and container build contexts.
- Command(s): rg -n "^FROM\s" --hidden --glob '!node_modules' || true; find . -iname Dockerfile
- Output: analysis-report/dockerfiles.txt

Task 7 — Tests and test frameworks
- Purpose: find tests, frameworks, and rough test counts.
- Files/dirs: tests/, spec/, __tests__, test/, jest.config, pytest.ini, tox.ini
- Commands: find tests -type f | wc -l; rg -n "^def test_|^it\(|describe\(" --hidden
- Output: analysis-report/tests-summary.txt

Task 8 — Top-level modules and packages
- Purpose: identify the main application code and entry points.
- Heuristic: look for directories with highest LOC from Task 2, package manifests pointing to main entries (package.json main, pyproject packages, go modules).
- Commands: parse manifests and correlate with cloc results.
- Output: add section to analysis-report/analysis.json listing top 3 candidate roots with reasons.

Task 9 — License and contributor files
- Purpose: check for LICENSE, CODE_OF_CONDUCT.md, contributors, and security policy.
- Commands: rg -n "^License|LICENSE$|CODE_OF_CONDUCT|SECURITY.md" -S || true
- Output: analysis-report/license.txt

Task 10 — Build and test commands (advisory only)
- Purpose: extract commands for build/test from README and manifests.
- Commands: parse package.json scripts, setup.cfg, pyproject, Makefile targets.
- Output: analysis-report/commands.txt

Task 11 — Produce human-readable summary
- Purpose: synthesize findings into SUMMARY.md with: top-level tree, language breakdown, main modules, CI status, test surface, dependency highlights, recommended next steps.
- Input: all previous artifact files.
- Output: analysis-report/SUMMARY.md

Task 12 — Produce structured JSON report
- Purpose: capture the results in analysis-report/analysis.json with schema:
  {
    "top_level": [...],
    "languages": {"Python": 1234, ...},
    "manifests": [...],
    "ci_files": [...],
    "dockerfiles": [...],
    "tests": {"count": 42, "paths": [...]}
  }
- Implementation: small Python script that reads intermediate artifacts and writes JSON.

Execution notes
- Commands assume a POSIX environment with common developer tools. If tools (cloc, tokei) are missing, the script should gracefully fall back.
- All outputs are read-only with respect to the source code; no changes made.

Who should run these tasks
- A single CI job or a developer machine. Each task is independent; tasks 1–3 should run first.

Estimated time
- Automated execution: 1–3 minutes for medium-sized repo (< 10k files). Manual review and summary drafting: 10–20 minutes.

Follow-ups and recommendations
- After producing the report, run dependency vulnerability scans, automated license scanning, and run CI tests to verify build status.
- If tests are missing, recommend adding a minimal smoke test and CI job.
