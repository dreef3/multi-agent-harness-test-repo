Implementation plan: Task-by-task plan to analyze the codebase structure for E2E Test Repo 1 (repo id: 4f966f8d-2e5c-4f97-b396-ef7fe727452f, branch: main)

Overview
- Create a results/ directory at the repo root and produce a set of read-only inspection outputs. Each task is independent and safe to run in parallel. Tasks assume execution from the repository root on branch main.

Tasks

Task 1 — Produce top-level file tree
- Purpose: enumerate all files and directories to give a high-level structure.
- Command(s):
  - mkdir -p results
  - find . -maxdepth 2 -not -path './.git*' -not -path './results*' -printf "%P\n" | sort > results/tree.txt
  - head -n 200 results/tree.txt > results/tree_head.txt
- Output: results/tree.txt, results/tree_head.txt
- Acceptance: results/tree.txt exists and contains the repository top-level entries and first-level subdirectories.

Task 2 — Detect primary languages and file counts
- Purpose: determine main languages and approximate size.
- Command(s):
  - find . -type f -not -path './.git/*' -not -path './results/*' | sed -n '1,20000p' > results/all_files.txt
  - awk -F. '/\./{print $NF}' results/all_files.txt | tr '[:upper:]' '[:lower:]' | sort | uniq -c | sort -nr > results/filetypes-count.txt
  - Create short language summary (map common extensions to languages); write results/language-summary.md
- Output: results/filetypes-count.txt, results/language-summary.md
- Acceptance: language-summary.md contains top 3 languages with counts.

Task 3 — Copy common manifest files
- Purpose: collect package manifests for dependency summaries
- Manifests to check and copy if present:
  - package.json
  - yarn.lock, package-lock.json
  - pyproject.toml, requirements.txt, setup.py
  - go.mod
  - Cargo.toml
  - Gemfile
  - composer.json
- Command(s):
  - mkdir -p results/manifests
  - for f in package.json package-lock.json yarn.lock pyproject.toml requirements.txt setup.py go.mod Cargo.toml Gemfile composer.json; do if [ -f "$f" ]; then cp "$f" results/manifests/; fi; done
- Output: results/manifests/* (copied manifests)
- Acceptance: All present manifests are in results/manifests.

Task 4 — Summarize dependency manifests
- Purpose: extract top-level metadata: name, version, scripts, dependencies
- Command(s):
  - If package.json exists: jq '{name,version,scripts,dependencies,devDependencies}' results/manifests/package.json > results/manifests/package-summary.json || echo "jq not available; skipping JSON extract" > results/manifests/package-summary.txt
  - If pyproject.toml exists: sed -n '1,200p' results/manifests/pyproject.toml > results/manifests/pyproject-head.toml
  - If go.mod exists: sed -n '1,200p' results/manifests/go.mod > results/manifests/go-mod-head.txt
  - Compose results/manifests/manifest-summary.md with findings
- Output: results/manifests/manifest-summary.md (and optional parsed files)
- Acceptance: manifest-summary.md lists each manifest discovered and key entries.

Task 5 — Detect CI and workflows
- Purpose: find CI configuration and summarize
- Command(s):
  - mkdir -p results/ci
  - if [ -d ".github/workflows" ]; then ls .github/workflows > results/ci/workflows.txt; for f in .github/workflows/*; do echo "---- $f" >> results/ci/workflows_raw.txt; sed -n '1,200p' "$f" >> results/ci/workflows_raw.txt; done; fi
  - if [ -f ".gitlab-ci.yml" ]; then cp .gitlab-ci.yml results/ci/.; fi
  - if [ -f "circle.yml" ]; then cp circle.yml results/ci/.; fi
  - Summarize into results/ci.md
- Output: results/ci/*, results/ci.md
- Acceptance: ci.md lists workflow filenames and a one-line summary for each (e.g., language matrix, build/test steps) where feasible.

Task 6 — Enumerate tests and recommend test commands
- Purpose: locate test frameworks and provide commands to run test suite
- Command(s):
  - grep -RIn "pytest\|unittest\|mocha\|jest\|ava\|go test\|cargo test\|dotnet test" --exclude-dir=.git --exclude-dir=results || true > results/tests_raw_grep.txt
  - find . -type f -iname "*_test.*" -o -iname "test_*.py" -o -iname "*.spec.*" > results/test_files.txt
  - If package.json: jq -r '.scripts // {}' results/manifests/package.json > results/package-scripts.txt || true
  - Write results/tests.md with suggested commands based on detected languages: e.g., npm test / pytest -q / go test ./...
- Output: results/test_files.txt, results/tests.md
- Acceptance: tests.md provides at least one command to run tests or states that no tests were detected.

Task 7 — Check for Dockerfile / Makefile / infra
- Purpose: find developer tooling and common entrypoints
- Command(s):
  - for f in Dockerfile docker-compose.yml Makefile .env sample.env; do if [ -f "$f" ]; then cp "$f" results/; fi; done
  - Write results/infra.md noting presence of these files and a one-paragraph description.
- Output: results/Dockerfile (if present), results/Makefile (if present), results/infra.md
- Acceptance: infra.md lists detected infra files and recommended next steps (e.g., inspect Dockerfile for ports, base image).

Task 8 — Produce final summary
- Purpose: aggregate outputs into a single human-friendly report.
- Command(s):
  - Create results/summary.md containing:
    - repository id and branch
    - top-level files and directories (first 100 lines of results/tree_head.txt)
    - primary languages and file counts (from language-summary.md)
    - manifests present and top-level dependencies (from manifest-summary.md)
    - CI presence (from results/ci.md)
    - tests location and commands (from results/tests.md)
    - developer tooling found (from results/infra.md)
    - recommended next steps (e.g., run tests, open issues for missing LICENSE, run static analysis)
- Output: results/summary.md
- Acceptance: results/summary.md is present and succinctly answers the questions in the spec.

Execution notes
- All commands should be run from repository root on branch main.
- Tools assumed to be available: sh, find, sed, awk, sort, uniq, cp, mkdir, jq (if available). Scripts should fail gracefully if jq is missing.
- Tasks are read-only and should never modify tracked files (they write into results/).

Deliverables
- Commit (by the implementation agent) of the results/ directory with the generated artifacts.
- A PR will be opened by the harness containing the results; the summary.md should be the primary reviewer-facing artifact.

If you want, after these tasks complete I can prepare a follow-up plan to (a) run unit tests, (b) run linters and static analyzers, and (c) create an initial Developer Onboarding doc based on findings.
