Implementation Plan: Repository Structure Analysis (for E2E Test Repo)

Overview
- Create a sequence of self-contained tasks that a sub-agent can run to clone the repository and produce the requested analysis artifacts. Each task includes exact commands to run, expected outputs, and files to create. No task requires modifying source code; the final step creates report files and opens a PR.

Repository
- URL: https://github.com/dreef3/multi-agent-harness-test-repo.git
- Default branch: main
- Repository ID: a9ff8893-eb13-47f6-a51a-1fad29e0fecb

Tasks

Task 1 — Clone repository
- Purpose: Obtain a local copy for analysis.
- Commands:
  - git clone https://github.com/dreef3/multi-agent-harness-test-repo.git repo
  - cd repo
  - git checkout main
- Expected outcome: A local working tree at ./repo.
- Outputs: local repo directory (./repo)

Task 2 — Produce top-level tree and counts
- Purpose: Capture the repository directory layout and file counts.
- Commands (run from inside repo):
  - find . -maxdepth 2 -type d -printf "%P\n" | sed 's/^\.//g' | sort > /tmp/top_dirs.txt
  - find . -maxdepth 3 -type f | sed 's|^./||' | sort > /tmp/top_files.txt
  - echo "Total files: $(wc -l < /tmp/top_files.txt)" > /tmp/file_counts.txt
- Expected outcome: lists of top-level directories and files and total file count.
- Outputs to commit: analysis/top_dirs.txt, analysis/top_files.txt, analysis/file_counts.txt

Task 3 — Language detection and primary language
- Purpose: Identify primary languages (using cloc and GitHub Linguist as fallback).
- Commands:
  - cloc --json . > analysis/cloc-report.json || true
  - # If cloc is not available, use a simple heuristic:
  - find . -type f -name "*.py" | wc -l > analysis/python_files.count || true
  - find . -type f -name "*.js" | wc -l > analysis/js_files.count || true
- Expected outcome: a cloc JSON or simple file counts indicating likely languages.
- Outputs: analysis/cloc-report.json (or small counts files)

Task 4 — Detect package/manifest files and package managers
- Purpose: Find build manifests and lockfiles to determine package managers in use.
- Commands:
  - for f in package.json pyproject.toml requirements.txt Pipfile setup.py setup.cfg Poetry.lock yarn.lock package-lock.json go.mod go.sum Cargo.toml Gemfile Gemfile.lock build.gradle build.gradle.kts settings.gradle; do if [ -f "$f" ]; then echo "$f"; fi; done > analysis/manifests.txt
  - grep -RIn "^name\|^version\|^dependencies\|^devDependencies\|^buildscript" --binary-files=without-match package.json pyproject.toml 2>/dev/null || true
- Expected outcome: list of manifests and lockfiles.
- Outputs: analysis/manifests.txt

Task 5 — Detect CI/CD and workflow files
- Purpose: Identify GitHub Actions, other CI configs.
- Commands:
  - ls -R .github/workflows 2>/dev/null | sed 's/^/github_workflow: /' > analysis/ci_files.txt || true
  - for f in .travis.yml circle.yml .gitlab-ci.yml Jenkinsfile; do if [ -f "$f" ]; then echo "$f"; fi; done >> analysis/ci_files.txt
- Expected outcome: list of CI files.
- Outputs: analysis/ci_files.txt

Task 6 — Detect tests and test frameworks
- Purpose: Find test directories and common test invocation scripts.
- Commands:
  - find . -type d -name "test*" -print > analysis/test_dirs.txt || true
  - grep -RIn "pytest\|unittest\|jest\|mocha\|go test\|cargo test" 2>/dev/null | head -n 200 > analysis/test_signatures.txt || true
- Expected outcome: candidate test directories and any test runner signatures.
- Outputs: analysis/test_dirs.txt, analysis/test_signatures.txt

Task 7 — Detect Dockerfiles and build scripts
- Purpose: Locate Dockerfiles, Makefiles, and CI build scripts.
- Commands:
  - find . -type f -name Dockerfile -o -name "Dockerfile.*" > analysis/dockerfiles.txt || true
  - if [ -f Makefile ]; then echo "Makefile" > analysis/has_makefile.txt; fi
  - grep -RIn "docker build\|docker run\|FROM " --binary-files=without-match . | head -n 200 > analysis/docker_references.txt || true
- Outputs: analysis/dockerfiles.txt, analysis/has_makefile.txt, analysis/docker_references.txt

Task 8 — Dependency surface and lockfile inspection
- Purpose: Collect direct dependencies by reading manifests and lockfiles (do not install).
- Commands:
  - # For npm/yarn
  - if [ -f package.json ]; then jq '.dependencies,.devDependencies' package.json > analysis/package.json.deps.json || true; fi
  - # For Python
  - if [ -f pyproject.toml ]; then grep -P "^\[tool.poetry.dependencies\]|^\[project.dependencies\]" -n pyproject.toml -n > /dev/null 2>&1 && cp pyproject.toml analysis/pyproject.toml || true; fi
  - # For go
  - if [ -f go.mod ]; then cp go.mod analysis/go.mod || true; fi
- Outputs: analysis/package.json.deps.json, analysis/pyproject.toml (copied), analysis/go.mod

Task 9 — Summarize findings into Markdown report
- Purpose: Convert collected artifacts into docs/codebase-analysis.md and checklist.txt.
- Report outline (docs/codebase-analysis.md):
  - Repo metadata (URL, default branch)
  - Top-level layout (from analysis/top_dirs.txt)
  - Language breakdown (from cloc or counts)
  - Manifests & package managers (analysis/manifests.txt)
  - CI & workflows (analysis/ci_files.txt)
  - Tests (analysis/test_dirs.txt + signatures)
  - Docker/build (analysis/dockerfiles.txt)
  - Dependency highlights (analysis/*.deps files)
  - Risk/notes and recommended next steps (e.g., add license, add CI if missing, add CONTRIBUTING.md, run static analysis)
- Commands to assemble report (examples to run inside repo):
  - mkdir -p docs analysis
  - # Use simple templating via a here-doc; example:
  - cat > docs/codebase-analysis.md <<'MD'
  - # Codebase Analysis
  - (insert summarized content by reading the analysis/*.txt files)
  - MD
  - # create checklist
  - cat > docs/checklist.txt <<'TXT'
  - README: $(test -f README.md && echo "present" || echo "missing")
  - LICENSE: $(test -f LICENSE && echo "present" || echo "missing")
  - CI: $(test -d .github/workflows && echo "github actions present" || echo "ci missing")
  - TXT
- Outputs to commit: docs/codebase-analysis.md, docs/checklist.txt, analysis/*

Task 10 — Commit and open PR
- Purpose: Add the generated artifacts and open a PR on a planning branch.
- Commands (to be executed by the bot/human — do not run here):
  - git checkout -b analysis/codebase-structure
  - git add docs analysis
  - git commit -m "Add codebase analysis report"
  - git push --set-upstream origin analysis/codebase-structure
  - # Open PR via GitHub UI or gh cli: gh pr create --title "Codebase structure analysis" --body "Adds codebase analysis artifacts" --base main
- Expected outcome: A PR containing the report and analysis artifacts.
