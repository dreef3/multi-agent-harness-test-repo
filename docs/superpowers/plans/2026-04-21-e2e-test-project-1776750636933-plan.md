Implementation plan: step-by-step tasks to analyze repository structure for 731d7059-8b54-40a5-89be-6ad22438ab06 (multi-agent-harness-test-repo)

Overview
- This plan breaks the analysis into discrete tasks. Each task is self-contained and runnable on a checked-out copy of the repository. Outputs are machine-readable where possible (JSON or plain text) to enable later automation.

Prerequisite
- Clone the repo locally: git clone https://github.com/dreef3/multi-agent-harness-test-repo.git && cd multi-agent-harness-test-repo
- Use a POSIX shell with common tools: bash, jq, sed, awk, tree (or find), grep, wc, python3 (optional), cargo/go/npm/pip depending on languages found.

Tasks

Task 1 — Top-level tree and counts
- Goal: produce a short file tree and counts of top-level directories and file counts by extension.
- Commands:
  - find . -maxdepth 2 -mindepth 1 -printf "%P\n" | sort > analysis/top_level_entries.txt
  - find . -type f | sed 's|\./||' | sed -E 's|.*\.(.*)$|\1|' | sort | uniq -c | sort -rn > analysis/file_extension_counts.txt
  - (optional) tree -L 2 -a -I vendor,node_modules > analysis/tree_level2.txt
- Output files (in repo): analysis/top_level_entries.txt, analysis/file_extension_counts.txt, analysis/tree_level2.txt

Task 2 — Identify primary languages and build files
- Goal: Find common language indicators and build config files.
- Paths/Files to inspect (exact): package.json, pyproject.toml, setup.py, requirements.txt, Pipfile, Pipfile.lock, go.mod, Cargo.toml, Makefile, Dockerfile, .github/workflows/*, .gitlab-ci.yml, .circleci/config.yml
- Commands:
  - for f in package.json pyproject.toml setup.py requirements.txt Pipfile Pipfile.lock go.mod Cargo.toml Makefile Dockerfile .gitlab-ci.yml; do [ -f "$f" ] && echo "$f: exists"; done > analysis/build_files_presence.txt
  - ls -la .github/workflows > analysis/github_workflows_list.txt || true
- Inspect language-specific files for versions and main entry points (example: read package.json to get main, scripts, engines). Use jq: jq '{name,version,scripts,engines,depen­dencies,devDependencies}' package.json > analysis/package_json_summary.json (if package.json exists)

Task 3 — README, LICENSE, CONTRIBUTING
- Goal: determine presence and quality of docs and usage instructions
- Files: README.md, README.*, LICENSE, LICENSE.*, CONTRIBUTING.md, CODE_OF_CONDUCT.md, CHANGELOG.md
- Commands:
  - for f in README.md LICENSE CONTRIBUTING.md CODE_OF_CONDUCT.md CHANGELOG.md; do [ -f "$f" ] && wc -l "$f" || echo "$f: missing"; done > analysis/docs_presence.txt
  - head -n 200 README.md > analysis/README_head.md

Task 4 — CI and automation
- Goal: discover CI pipelines, matrix, and build/test commands.
- Files: .github/workflows/*.yml, .gitlab-ci.yml, circleci/config.yml, Jenkinsfile
- Commands:
  - grep -R "runs-on\|matrix\|python-version\|node-version\|go-version\|steps" .github/workflows -n || true > analysis/ci_summary_github.txt
  - sed -n '1,200p' .github/workflows/*.yml > analysis/github_workflows_combined.yml || true
- Check for badges in README.md referencing CI services: grep -E "travis-ci|github.com/.*/actions|circleci" README.md || true > analysis/readme_badges.txt

Task 5 — Tests: framework and how to run
- Goal: identify test frameworks, unit test directories, and sample commands to run tests.
- Typical paths: tests/, test/, src/*/test, __tests__/, spec/
- Commands:
  - rg "pytest|unittest|mocha|jest|ava|go test|cargo test" -n || true > analysis/test_frameworks_found.txt
  - find . -type d -name "test*" -o -name "__tests__" > analysis/test_dirs.txt
  - Extract test script from package.json: jq -r '.scripts.test' package.json > analysis/npm_test_script.txt || true

Task 6 — Dependencies and security flags
- Goal: list direct dependencies and flag outdated ones.
- Commands per ecosystem:
  - Node.js: if package.json exists: jq -r '.dependencies, .devDependencies' package.json > analysis/deps_node.json; npm install --package-lock-only; npm outdated --json > analysis/npm_outdated.json
  - Python: if requirements.txt or pyproject.toml: pip install --no-deps -r requirements.txt -t /tmp/deps-inspect || true; pip list --format=json > analysis/pip_list.json; safety check with pip-audit/pipdeptree where available.
  - Go: go list -m all > analysis/go_mod_deps.txt
  - Rust: cargo metadata --format-version 1 --no-deps > analysis/cargo_metadata.json
- If environments are not available, fallback to parsing lock files (package-lock.json, Pipfile.lock, poetry.lock).

Task 7 — Source code layout and entrypoints
- Goal: find source directories and main executables or packages.
- Common paths: src/, cmd/, pkg/, internal/, lib/, app/, main.go, server.py, manage.py
- Commands:
  - rg "package main|def main\(|if __name__ == '__main__'|func main\(|module.exports|export default" -n || true > analysis/main_candidates.txt
  - find src cmd pkg -maxdepth 2 -type d > analysis/src_candidate_dirs.txt || true

Task 8 — Tests execution (optional and gated)
- Goal: run the test suite in a controlled environment and collect pass/fail and runtimes.
- Steps (run only if safe and with available environment):
  - For Node: npm ci && npm test --silent
  - For Python: python3 -m venv .venv && .venv/bin/pip install -r requirements.txt && pytest -q --disable-warnings --maxfail=1 --junitxml=analysis/test_results.xml
  - For Go: go test ./... -json > analysis/go_test_results.json
- Capture console output and exit codes in analysis/test_run_summary.txt

Task 9 — Size and large files
- Goal: identify large files or folders that may be checked in.
- Commands:
  - git rev-list --objects --all | sort -k2 > analysis/git_objects.txt (requires repo)
  - du -h --max-depth=3 | sort -hr | head -n 50 > analysis/top_disk_usage.txt
  - find . -type f -size +5M -printf "%p %k KB\n" > analysis/large_files.txt

Task 10 — Produce final report
- Goal: aggregate findings into a Markdown report with recommendations.
- Contents:
  - Executive summary (1–3 paragraphs)
  - File tree and languages
  - Build and test instructions (how to get started)
  - CI summary and badge status
  - Dependency summary and vulnerabilities/outdated libs
  - Notable issues and prioritized recommendations (High/Medium/Low)
  - Appendix: raw artifacts from analysis/*
- Command:
  - Use a small script to assemble: cat templates/summary_header.md analysis/* > analysis/final_report.md

Outputs and locations
- analysis/ (contains all intermediate files)
- analysis/final_report.md (primary deliverable)

Timings and sequencing
- Expected time: quick exploratory run (5–15 minutes) to collect file lists and build-file presence; deeper runs (dependency audit, running tests) may require environment setup (15–60 minutes).
- Tasks can be parallelized: Task 1–4, 6, 7, 9 can be executed in parallel. Task 8 (running tests) should be gated and run after confirming environment needs.

Notes and safety
- Do not make code changes. All tasks are read-only.
- Running tests or dependency installs may run arbitrary code (install scripts). Only run these in a sandboxed environment or CI runner.
- If tests are not to be executed, skip Task 8 and still produce the report marking tests as un-run.

Example commands and snippets (no implementation changes)

- Show package.json summary: jq '{name,version,scripts}' package.json
- Show failures quickly: (npm test 2>&1) | tee analysis/npm_test_output.txt
