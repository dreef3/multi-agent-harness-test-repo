Implementation Plan: Repository analysis for E2E Test Repo 1 (ID: 136ad42f-fdc8-41fd-a515-d23075abe901)

Overview
- This plan defines a sequence of self-contained tasks that a sub-agent (or human) will execute in the repository checkout to produce the analysis artifacts described in the spec. Each task includes exact commands, expected outputs, and target file paths. All outputs will be written under analysis/ in the repository root so they can be committed and reviewed.

Assumptions
- You have a local clone of the repository on a machine with standard UNIX tools available.
- Preferred utilities (if available): tree, rg (ripgrep), cloc or tokei, git, jq, npm, pip, mvn, gradle, go, cargo. If not available, the tasks include fallbacks.
- Tasks are read-only to source files; they create new files under analysis/ only.

Top-level plan (task list)

Task 1 — Prepare workspace
- Purpose: create analysis directory structure
- Commands:
  mkdir -p analysis/structure_raw analysis/dependencies analysis/graphs
- Outputs (paths):
  analysis/ (tree created)

Task 2 — Capture top-level file/directory listing
- Purpose: record the repo top-level layout
- Commands (preferable):
  tree -a -L 2 > analysis/structure_raw/tree.txt
  # Fallback if tree is not installed:
  (ls -la && find . -maxdepth 2 -mindepth 1 -print) > analysis/structure_raw/tree.txt
- Outputs:
  analysis/structure_raw/tree.txt

Task 3 — Full file manifest (file paths only)
- Purpose: comprehensive list of files for scanning
- Commands:
  find . -type f -not -path './.git/*' | sort > analysis/structure_raw/file_manifest.txt
- Outputs:
  analysis/structure_raw/file_manifest.txt

Task 4 — Language and code size summary
- Purpose: detect languages and code size (lines of code)
- Commands (try cloc then fallback to tokei):
  cloc --json --out=analysis/structure_raw/cloc.json . || tokei --output json > analysis/structure_raw/cloc.json
- If neither tool exists, produce a simple fallback:
  awk 'END{print "no cloc/tokei available"}' > analysis/structure_raw/cloc.json
- Outputs:
  analysis/structure_raw/cloc.json

Task 5 — Quick grep for important manifest files
- Purpose: locate dependency manifests, CI files, Dockerfiles and build scripts
- Commands:
  rg -n --hidden "(^name:|^version:|package.json|pyproject.toml|requirements.txt|Pipfile|setup.py|pom.xml|build.gradle|go.mod|Cargo.toml|Gemfile|Dockerfile|\.github/workflows/|\.gitlab-ci.yml|circleci|Jenkinsfile)" -S --no-ignore > analysis/structure_raw/manifests_grep.txt || true
- Outputs:
  analysis/structure_raw/manifests_grep.txt

Task 6 — Dependency extraction per language (run only if manifest found)
- Purpose: produce dependency lists for each detected language
- Approach: check for files, then run appropriate commands. All outputs go under analysis/dependencies/.
- Exact checks/commands (examples):
  # npm / node
  if [ -f package.json ]; then jq -r '.dependencies + .devDependencies | keys[]' package.json > analysis/dependencies/npm-deps.txt || npm ls --depth=0 --json > analysis/dependencies/npm-deps.json; fi

  # Python
  if [ -f requirements.txt ]; then cp requirements.txt analysis/dependencies/python-requirements.txt; fi
  if [ -f pyproject.toml ]; then cp pyproject.toml analysis/dependencies/pyproject.toml; fi

  # Go
  if [ -f go.mod ]; then go list -m all > analysis/dependencies/go-mod-list.txt || cp go.mod analysis/dependencies/go.mod; fi

  # Maven
  if [ -f pom.xml ]; then mvn -q dependency:tree -DoutputFile=analysis/dependencies/maven-deps.txt || cp pom.xml analysis/dependencies/pom.xml; fi

  # Gradle
  if [ -f build.gradle ] || [ -f build.gradle.kts ]; then ./gradlew dependencies --no-daemon --console=plain > analysis/dependencies/gradle-deps.txt || cp build.gradle* analysis/dependencies/; fi

  # Cargo (Rust)
  if [ -f Cargo.toml ]; then cargo tree --prefix none > analysis/dependencies/cargo-deps.txt || cp Cargo.toml analysis/dependencies/Cargo.toml; fi

  # Ruby
  if [ -f Gemfile ]; then bundle list --local > analysis/dependencies/bundle-list.txt || cp Gemfile analysis/dependencies/Gemfile; fi

- Outputs (examples):
  analysis/dependencies/npm-deps.txt
  analysis/dependencies/python-requirements.txt
  analysis/dependencies/maven-deps.txt
  etc.

Task 7 — Detect CI and Docker usage
- Purpose: identify CI configs and Dockerfiles and summarize
- Commands:
  rg -n "^name:|Dockerfile|\.github|\.gitlab-ci.yml|circleci|Jenkinsfile" -S --no-ignore || true > analysis/structure_raw/ci_and_docker_grep.txt
  # Also explicitly copy detected CI files into analysis
  mkdir -p analysis/ci_and_docker
  if [ -d .github/workflows ]; then cp -r .github/workflows analysis/ci_and_docker/ || true; fi
  if [ -f .gitlab-ci.yml ]; then cp .gitlab-ci.yml analysis/ci_and_docker/; fi
  if [ -f Dockerfile ]; then cp Dockerfile analysis/ci_and_docker/Dockerfile; fi
- Outputs:
  analysis/structure_raw/ci_and_docker_grep.txt
  analysis/ci_and_docker/*

Task 8 — Tests and test-related scripts detection
- Purpose: determine if repo contains automated tests and how to run them
- Commands:
  rg -n "(^pytest|unittest|mocha|jest|pytest.ini|tox.ini|__tests__|tests/|spec/)" -S --no-ignore > analysis/structure_raw/tests_grep.txt || true
  # Look for test scripts in package.json
  if [ -f package.json ]; then jq -r '.scripts.test // empty' package.json > analysis/structure_raw/npm_test_script.txt || true; fi
- Outputs:
  analysis/structure_raw/tests_grep.txt
  analysis/structure_raw/npm_test_script.txt

cloc=json.load(open('analysis/structure_raw/cloc.json')) if os.path.exists('analysis/structure_raw/cloc.json') else {}
# construct a short markdown file by reading the raw artifacts
PY
  # If python is not available, use here-doc with shell to aggregate key files.

- Output:
  analysis/structure_report.md

Task 10 — Machine-readable summary
- Purpose: create analysis/summary.json containing key fields: repo_id, top_level_dirs, languages (name + LOC), dependency_manifests (list), ci_files (list), dockerfiles (list), tests_present (bool), recommended_next_steps (array)
- Example content keys (JSON) and creation commands (jq or python)

Task 11 — Optional visualizations
- Purpose: create module graph or dependency graph if possible
- Commands (examples):

Task 12 — Commit artifacts and open PR
- Purpose: create a branch, commit analysis/ directory and open a PR with summary
- Recommended branch name: analysis/structure-report
- Commands (do not run from planning agent, sub-agent should run):
  git checkout -b analysis/structure-report
  git add analysis
  git commit -m "Add automated repository structure analysis"
  git push --set-upstream origin analysis/structure-report
  # PR creation is handled by the harness; include summary in PR body or CI will pick it up.

Notes and fallbacks
- If some language-specific tooling is not installed, copy manifests into analysis/dependencies so reviewers can inspect manually.
- If running dependency tools may attempt network access (e.g., npm install), avoid running install; prefer read-only list commands (npm ls, pipdeptree without installing). Document any network steps clearly in the PR text.
- Keep the analysis reproducible by committing scripts used (e.g., analysis/run_analysis.sh) if desired.

Estimated time
- Automated run: typically 1-5 minutes for medium-sized repos (no installs), longer if complex builds are invoked.

Acceptance criteria
- analysis/ directory added to a branch containing all artifacts above and a clear structure_report.md summarizing findings.
