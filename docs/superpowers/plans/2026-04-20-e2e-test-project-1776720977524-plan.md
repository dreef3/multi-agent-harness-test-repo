Implementation plan: repository analysis for multi-agent-harness-test-repo (id: f38a09ea-cf13-4846-8501-16f41cf338ff)

Overview
- The plan below is a sequence of self-contained read-only tasks that an implementer or sub-agent can run against the repository. Each task includes exact file paths and command examples (bash) to run, what to extract from the output, and how to record the findings.
- Do not modify files or run tests unless explicitly noted and permitted.

Assumptions
- You have a local clone of the repository (default branch: main) or read access via an automation agent.
- Standard CLI tools are available: ls, find, rg/grep, sed, awk, jq, head, tail, cat. cloc is highly recommended if available.

Tasks

Task 1 — Top-level inventory (files and directories)
- Purpose: capture the top-level layout and quick counts.
- Commands:
  - ls -la
  - find . -maxdepth 2 -mindepth 1 -print | sed 's|^./||' | sort
  - tree -L 2 || (if tree not installed) use: find . -maxdepth 2 -type d -or -type f
- Files/paths to check (explicit): ./README.md, ./README, ./docs/, ./src/, ./lib/, ./packages/, ./tests/, ./e2e/, ./Dockerfile, ./docker-compose.yml, ./Makefile, ./package.json, ./pyproject.toml, ./requirements.txt, ./go.mod
- Expected outputs:
  - Top-level list of directories and files. Note presence/absence of canonical project files.
- How to record:
  - A short bullet list of top-level items and counts (directories, files).

Task 2 — Detect languages and dependency manifests
- Purpose: determine primary languages and package managers.
- Commands (search for canonical files):
  - test -f package.json && echo "Node project"
  - test -f pyproject.toml && echo "Python (pyproject)"
  - test -f requirements.txt && echo "Python (requirements.txt)"
  - test -f go.mod && echo "Go"
  - test -f Cargo.toml && echo "Rust"
  - test -f pom.xml && echo "Maven/Java"
  - test -f build.gradle && echo "Gradle/Java/Kotlin"
  - test -f Gemfile && echo "Ruby"
  - test -f composer.json && echo "PHP"
  - test -f package-lock.json || test -f yarn.lock || test -f pnpm-lock.yaml
- Additional automated language breakdown (if cloc available):
  - cloc --json . > cloc.json
  - jq '.' cloc.json
- Files to open and parse (examples):
  - package.json: extract name, version, scripts, main, dependencies, devDependencies
    - jq '.name, .version, .scripts' package.json
  - pyproject.toml: show [tool.poetry] or [project]
  - requirements.txt: list pinned deps
  - go.mod: module name and require lines
- How to record:
  - Primary languages (1-3) with evidence (file paths and key fields), list of dependency manifests found with brief parsed results (top-level name/version, notable dependencies).

Task 3 — CI/CD and workflows
- Purpose: find pipelines and automated checks.
- Paths to inspect:
  - .github/workflows/*.yml
  - .gitlab-ci.yml
  - Jenkinsfile
  - azure-pipelines.yml
  - .circleci/config.yml
- Commands:
  - ls -la .github/workflows || echo "no .github/workflows"
  - rg "runs-on:|uses:|workflow_dispatch|on:" .github/workflows -n || true
  - head -n 200 .github/workflows/*.yml (for each file)
- What to extract:
  - CI triggers (push, pull_request, schedule), job names, languages/containers used, test commands invoked, caching strategies, matrix builds, artifact publish steps.
- How to record:
  - For each workflow file, one-line summary: triggers, jobs, test commands, notable steps (build/publish).

Task 4 — Test discovery and test commands
- Purpose: identify test frameworks and where tests live.
- Paths to check:
  - tests/, spec/, __tests__/, cypress/, e2e/, integration_tests/
  - package.json scripts.test
  - pyproject.toml or setup.cfg for pytest settings
  - jest.config.js, jest.config.ts
- Commands:
  - rg "describe\(|test\(|it\(|pytest\.|unittest\.|from pytest|@pytest" -n tests || true
  - jq '.scripts.test' package.json || true
  - head -n 200 pytest.ini tox.ini setup.cfg || true
- How to record:
  - List test directories, primary test command(s) to run, number of test files discovered (approx.), any e2e frameworks (Cypress, Playwright, Selenium).

Task 5 — Docker, containerization, and infra
- Purpose: find container and deployment assets.
- Paths to inspect: Dockerfile*, docker-compose.yml*, k8s/, manifests/, deploy/
- Commands:
  - test -f Dockerfile && head -n 120 Dockerfile
  - rg "FROM " Dockerfile -n || true
  - ls -la k8s || true
- How to record:
  - Presence of Dockerfile(s), base images used, exposed ports, compose services, and k8s manifests if present.

Task 6 — Build and development scripts
- Purpose: find common dev commands and helper scripts.
- Files: Makefile, scripts/*, package.json scripts, tools/ or bin/
- Commands:
  - head -n 200 Makefile || true
  - ls -la scripts || true
  - jq '.scripts' package.json || true
- How to record:
  - Key developer commands (e.g., make build/test/lint, npm run build/test/lint), location of helper scripts.

Task 7 — Licensing and contribution guidance
- Files: LICENSE, LICENSE.* , CONTRIBUTING.md, CODE_OF_CONDUCT.md
- Commands: head -n 40 LICENSE || true
- How to record:
  - License type (if obvious), presence of contribution guidelines.

Task 8 — Security and dependency lockfiles
- Files: package-lock.json, yarn.lock, pnpm-lock.yaml, Pipfile.lock, poetry.lock, Gemfile.lock
- Commands:
  - test -f package-lock.json && head -n 40 package-lock.json
  - test -f yarn.lock && head -n 40 yarn.lock
- How to record:
  - Which lockfiles exist (helps pin reproducible builds), any obvious outdated constraints if easily visible.

Task 9 — Produce the analysis report
- Purpose: compile the outputs from Tasks 1–8 into a structured report with these sections:
  1. Executive summary (primary languages, test status, CI presence, Docker/infra presence)
  2. Top-level file tree snapshot (top 2 levels)
  3. Dependency manifests found (parsed key fields)
  4. CI/workflow summary (one line per workflow file)
  5. Test layout and test commands
  6. Notable scripts and build steps
  7. Licensing and contribution notes
  8. Potential issues and recommended next steps
- Format: Markdown file named analysis/structure-report.md in the working directory (or provide as PR comment). Include verbatim snippets (<=200 lines) from key files for evidence; reference file paths.

Task 10 — Optional: test execution (manual approval required)
- Purpose: attempt to run unit tests in an isolated environment. Only do this after approval; this step is optional and not required for the structure analysis.
- Examples:
  - Node (if package.json): npm ci && npm test
  - Python (if requirements.txt or pyproject): python -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt && pytest -q
  - Go: go test ./...
- Expected outcomes and how to record: pass/fail summary, failing test names and stack traces, time taken.

Deliverables
- analysis/structure-report.md (Markdown) with the sections described in Task 9.
- A small attachments folder with these files (if allowed): cloc.json (if cloc run), workflow summaries, script outputs.

Estimated effort and assignees
- Quick scan (Tasks 1–4): 15–30 minutes for a small repo; 1–2 hours for medium/large.
- Full report (including Tasks 5–9): 1–3 hours depending on repo size.
- Assignee: a single read-only agent or a human reviewer.

Notes for sub-agent implementers
- Use the exact commands and file paths above. If a path does not exist, record that fact explicitly.
- Keep all output snippets short; for large files, capture only the first 200 lines or the most relevant sections.
- Do not install or run anything that modifies the repository without explicit permission.

Ready-to-run command sequence (ordered, copy/paste)
1) Top-level listing:
   find . -maxdepth 2 -mindepth 1 -print | sed 's|^./||' | sort > analysis/top-level.txt
2) Detect manifests/languages:
   for f in package.json pyproject.toml requirements.txt go.mod Cargo.toml pom.xml build.gradle Gemfile composer.json; do test -f $f && echo "$f: present"; done > analysis/manifests.txt
3) CI files:
   ls -la .github/workflows > analysis/workflows-listing.txt || true
4) Test discovery (example for Node/Python):
   test -f package.json && jq '.scripts.test' package.json > analysis/node-test-command.txt || true
   test -f requirements.txt && head -n 200 requirements.txt > analysis/requirements-head.txt || true
5) Run cloc if available:
   command -v cloc >/dev/null && cloc --json . > analysis/cloc.json || true
6) Compile report: create analysis/structure-report.md with the findings above.

End of plan.
