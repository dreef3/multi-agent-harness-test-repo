Implementation Plan: Codebase Structure Analysis for E2E Test Repo 1

Overview

This plan defines a set of deterministic, self-contained tasks for an implementation
sub-agent to run against the repository (UUID: d4d1a0d6-7f00-4a1b-b2f4-67cdaa66afde,
URL: https://github.com/dreef3/multi-agent-harness-test-repo.git). Each task
must be runnable on a typical Linux CI environment with git, find/tree, cloc
(optional), jq, and common language toolchains available.

Deliverables (files created/updated in the repo root)

- docs/codebase-analysis.md — human-readable analysis and summary.
- diagnostics/codebase-summary.json — structured summary (JSON).

High-level tasks

Task 1 — Clone repository and checkout default branch
- Purpose: Acquire repository contents for analysis.
- Commands:
  - git clone https://github.com/dreef3/multi-agent-harness-test-repo.git repo
  - cd repo
  - git fetch --all --tags
  - git checkout main
- Output: repository checked out at ./repo
- Acceptance: exit code 0; repository contains .git

Task 2 — Produce top-level tree and file-listing
- Purpose: Capture the repository layout at multiple depths.
- Commands:
  - tree -a -L 2 || (find . -maxdepth 2 -print | sed 's|^./||') > analysis/top-level-tree.txt
  - find . -type f -name "*" -print > analysis/all-files.txt
- Output: analysis/top-level-tree.txt, analysis/all-files.txt
- Acceptance: files exist and are non-empty

Task 3 — Detect languages and size breakdown
- Purpose: Determine primary languages and approximate size.
- Commands (preferred):
  - cloc --json . > analysis/cloc.json
  If cloc is unavailable, fall back to a file-extension count:
  - python3 - <<'PY'
import os,sys,collections
ext=collections.Counter()
for root,_,files in os.walk('.'):
    for f in files:
        ext=os.path.splitext(f)[1].lower() or '<noext>'
        ext[0]
        ext[0]
PY
  (Instead: use find and awk to count extensions.)
- Output: analysis/cloc.json or analysis/extension-stats.txt
- Acceptance: JSON with language names and LOC or a text file with top 10 extensions

Task 4 — Identify package/manifest files and parse core metadata
- Purpose: Find package.json, pyproject.toml, requirements.txt, go.mod, Gemfile,
  Cargo.toml, setup.py, etc., and extract name/version/dependencies where possible.
- Commands (example):
  - find . -maxdepth 3 -type f \( -name package.json -o -name pyproject.toml -o -name requirements.txt -o -name go.mod -o -name Cargo.toml -o -name Gemfile -o -name setup.py \) -print > analysis/manifests.txt
  - For each manifest:
    - package.json: jq '{name: .name, version: .version, scripts: .scripts, dependencies: .dependencies, devDependencies: .devDependencies}' package.json > analysis/package.json.summary.json
    - pyproject.toml: parse toml with python (toml library) to extract [tool.poetry] or [project] keys into a small JSON
    - go.mod: head -n 20 go.mod > analysis/go.mod.head.txt
- Output: analysis/manifests.txt and per-manifest summaries under analysis/
- Acceptance: manifests list produced; at least one manifest parsed if present

Task 5 — Identify CI, workflows, and automation
- Purpose: Locate CI config files (GitHub Actions, GitLab CI, CircleCI, Travis, Azure).
- Commands:
  - find . -path './.github/workflows/*' -type f -print > analysis/ci-files.txt || true
  - find . -maxdepth 3 -type f \( -name .gitlab-ci.yml -o -name circle.yml -o -name azure-pipelines.yml -o -name .travis.yml \) -print >> analysis/ci-files.txt || true
- Output: analysis/ci-files.txt
- Acceptance: File lists CI files or is empty (explicitly empty allowed)

Task 6 — Locate tests, examples, and docs
- Purpose: Find common test directories (tests, specs), examples, README(s), and docs.
- Commands:
  - find . -type d \( -iname tests -o -iname test -o -iname examples -o -iname docs -o -iname spec -o -iname specs \) -print > analysis/test-and-doc-dirs.txt
  - grep -R "pytest\|unittest\|mocha\|jest\|go test\|cargo test" -n --binary-files=without-match || true > analysis/test-hints.txt
  - head -n 200 README.md > analysis/README.head.md || true
- Output: analysis/test-and-doc-dirs.txt, analysis/test-hints.txt, analysis/README.head.md
- Acceptance: Lists created and README captured if present

Task 7 — Summarize build scripts and developer workflow files
- Purpose: Identify Makefile, Dockerfile(s), build scripts, and developer tooling.
- Commands:
  - find . -maxdepth 3 -type f -name Makefile -o -name Dockerfile -o -name "Dockerfile.*" -o -name "build.sh" -o -name "ci.sh" -print > analysis/build-and-dev-files.txt
- Output: analysis/build-and-dev-files.txt
- Acceptance: list exists

Task 8 — Produce diagnostics/codebase-summary.json
- Purpose: Aggregate findings into a machine-readable JSON file.
- Schema (example):
  {
    "repo": "d4d1a0d6-7f00-4a1b-b2f4-67cdaa66afde",
    "url": "https://github.com/dreef3/multi-agent-harness-test-repo.git",
    "top_level_dirs": ["src","docs","tests"],
    "languages": {"Python": 45, "JavaScript": 35},
    "manifests": [{"path":"package.json","type":"node","name":"...","version":"..."}],
    "ci_files": [".github/workflows/ci.yml"],
    "test_dirs": ["tests"],
    "build_files": ["Makefile","Dockerfile"],
    "notes": "brief note about monorepo, submodules, or other observations"
  }
- Commands: create a small Python script that reads the analysis/* files and emits the JSON at diagnostics/codebase-summary.json
- Output: diagnostics/codebase-summary.json
- Acceptance: valid JSON parseable by jq; includes keys above

Task 9 — Write human-readable report: docs/codebase-analysis.md
- Purpose: Produce a clear narrative report with findings and recommended next steps.
- Required sections:
  - Title and repository metadata (URL, default branch)
  - Top-level layout (brief list and description of top-level directories)
  - Languages and sizes (summary of cloc or fallback)
  - Manifests and dependency managers (list and brief contents)
  - CI and automation (list and short description)
  - Tests and coverage (where tests live, how to run them if obvious)
  - Build and run instructions discovered
  - Recommendations and suggested follow-ups
- Commands: write the Markdown file by assembling snippets from analysis/* and diagnostics JSON
- Output: docs/codebase-analysis.md
- Acceptance: file exists and contains the listed sections

Task 10 — Commit the artifacts
- Purpose: Persist artifacts to the repository for review.
- Commands:
  - git checkout -b ci/codebase-analysis
  - mkdir -p docs diagnostics analysis
  - cp analysis/* docs/ or add directly the generated files into docs and diagnostics
  - git add docs/codebase-analysis.md diagnostics/codebase-summary.json analysis/*
  - git commit -m "chore: add automated codebase structure analysis"
  - git push --set-upstream origin ci/codebase-analysis
- Output: branch pushed with artifacts
- Acceptance: push succeeded; branch exists on remote

Execution order and parallelization
- Tasks 1->2->3->4->5->6->7 must run sequentially because each depends on the repo checkout.
- Tasks 8 and 9 can run after tasks 2-7 complete (they aggregate artifacts).
- Task 10 runs last.

Tools, permissions, and environment
- Requires: git, python3 (with toml and json libraries), jq (optional), cloc (optional), tree or find,
  and network access to clone the repo.
- The agent executing these tasks must have permission to push a new branch to the repository (or prepare artifacts for a PR).

Estimated effort
- Automated execution: ~5–15 minutes depending on repo size and network speed.
- Human review of the generated report: ~5–15 minutes.

Acceptance criteria (summary)
- docs/codebase-analysis.md and diagnostics/codebase-summary.json are present in a new branch and contain the sections/keys described.
- The analysis is accurate with respect to file listings and manifests found.

Notes and follow-ups
- If the repository is large (>100k files), adjust the tree depth and use sampling for cloc.
- If submodules are present, the agent should detect .gitmodules and optionally recurse.
- If the repo is private or requires credentials, the execution agent must be provided appropriate access.

End of plan.
