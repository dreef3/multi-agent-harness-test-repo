Implementation Plan: Task-by-task repository structure analysis for multi-agent-harness-test-repo

Repository: multi-agent-harness-test-repo
Repo UUID: eb462090-58cf-4e8e-8a34-8c16ce308297
Default branch: main

Overview
This plan gives a step-by-step set of tasks a sub-agent should execute to produce a definitive analysis of the repository layout and produce a short report. Each task lists exact file paths or command lines to run, the data to collect, success criteria, and an expected example of the findings to include in the final report.

General notes for the executor
- Use the repository checkout at the default branch (main).
- Use bash commands to enumerate files and read to inspect file contents.
- Keep outputs concise; capture only what’s needed for the structural report (file lists, key file contents, test script names, CI workflows).

Task 0 — Setup (manual)
- Clone the repo (agent will handle cloning). Confirm branch main is checked out.
- Working directory: repo root.
- Deliverable: confirmation that repository is available locally.

Commands (example, run in repo root)
- git status --porcelain --branch
- git rev-parse --abbrev-ref HEAD

Task 1 — Top-level inventory
Goal: Enumerate top-level files and folders and capture sizes and types.
Commands
- ls -la
- find . -maxdepth 2 -mindepth 1 -type d -print | sed 's|\./||g'
- find . -maxdepth 1 -type f -print
What to capture
- Top-level directories and files, and note presence of: README.md, LICENSE, package.json, pyproject.toml, go.mod, Cargo.toml, requirements.txt, setup.cfg, setup.py, pnpm-lock.yaml, yarn.lock, package-lock.json, Pipfile, Dockerfile, .github/
Success criteria
- A bullet list of top-level files and directories included in the final report.

Task 2 — Language and package manager detection
Goal: Identify primary languages and package managers used.
Commands
- Check for common manifest files (files in repo root):
  - cat package.json
  - cat pyproject.toml
  - cat go.mod
  - cat Cargo.toml
  - cat setup.py
  - cat requirements.txt
- If multiple manifests exist, list them and mark multi-language.
What to capture
- The file names found and the package manager(s) to use for installing dependencies.
Success criteria
- Clear statement: e.g., "Node.js (package.json) with npm/yarn/pnpm" or "Python (pyproject.toml + requirements.txt)".

Task 3 — Source directories and entry points
Goal: Locate main source folders and application entry points.
Commands
- Common folders to check: src/, lib/, app/, server/, backend/, frontend/, cmd/, pkg/ (for Go), crates/ (for Rust), tests/.
- find src -maxdepth 2 -type f -name '*.js' -or -name '*.ts' -or -name '*.py'  (modify if folder not present)
- If package.json exists, run: jq '.main, .scripts' package.json || true
What to capture
- Source directories and main files (e.g., src/index.js, src/main.py) and package.json scripts helpful for running the app.
Success criteria
- List of source directories and declared entry points or start/test scripts.

Task 4 — Tests and E2E suites
Goal: Identify unit and end-to-end test suites and how to run them.
Commands
- Check for directories: test/, tests/, e2e/, integration/.
- If package.json present: jq '.scripts.test' package.json || true
- Search for common test frameworks: ripgrep patterns for jest, mocha, pytest, unittest, playwright, cypress, playwright.config.js, cypress.json
  - rg "\b(jest|mocha|pytest|playwright|cypress|vitest)\b" --hidden || true
What to capture
- Names/paths of test suites, how to run (npm test, pytest, etc.), approximate count of tests (if quick scan possible), presence of CI e2e job.
Success criteria
- A short section in the report describing test commands and whether tests appear runnable locally.

Task 5 — Continuous Integration / GitHub Actions
Goal: Inspect .github/workflows for CI workflows and understand the checks.
Commands
- ls -la .github/workflows || true
- For each *.yml file: sed -n '1,200p' .github/workflows/*.yml
What to capture
- Workflow names, triggers (push/pull_request), jobs (build/test/e2e), required matrix platforms, and any secrets or artifacts referenced.
Success criteria
- A list of the CI workflow files and a short paragraph summarizing what they run and when.

Task 6 — Containerization and deployment infra
Goal: Find Dockerfiles, docker-compose, Kubernetes manifests, Terraform, or cloud config.
Paths to check
- Dockerfile (repo root), docker/*, docker-compose.yml, k8s/, charts/, terraform/
Commands
- rg "^FROM\b" Dockerfile* || true
- ls -la docker || true
What to capture
- Whether a Dockerfile exists, how to build, and any multi-stage build indications.
Success criteria
- Short note in report: "Dockerfile present — build with docker build -t repo:tag ." or "No container config found."

Task 7 — Scripts, tooling, and workspace config
Goal: Inventory scripts and developer tooling (Makefile, npm scripts, bin/, scripts/)
Commands
- cat Makefile || true
- If package.json: jq '.scripts' package.json || true
- ls -la scripts || true
What to capture
- Developer commands (make test, npm run dev), lint/format configs (eslint, prettier, black), and any contribution instructions.
Success criteria
- List of common developer commands and any missing or unusual tooling.

Task 8 — Documentation and examples
Goal: Check README.md, docs/, examples/ for usage and architecture docs.
Commands
- sed -n '1,200p' README.md || true
- ls -la docs || true
What to capture
- Presence and quality of README, quickstart instructions, and any architecture/design docs.
Success criteria
- Final report includes a README summary and any missing items (no quickstart, out-of-date docs).

Task 9 — Licensing and contribution files
Goal: Verify LICENSE, CODE_OF_CONDUCT, CONTRIBUTING.md, and .github/CODEOWNERS
Commands
- cat LICENSE || true
- cat CONTRIBUTING.md || true
What to capture
- License type and presence of contribution guidance.
Success criteria
- License identified and listed in the report.

Task 10 — Run tests and lint (optional, if requested)
Goal: Execute test suite and linters inside a controlled environment.
Commands (examples — run only if environment prepared)
- For Node: npm ci && npm test
- For Python: python -m venv .venv && .venv/bin/pip install -r requirements.txt && pytest -q
What to capture
- Test pass/fail, test logs, failing tests, and flake/lint warnings.
Success criteria
- Tests run to completion and logs captured; report indicates pass rate and failing tests.

Final deliverable
- A concise "Repository Structure Report" containing:
  - Top-level inventory (Task 1)
  - Languages and package managers (Task 2)
  - Source layout and entry points (Task 3)
  - Test suites and run commands (Task 4)
  - CI/workflow summary (Task 5)
  - Containerization and deployment infra (Task 6)
  - Developer tooling and scripts (Task 7)
  - Documentation quality (Task 8)
  - License and contribution files (Task 9)
  - (Optional) Test & lint results (Task 10)

Format for the report (one-page summary + appendices):
- One-page summary bullets with quick actionable items and any blockers.
- Appendix A: verbatim file lists for top-level and src dirs (truncated as needed).
- Appendix B: extracted key files (package.json scripts, CI workflow names) truncated to 200 lines each.

Estimated effort and ordering
- Tasks 1–5: 30–60 minutes
- Tasks 6–9: 15–30 minutes
- Task 10 (running tests): 30–90 minutes depending on environment and CI setup

Acceptance criteria
- The report must allow an engineer unfamiliar with the repository to: (a) identify how to install dependencies, (b) run the app and tests, and (c) run CI locally or understand what CI checks run.

Post-plan
- After the plan is committed and merged, use dispatch_tasks to assign implementing agents to run the tasks described above and produce the actual report and logs.

Notes for the integrator
- This plan intentionally avoids implementing fixes; it focuses purely on discovery and reporting. If follow-up remediation is requested, produce separate task descriptions for code changes.
