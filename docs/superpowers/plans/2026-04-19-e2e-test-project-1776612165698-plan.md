Implementation Plan: Task-by-task Codebase Structure Analysis

Context
- Repository: multi-agent-harness-test-repo
- Repository ID: f0511a0a-da77-405f-aa54-92cd9514187b
- Workdir: assume root of repository when commands are executed
- Output location: ./analysis/

High-level sequence
1) Gather a curated file tree and top-level listing
2) Detect languages and manifests
3) Locate CI and Docker artifacts
4) Locate tests and testing frameworks
5) Identify entry points and common scripts
6) Parse manifests for dependencies and lockfiles
7) Produce JSON summary and Markdown report with recommendations

Detailed tasks (self-contained, runnable locally in repo root)

Task 1 — Collect file tree and top-level layout
- Purpose: quick visual of repository structure and top-level directories
- Commands to run:
  - mkdir -p analysis/artifacts
  - find . -path ./\.git -prune -o -type d -maxdepth 2 -print | sed 's#^\./##' > analysis/artifacts/top_dirs.txt
  - find . -path ./\.git -prune -o -type f -not -path './node_modules/*' -not -path './venv/*' -not -path './.venv/*' -maxdepth 3 -print | sed 's#^\./##' > analysis/artifacts/file_tree.txt
- Expected output files:
  - analysis/artifacts/top_dirs.txt
  - analysis/artifacts/file_tree.txt
- Notes: limit maxdepth to 3 to keep outputs readable; deeper scanning done in later tasks if needed.

Task 2 — Language and manifest detection
- Purpose: determine primary languages and presence of manifests
- Commands to run (examples):
  - rg --hidden --files --glob '!node_modules' --glob '!.git' '\.(py|js|ts|go|rs|java|kt|php|rb|cs)$' -n || true
  - for manifest in package.json pyproject.toml setup.py requirements.txt Pipfile poetry.lock go.mod Cargo.toml Gemfile composer.json composer.lock; do [ -f "$manifest" ] && echo "$manifest" >> analysis/artifacts/manifests_found.txt || true; done
  - awk 'NR<=200{print FILENAME":"NR":"$0}' $(ls -1 */package.json 2>/dev/null || true) > /dev/null || true
- Expected output files:
  - analysis/artifacts/manifests_found.txt (list of found manifests)
  - analysis/artifacts/language_summary.json (see next task for generation)

Task 3 — Create a language summary (machine-friendly)
- Purpose: summarize by file extension counts and manifest presence
- Approach/commands:
  - (a) Count common file extensions:
    - bash -lc "printf '{' > analysis/artifacts/language_summary.json; echo '\"stats\":{' >> analysis/artifacts/language_summary.json; for ext in py js ts go rs java kt php rb cs; do cnt=$(rg --hidden --files --glob '!node_modules' --glob '!.git' -g "**/*.$ext" -n --silent | wc -l); printf '\"%s\":%s,' \"$ext\" $cnt >> analysis/artifacts/language_summary.json; done; sed -i 's/,$/},/' analysis/artifacts/language_summary.json || true"
  - (b) Append detected manifests into the JSON by reading analysis/artifacts/manifests_found.txt
- Expected output:
  - analysis/artifacts/language_summary.json
- Notes: If ripgrep (rg) isn't available, use find + grep as fallback.

Task 4 — Detect CI/CD configurations
- Purpose: find workflow and CI config files
- Commands:
  - rg -n --hidden --glob '!.git' "^" -g ".github/workflows/*" || true > analysis/artifacts/ci_files.txt
  - [ -f .gitlab-ci.yml ] && echo ".gitlab-ci.yml" >> analysis/artifacts/ci_files.txt || true
  - [ -f circle.yml ] && echo "circle.yml" >> analysis/artifacts/ci_files.txt || true
- Expected output:
  - analysis/artifacts/ci_files.txt

Task 5 — Detect Docker artifacts and infra files
- Purpose: list Dockerfiles and docker-compose files
- Commands:
  - rg -n --hidden --glob '!.git' "^" -g "**/Dockerfile" || true > analysis/artifacts/docker_files.txt
  - rg -n --hidden --glob '!.git' "docker-compose.yml" || true >> analysis/artifacts/docker_files.txt
- Expected output:
  - analysis/artifacts/docker_files.txt

Task 6 — Locate tests and determine test frameworks
- Purpose: find tests, classify unit/integration/e2e
- Commands:
  - rg -n --hidden --glob '!.git' "test_.*\.py|.*_test\.py|__tests__|.*\.spec\.ts|.*\.test\.ts|.*\.spec\.js|.*\.test\.js" || true > analysis/artifacts/tests_list.txt
  - look for common test runners: presence of jest.config.js, pytest.ini, tox.ini, jest in package.json scripts, actions that run npm test or pytest
  - Extract package.json scripts: [ -f package.json ] && jq -r '.scripts' package.json > analysis/artifacts/package_scripts.json || true
- Expected output:
  - analysis/artifacts/tests_list.txt
  - analysis/artifacts/package_scripts.json (if applicable)

Task 7 — Identify entry points and common scripts
- Purpose: find top-level executable scripts, CLIs, or server startup files
- Commands:
  - rg -n --hidden --glob '!.git' "^#!/.*" || true > analysis/artifacts/shebangs.txt
  - [ -f package.json ] && jq -r '.main, .bin' package.json > analysis/artifacts/package_entrypoints.txt || true
  - Search for 'if __name__ == "__main__"' occurrences: rg "if __name__ == \"__main__\"" || true > analysis/artifacts/py_entrypoints.txt
- Expected output:
  - analysis/artifacts/shebangs.txt
  - analysis/artifacts/package_entrypoints.txt
  - analysis/artifacts/py_entrypoints.txt

Task 8 — Parse manifests for dependencies and lockfiles
- Purpose: gather dependency files for quick assessment
- Commands/examples:
  - If package.json exists: jq '{name: .name, version: .version, dependencies: .dependencies, devDependencies: .devDependencies, scripts: .scripts}' package.json > analysis/artifacts/package_json_summary.json || true
  - If pyproject.toml exists: copy to artifacts and optionally extract [tool.poetry] or [project]
  - If requirements.txt exists: cp requirements.txt analysis/artifacts/requirements.txt || true
  - For go.mod, Cargo.toml, Gemfile, etc: copy into artifacts
- Expected output:
  - analysis/artifacts/*_summary.json or copies of manifest files

Task 9 — Produce human-readable report and recommendations
- Purpose: synthesize findings into analysis/codebase_report.md
- Content to include:
  - Executive summary: primary languages, size (file counts), presence/absence of README/LICENSE, CI, Docker, tests
  - Detailed sections: Directory layout, language summary (from JSON), CI and Docker, tests, entrypoints, dependency health (presence of lockfiles), recommended next actions
  - Attach paths to artifact files for inspection
- Template (high level):
  - Title and repository id
  - Methodology (commands used)
  - Findings (bulleted)
  - Recommendations (short-term and longer-term)
- Command to assemble: a small shell script or here-doc that concatenates selected artifact files and renders a Markdown file at analysis/codebase_report.md

Task 10 — Validate and finalize
- Purpose: ensure artifacts exist and are non-empty, and update timestamps
- Commands:
  - for f in analysis/artifacts/*; do echo "---- $f ----" >> analysis/codebase_report.md; sed -n '1,200p' "$f" >> analysis/codebase_report.md; done
  - echo "Report generated at: $(date -u)" >> analysis/codebase_report.md

Notes, assumptions, and fallbacks
- The plan uses ripgrep (rg) and jq where available. If not installed, fallback to find/grep and python -c to parse JSON.
- Commands avoid modifying source files and avoid running tests or builds.
- For large repositories, consider increasing maxdepths or performing targeted deeper scans on directories like src/ or packages/.

Estimated timeline
- Single-pass automation: 10–20 minutes for small-to-medium repo. Large monorepos may take 30+ minutes and may require batching.
