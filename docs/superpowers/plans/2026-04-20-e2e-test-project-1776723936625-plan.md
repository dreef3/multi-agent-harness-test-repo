Implementation Plan: Task-by-task codebase analysis

Overview

This plan defines a sequence of tasks that an implementer or automated sub-agent will run against the E2E Test Repo to generate CODEBASE_ANALYSIS.md. Each task lists exact commands, files to inspect, expected outputs and acceptance criteria. No implementation code will be added to the repository by this plan — only the analysis report file will be created and committed via the harness workflow.

Repository

- URL: https://github.com/dreef3/multi-agent-harness-test-repo.git
- Branch: main

Deliverable

- File: CODEBASE_ANALYSIS.md (root of repository)
- Contents: summary, language & framework breakdown, top-level file/directory tree, manifests, build/test commands, CI and containerization summary, tests and coverage, risk areas, recommended follow-ups.

Tasks

Task 1 — Clone and setup (manual or CI-run)
- Purpose: get a local copy to analyze
- Commands:
  - git clone https://github.com/dreef3/multi-agent-harness-test-repo.git
  - cd multi-agent-harness-test-repo
  - git checkout main
- Acceptance: repo checked out locally

Task 2 — Top-level inventory
- Purpose: list top-level files/dirs and quick stats
- Commands:
  - ls -la
  - find . -maxdepth 2 -type d -print | sed 's/^/DIR: /'
  - wc -l $(find . -maxdepth 2 -type f)
- Files to capture: README.md, LICENSE, .gitignore, CODE_OF_CONDUCT.md, CONTRIBUTING.md (if present)
- Acceptance: a short list of top-level items recorded in report

Task 3 — Language and manifest detection
- Purpose: detect primary languages and package manifests
- Commands (order matters):
  - test -f package.json && echo "node" || true
  - test -f pyproject.toml && echo "python" || true
  - test -f requirements.txt && echo "python-reqs" || true
  - test -f go.mod && echo "go" || true
  - test -f Cargo.toml && echo "rust" || true
  - test -f pom.xml && echo "java-maven" || true
  - test -f build.gradle && echo "java-gradle" || true
  - test -f composer.json && echo "php" || true
  - test -f Gemfile && echo "ruby" || true
  - grep -Iq "^#!" || true
- Files to inspect: package.json, pyproject.toml, go.mod, Cargo.toml, pom.xml, build.gradle
- Acceptance: list of manifests and primary language(s) identified

Task 4 — Examine README and developer docs
- Purpose: extract stated build/test commands and purpose
- Commands:
  - sed -n '1,200p' README.md
  - rg "build|test|run|docker|compose|how to" || true
- What to record: declared setup steps, examples, architecture diagrams
- Acceptance: README-derived build/test/run steps included in report

Task 5 — Inspect CI and workflows
- Purpose: identify continuous integration pipelines and matrix
- Files to inspect:
  - .github/workflows/*.yml
  - .gitlab-ci.yml
  - azure-pipelines.yml
  - Jenkinsfile
- Commands:
  - ls -la .github/workflows || true
  - sed -n '1,240p' .github/workflows/*.yml || true
- What to record: steps, build matrix, test commands, coverage uploads
- Acceptance: CI summary recorded with references to workflow files

Task 6 — Containerization and deployment artifacts
- Purpose: find Dockerfile, docker-compose.yml, k8s manifests
- Commands:
  - test -f Dockerfile && sed -n '1,200p' Dockerfile || true
  - test -f docker-compose.yml && sed -n '1,200p' docker-compose.yml || true
  - find . -type f -name "*.yaml" -o -name "*.yml" | rg -i "k8s|deployment|service" || true
- Acceptance: containerization and deployment artifacts summarized

Task 7 — Source layout and entry points
- Purpose: identify main source dirs, binaries, microservices
- Files/dirs to inspect: src/, lib/, pkg/, cmd/, app/, main.go, index.js
- Commands:
  - tree -L 2 || (find . -maxdepth 2 -print)
  - rg "module.exports|package main|if __name__ == '__main__'|func main\(" -n || true
- Acceptance: list of primary source directories and entrypoints

Task 8 — Tests, coverage, and quality tools
- Purpose: find tests, test runners, linters, CI badges
- Files/dirs: test/, tests/, spec/, jest.config.js, pytest.ini, .coverage
- Commands:
  - find . -type d -name "test*" -print
  - rg "pytest|unittest|jest|mocha|go test|cargo test|mvn test|gradle test" -n || true
- Acceptance: test frameworks and locations recorded; existence of coverage reports noted

Task 9 — Dependency and security review
- Purpose: quickly surface large/outdated dependencies and advisories
- Commands:
  - For Node: jq '.dependencies + .devDependencies' package.json || true
  - For Python: pip-compile? (optional) or list requirements.txt
  - For Go: cat go.mod
  - Suggest running dependency scanning tools (npm audit, pip-audit, snyk)
- Acceptance: manifest snapshots included and recommendations for running audits

Task 10 — Produce CODEBASE_ANALYSIS.md
- Purpose: assemble all collected data into a human-readable report
- File path: ./CODEBASE_ANALYSIS.md
- Required sections:
  - Title and report generation date
  - Short summary (1-3 paragraphs)
  - Language and manifests (evidence lines showing filenames/contents)
  - Top-level tree (2 levels)
  - Key files and where to look for build/test/entrypoints
  - CI summary and containerization summary
  - Tests, coverage, and quality tools
  - Risk hotspots and maintenance recommendations
  - Next steps and suggested tickets
- Example snippets to include in the file (use actual captured command outputs). Use triple-backticks blocks for small excerpts (<=20 lines).
- Acceptance criteria: file exists, contains all required sections and at least one evidence excerpt from the repo.

Task 11 — Review and PR
- Purpose: run a quick review and open a PR with the report
- Steps:
  - git checkout -b analysis/codebase-report
  - Add CODEBASE_ANALYSIS.md, commit and push
  - Open PR via harness (the harness will handle the PR creation automatically). Include summary and link to this plan.
- Acceptance: PR created with a single file containing the report

Optional Task 12 — Automation (sub-agent)
- Purpose: implement an automated job (script) that runs the above commands and writes CODEBASE_ANALYSIS.md. This is useful for recurring audits.
- Location suggestion: scripts/codebase-analysis.sh (executable)
- Outline of script behavior: run detection commands, capture outputs to a temp folder, render template into CODEBASE_ANALYSIS.md, exit non-zero on unexpected failures.
- Note: Adding such a script is optional and should be reviewed before committing.

Estimated effort

- Manual run by an engineer: 30–90 minutes depending on repository size
- Automated script creation & validation: 2–4 hours

Acceptance criteria (final)

- CODEBASE_ANALYSIS.md present in repo root.
- Report contains the required sections and evidence excerpts.
- At least one CI workflow and one manifest file referenced if present in the repo.

Notes and constraints

- This plan intentionally avoids running test suites or making changes other than adding the report file.
- If the repository is large (>50K files), use find with pruning (e.g., -path './node_modules' -prune).
