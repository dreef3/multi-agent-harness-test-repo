Implementation plan: step-by-step tasks to analyze the repo structure and produce the deliverables. All commands assume a POSIX shell and the repository cloned to ./multi-agent-harness-test-repo (git clone https://github.com/dreef3/multi-agent-harness-test-repo.git).

Deliverables
- report/summary.md — human-readable one-page summary
- report/manifest.json — machine-readable manifest of key files, languages, test commands, CI files
- report/file-tree.txt — top-level file tree (depth 2 or 3)
- report/actions/ — extracted CI workflow snippets (if any)

Task 1 — Clone and baseline listing (1-2m)
- Commands:
  - git clone https://github.com/dreef3/multi-agent-harness-test-repo.git
  - cd multi-agent-harness-test-repo
  - git rev-parse --abbrev-ref HEAD  # confirm branch
  - ls -la
  - tree -a -L 2 > ../report/file-tree.txt  # or find . -maxdepth 2 -print

Files to capture: README.md, LICENSE, CONTRIBUTING.md, CODE_OF_CONDUCT.md

Task 2 — Language and size breakdown (2-5m)
- Commands (choose available tool):
  - cloc --yaml . > ../report/cloc.yml
  - scc --by-file --output-type json . > ../report/scc.json
  - If neither installed, use: rg --files | sed -E 's/.*\.([a-z0-9]+)$/\1/' | sort | uniq -c

Task 3 — Dependency manifest (3-8m)
- Look for common dependency files at repo root and common subdirs:
  - package.json, yarn.lock, pnpm-lock.yaml
  - requirements.txt, Pipfile, Pipfile.lock, pyproject.toml, setup.py
  - go.mod, go.sum
  - Cargo.toml, Cargo.lock
  - Gemfile, Gemfile.lock
  - Gemfile
  - build.gradle, gradle.properties
- Commands examples:
  - rg -n "^name|^version|^dependencies|^devDependencies|^requires" --hidden --glob '!node_modules' package.json || true
  - ls -1 package.json pyproject.toml requirements.txt go.mod Cargo.toml Gemfile 2>/dev/null || true
- For each found file, copy or extract the dependencies list into report/ (e.g., report/deps/package.json.json)

Task 4 — CI/CD and workflows (2-5m)
- Inspect .github/workflows, .gitlab-ci.yml, .circleci/config.yml, Jenkinsfile, Dockerfile(s), and Makefile.
- Commands:
  - rg -n "jobs:|name:|runs-on:|dockerfile|uses:|steps:" .github/workflows || true
  - find .github/workflows -type f -maxdepth 1 -print -exec sed -n '1,200p' {} \; > ../report/actions/_workflows_snippets.txt
- Extract test steps and commands from these workflow files into report/actions/workflow-tests.txt

Task 5 — Test discovery and commands (3-10m)
- Look for test directories and common test commands in package.json scripts, pyproject[tool.pytest], tox.ini, setup.cfg, jest.config.js, tests/, spec/, __tests__/.
- Commands:
  - rg -n "\b(test|pytest|jest|mocha|go test|cargo test)\b" --hidden || true
  - jq -r '.scripts // {}' package.json 2>/dev/null || true
- Record discovered test commands into report/manifest.json under tests[] with the command and path.

Task 6 — Static checks and quick risks (5-15m)
- Check for missing/incomplete metadata: no README, no LICENSE, no CI.
- Check for pinned versions vs ranges in dependency files.
- Commands:
  - test -f README.md || echo MISSING_README
  - rg -n "[~^]?[0-9]+\.[0-9]+\.[0-9]+" package.json || true
- Add a short findings.md listing potential quick fixes.

Task 7 — Create final outputs (5-10m)
- report/summary.md: concise overview — top directories, languages, key dependency files, CI presence, test command(s), risks, recommended next steps.
- report/manifest.json: JSON with fields: repo, branch, files: {path: sha1? optional}, languages: {name: bytes/lines}, dependency_files: [], ci_files: [], test_commands: [], recommendations: []
- report/file-tree.txt, report/actions/* as collected.

Examples of expected manifest.json structure
{
  "repo": "dreef3/multi-agent-harness-test-repo",
  "branch": "main",
  "languages": {"Python": 1200, "JavaScript": 800},
  "top_level_dirs": ["src","tests","docs"],
  "dependency_files": ["package.json","pyproject.toml"],
  "ci_files": [".github/workflows/ci.yml"],
  "test_commands": [{"path":"package.json","cmd":"npm test"}],
  "recommendations": ["Run CI tests locally","Add LICENSE"]
}

Task 8 — Review and handoff (10-20m)
- Manual review of the generated artifacts and short write-up of recommended next steps prioritized by impact and effort.

Estimated effort: 30–60 minutes for a single-engineer static pass depending on repo size. If repo is very large (>5000 files), increase time estimate.

Notes and constraints
- This is a static analysis pass only; do not run tests or build steps that change state (unless explicitly requested). If test execution is requested later, create separate tasks to run in isolated CI or ephemeral containers.
- Keep all output inside report/ in repository root so it can be reviewed and committed.
