Implementation Plan: Task-by-task plan to perform repository structure analysis

Context
- Repository: E2E Test Repo
- Repository ID: 8d4da9fc-ff5d-448f-a5fe-9373bfeaa4ca
- Goal: Implement read-only analysis that outputs analysis/REPORT.md and analysis/summary.json

High-level strategy
- Create a small set of scripts and automation that examine the filesystem and common manifest files, collect metrics (file counts, LOC per language via cloc or fallback to simple heuristics), identify CI, tests, Dockerfiles, and produce a structured JSON plus a readable report.
- Break the work into self-contained tasks so sub-agents can run them in parallel after the planning PR is merged.

Tasks (each is self-contained and references repositoryId above)

Task 1 — Repository inventory (repo root scan)
- Purpose: list top-level files and directories, capture repository size and list of large files.
- Files/commands to run:
  - Run: ls -la
  - Run: find . -maxdepth 2 -type d -printf "%p\n" | sort
  - Run: find . -type f -printf "%s %p\n" | sort -nr | head -n 30
  - Detect large files > 5MB and list them
- Outputs:
  - analysis/inventory.txt (raw command outputs)
  - JSON snippet with top-level dirs and large-files array

Task 2 — Language and LOC detection
- Purpose: detect primary languages and approximate lines of code per language.
- Approach: prefer cloc if available; otherwise use file extension heuristics (wc -l on found files) and a mapping of extensions to languages.
- Commands/examples:
  - cloc --json . > analysis/cloc.json  (if cloc installed)
  - fallback: use find and awk to map extensions to counts, e.g., find . -type f -name "*.*" -print0 | xargs -0 awk 'END{print NR}' on each file grouped by extension
- Outputs:
  - analysis/cloc.json or analysis/language_summary.json

Task 3 — Dependency and manifest discovery
- Purpose: find package manager manifests and extract top-level metadata (name, version, dependencies count)
- Files to inspect (if present): package.json, package-lock.json, yarn.lock, pnpm-lock.yaml, pyproject.toml, requirements.txt, Pipfile, Pipfile.lock, setup.py, go.mod, Cargo.toml, Gemfile, Gemfile.lock, pom.xml, build.gradle, build.gradle.kts
- Commands/examples:
  - For Node: jq '.name, .version, .dependencies, .devDependencies' package.json
  - For Python: parse pyproject.toml with python -c 'import tomllib; print(...)'
- Outputs:
  - analysis/manifests/ (copy of detected manifests)
  - analysis/manifests_summary.json — an entry per manifest with parsed metadata and counts

Task 4 — CI / workflow detection
- Purpose: detect CI configurations and identify main pipeline files and triggers
- Files to look for: .github/workflows/*.yml, .gitlab-ci.yml, .circleci/config.yml, .travis.yml, azure-pipelines.yml
- Commands:
  - ls .github/workflows || true
  - For each workflow YAML, output the top-level keys and job names (yq or python yaml safe_load)
- Outputs:
  - analysis/ci/ (copies of workflow files)
  - analysis/ci_summary.json with workflow metadata

Task 5 — Test discovery and test commands
- Purpose: find test folders and common test runners (jest, pytest, go test, cargo test, mvn test)
- Places to check: tests/, test/, src/**/test, package.json scripts.test, pyproject tool.poetry.scripts/tests
- Commands:
  - grep -R "pytest\|jest\|mocha\|ava\|go test\|cargo test\|mvn test" -n || true
  - Check package.json scripts.test field
- Outputs:
  - analysis/tests_summary.json and list of test files found

Task 6 — Docker / containers
- Purpose: identify Dockerfile(s) and docker-compose files and summarize base images and build steps
- Files: Dockerfile, **/Dockerfile, docker-compose.yml, docker-compose.*.yml
- Commands:
  - grep -n "FROM" **/Dockerfile || true
  - Parse docker-compose for services and images
- Outputs:
  - analysis/docker_summary.json and copies of Dockerfiles

Task 7 — Documentation, licensing, and contributor info
- Purpose: locate README*, docs/, CHANGELOG*, LICENSE, CODE_OF_CONDUCT.md, CONTRIBUTING.md
- Commands:
  - ls README* docs LICENSE* || true
  - Extract first 200 lines of README and LICENSE
- Outputs:
  - analysis/docs_excerpt.md and analysis/license.txt

Task 8 — Potential red flags and recommendations
- Purpose: flag issues like missing LICENSE, no CI, no tests, very large files, and insecure-looking manifests (e.g., gitignored lockfiles missing)
- Approach: apply simple heuristics from the collected outputs and produce a prioritized recommendations list
- Outputs:
  - analysis/REPORT.md (narrative), analysis/summary.json (structured), analysis/actions.md (recommended next steps)

Task 9 — Orchestration script and output formatting
- Purpose: produce scripts/analyze_repo.sh that runs the above tasks and writes outputs to analysis/.
- Key requirements for the script:
  - Read-only; no network access
  - Works in POSIX shell with common utilities (find, grep, sed, awk, jq, python)
  - Exit code 0 on success
- Example top-level flow in script (pseudocode):
  - mkdir -p analysis
  - run Task1 -> append to analysis/inventory.txt and JSON
  - run Task2 -> write analysis/cloc.json or analysis/language_summary.json
  - run Task3..Task8 in sequence and produce final analysis/REPORT.md and analysis/summary.json

Task 10 — Review, QA, and PR notes
- Purpose: provide guidance for reviewers to run the script locally and validate outputs.
- Include in plan: exact commands to run locally (bash scripts/analyze_repo.sh), expected artifacts (list), and checks (exit code, presence of files).

Deliverables (what will be added in implementation PRs)
- scripts/analyze_repo.sh (or scripts/python/analyze_repo.py)
- analysis/README.md (how to run, expected outputs)
- analysis/.gitignore (if needed to ignore large analysis files)
- sample output files in analysis/ (only if generated by CI; do not add sample outputs in the plan commit)

Timeline and parallelization
- These tasks can be executed in parallel by sub-agents after the planning PR is merged. Each task is self-contained and produces files under analysis/ or scripts/.

Acceptance criteria
- Running scripts/analyze_repo.sh at repo root produces analysis/REPORT.md and analysis/summary.json.
- REPORT.md contains a clear, actionable summary and pointers to exact files where issues were found.

Notes and constraints
- The planning PR will only contain this spec and implementation plan. Actual script and analysis files will be implemented in follow-up tasks.
- No destructive actions, no network calls, no credential use.
