Implementation Plan: Repository Structural Analysis for E2E Test Repo (repo id: 6ed2cd10-da8f-48a5-9543-b070a7487fea)

Overview
This plan breaks the analysis into concrete, self-contained tasks an implementation agent can run. Each task contains exact commands to run, files to inspect, expected outputs, and acceptance criteria. The plan assumes a POSIX environment with git, tree/find, ripgrep (rg) or grep, jq, and cloc (or cloc alternative) installed. If cloc is not available, wc -l and file-extension heuristics are provided as fallback.

Workspace
- Clone URL: https://github.com/dreef3/multi-agent-harness-test-repo.git
- Local checkout path used in commands below: ./multi-agent-harness-test-repo
- Output directory for artifacts: ./analysis/multi-agent-harness-test-repo-analysis

Tasks

Task 1 — Clone repository and capture top-level tree
- Goal: Get a baseline snapshot of repository layout and top-level files.
- Commands:
  - git clone --depth 1 https://github.com/dreef3/multi-agent-harness-test-repo.git ./multi-agent-harness-test-repo
  - cd ./multi-agent-harness-test-repo
  - tree -a -L 2 > ../analysis/multi-agent-harness-test-repo-analysis/top-tree.txt || find . -maxdepth 2 -print > ../analysis/multi-agent-harness-test-repo-analysis/top-tree.txt
  - ls -la > ../analysis/multi-agent-harness-test-repo-analysis/top-ls.txt
- Files produced:
  - analysis/multi-agent-harness-test-repo-analysis/top-tree.txt
  - analysis/multi-agent-harness-test-repo-analysis/top-ls.txt
- Acceptance: top-tree.txt lists top-level directories and files; top-ls.txt contains timestamps and permissions.

Task 2 — Identify primary languages and LOC
- Goal: Determine language breakdown and lines-of-code.
- Commands (preferred):
  - cloc --yaml --out=../analysis/multi-agent-harness-test-repo-analysis/cloc.yaml .
  - cloc --csv --out=../analysis/multi-agent-harness-test-repo-analysis/cloc.csv .
- Fallback if cloc missing:
  - find . -type f \( -name "*.py" -o -name "*.js" -o -name "*.ts" -o -name "*.go" -o -name "*.java" -o -name "*.rs" -o -name "*.c" -o -name "*.cpp" -o -name "*.cs" -o -name "*.sh" \) -print0 | xargs -0 wc -l > ../analysis/multi-agent-harness-test-repo-analysis/loc-by-file.txt
  - Summarize by extension using awk to produce a small JSON.
- Files produced: cloc.yaml and cloc.csv or loc-by-file.txt and a derived summary JSON.
- Acceptance: A clear top-3 languages by LOC are identified and recorded.

Task 3 — Detect dependency manifests and summarize dependencies
- Goal: Find and summarize dependency manifests.
- Commands:
  - rg --hidden "^(name: |dependencies:|package:|from: |^plugins:|^dependencies:|^requirements.txt$)" || true
  - List common manifest files and copy them into analysis dir if present:
    - cp package.json ../analysis/.../ || true
    - cp package-lock.json yarn.lock pnpm-lock.yaml ../analysis/.../ || true
    - cp pyproject.toml requirements.txt setup.py Pipfile ../analysis/.../ || true
    - cp go.mod Gopkg.toml ../analysis/.../ || true
    - cp Cargo.toml ../analysis/.../ || true
    - cp Gemfile Gemfile.lock ../analysis/.../ || true
- Additionally run language-specific tooling for summary (optional):
  - For Node.js: jq '.dependencies,.devDependencies' package.json > ../analysis/...
  - For Python: parse pyproject.toml or requirements.txt and list package names.
- Files produced: analysis copies of manifests and a short dependencies-summary.md.
- Acceptance: All detected manifests are recorded with a short list (top-level dependencies and versions).

Task 4 — Identify build, run, and test commands
- Goal: Find how the project is built and tested (NPM scripts, Makefile targets, tox, pytest, github workflows).
- Commands:
  - rg "scripts\":|pytest|tox|unittest|make|docker build|go build|mvn|gradle|gradlew" -n || true
  - If package.json present: jq '.scripts' package.json > ../analysis/.../scripts.json
  - If Makefile present: sed -n '1,200p' Makefile > ../analysis/.../Makefile-snapshot
  - Search for usage of test frameworks and copy relevant files.
- Files produced: scripts.json, Makefile-snapshot, tests-summary.md
- Acceptance: All build/test entrypoints are enumerated and at least one command for running tests is found or explicitly noted as missing.

Task 5 — Inspect CI and Docker files
- Goal: List CI workflows and Dockerfiles, and summarize their purposes.
- Commands:
  - ls -la .github/workflows || true
  - for f in .github/workflows/*.yml .github/workflows/*.yaml; do echo "--- $f"; yq e . $f 2>/dev/null || cat $f; done > ../analysis/.../ci-workflows.txt
  - rg -n "FROM |docker build|docker run|image: " . || true
  - cp Dockerfile ./analysis/.../Dockerfile || true
- Files produced: ci-workflows.txt, Dockerfiles copied.
- Acceptance: The analysis lists each workflow file and a short description of what it does (build, test, release).

Task 6 — Detect tests and coverage
- Goal: Enumerate test directories and compute test file counts; check for coverage config (coverage.py, pytest.ini, jest.config.js).
- Commands:
  - rg --hidden "describe\(|it\(|pytest\.|unittest|@Test|mocha|jest" -n tests || true
  - find . -type f -iname "*test*.py" -o -iname "*_spec.js" -o -iname "*.spec.ts" -o -iname "*Test.java" > ../analysis/.../test-files.txt
  - Check for coverage config files: ls coverage.* pytest.ini .coveragerc jest.config.js nyc.config.js || true
- Files produced: test-files.txt, coverage-config.txt
- Acceptance: A count of test files and presence/absence of coverage configuration.

Task 7 — Identify large files, generated code, and vendored dependencies
- Goal: Surface maintenance hotspots: files >1MB, node_modules, vendor directories, build artifacts.
- Commands:
  - find . -type f -size +1M -exec ls -lh {} \; > ../analysis/.../large-files.txt
  - rg --hidden "node_modules|vendor|dist|build|target|\.pyc$|\.o$" -n || true
- Files produced: large-files.txt, copy of any top large files if needed (but avoid copying large binaries by default).
- Acceptance: List of files larger than 1MB and candidate generated/vendored directories.

Task 8 — Produce final report and JSON metadata
- Goal: Assemble findings into human-readable report and machine-readable metadata.
- Actions:
  - Create analysis/README.md with: executive summary, languages, top directories, build/test commands, CI summary, risks, recommended next steps.
  - Create analysis/report.md with command outputs and detailed findings.
  - Create analysis/metadata.json containing keys: repo, date, total_files, total_loc, languages:{name:loc,files}, tests:{count,paths}, ci:{present:true,workflows:[files]}, dockerfiles:[paths], top_large_files:[{path,size}], manifests:[files].
- Acceptance: README.md and metadata.json exist and metadata.json is well-formed JSON.

Task 9 — Review and QA
- Goal: Ensure produced artifacts are consistent and readable.
- Actions:
  - Run jq . analysis/metadata.json to validate JSON.
  - Ensure report.md references the exact paths of artifacts.

Notes and alternatives
- If cloc is not available, use github-linguist or a simple extension-based wc -l approach. If yq is not installed for parsing YAML CI files, include raw YAML files in the report instead.
- If repository is large, consider shallow clone and selective file listing to avoid fetching large LFS blobs.

Acceptance criteria (overall)
- The final README.md contains a compact executive summary (<= 250 words) that answers: what is the repo (languages, main components), how to build/test it, whether CI exists, and 5 prioritized recommendations.
- metadata.json and report.md exist and include the items enumerated in Tasks 1–8.
