Implementation Plan: Repository Structure Analysis for E2E Test Repo 0 (id: 8f897ac1-8fb1-44a3-a525-98ec7ec02959)

Overview
- Break the work into small, self-contained tasks that can be executed by sub-agents. Each task will read files only and produce JSON/YAML outputs that the aggregator agent will combine into the final report.

Task 1 — Top-level inventory
- Purpose: Record the top-level file/directory list and sizes.
- Commands/files to read:
  - run: ls -la
  - read: README.md (if present), CONTRIBUTING.md, LICENSE
- Output: top_level.json with fields: files[], dirs[], README_present, LICENSE_present
- Files/paths to inspect: ./
- Notes: Identify the repository default branch is main (provided).

Task 2 — File tree (depth-limited)
- Purpose: Produce a tree of files up to 4 levels deep.
- Commands/files to run:
  - find . -maxdepth 4 -type f -print
- Output: tree.txt (newline-separated paths) and tree.json (nested structure)
- Files/paths to inspect: all files; exclude .git (if present)

Task 3 — Language and manifest detection
- Purpose: Detect primary languages and locate manifest files & lockfiles.
- Files to check (if present): package.json, package-lock.json, yarn.lock, pnpm-lock.yaml, pyproject.toml, setup.py, requirements.txt, Pipfile, Pipfile.lock, go.mod, Cargo.toml, Cargo.lock, Gemfile, composer.json
- Output: manifests.json with detected files and parsed relevant fields (name, version, scripts, dependencies)
- Notes: Provide the detection logic and exact manifest paths.

Task 4 — Dependency summary
- Purpose: Extract direct dependencies and devDependencies from manifests.
- Commands:
  - For Node: parse package.json -> list dependencies/devDependencies
  - For Python: parse pyproject.toml or requirements.txt
  - For Go/Rust: parse go.mod / Cargo.toml
- Output: deps.json with per-language dependency lists and counts.

Task 5 — Tests and test commands
- Purpose: Identify test directories, test frameworks, and test run commands.
- Files to inspect: package.json.scripts, pytest.ini, tox.ini, setup.cfg, jest.config.*, tests/, spec/
- Output: tests.json listing test files/globs and suggested run commands

Task 6 — CI / Workflow analysis
- Purpose: Locate CI configs and summarize pipelines.
- Files to inspect: .github/workflows/*.yml, .gitlab-ci.yml, .travis.yml, .circleci/config.yml, azure-pipelines.yml
- Output: ci.json with found workflows and the main steps (build/test/lint/deploy)

Task 7 — Docker and containerization
- Purpose: Find Dockerfile(s) and docker-compose files; extract base images and exposed ports.
- Files: Dockerfile*, docker-compose*.yml
- Output: docker.json listing Dockerfiles, base images, and ports

Task 8 — Entry points and packaging
- Purpose: Identify executable entrypoints (main files, bin/*, cmd/*) and package layout.
- Files: src/, lib/, main.py, index.js, bin/, cmd/
- Output: entrypoints.json describing entrypoints and recommended run commands

Task 9 — Documentation and onboarding
- Purpose: Evaluate README quality and presence of docs/ directory.
- Files: README.md, docs/
- Output: docs_review.md with checklist: installation, usage, tests, contributing, license

Task 10 — Security & sensitive file scan
- Purpose: Search for potential secrets and high-risk files (private keys, passwords) and missing .gitignore
- Commands:
  - rg -n "(password|secret|PRIVATE_KEY|AWS_ACCESS_KEY_ID|AKIA)" --hidden || true
  - check presence of .env, .secrets
- Output: secrets_report.json (do not print contents of secrets; only note matches and file paths)

Task 11 — Aggregate report
- Purpose: Combine outputs from Tasks 1–10 into a human-readable summary and prioritized recommendations.
- Output: repo_analysis_report.md containing:
  - 1-paragraph summary
  - Languages, manifests, CI, tests
  - File tree (top 4 levels)
  - Dependency counts
  - Top 10 recommended actions (priority-coded)

Task 12 — Quick wins PR checklist
- Purpose: Produce a checklist for small PRs: add/clarify README sections, add LICENSE if missing, add CI tests, add linting, add .gitignore entries.
- Output: quick_wins.md with exact files to create/modify and suggested commit messages.

Execution details (formatting, outputs, and locations)
- All task outputs must be written under ./analysis-output/ with per-task subfolders, e.g. ./analysis-output/task-01/top_level.json
- Use JSON for machine-readable outputs and .md/.txt for human-readable outputs.
- Keep each task self-contained: do not modify repository files.

Estimated effort and ordering
- Run Tasks 1–4 and 6 in parallel to gather fast insights (~5–10 minutes).
- Then run Tasks 5, 7–10 (~5 minutes).
- Aggregate (Task 11) and produce quick-wins (Task 12) (~10 minutes).

Next steps
- After this plan is committed, dispatch the above tasks to implementation agents, one task per agent. Each task must return outputs in the specified paths and follow the no-modify rule.

Notes & Constraints
- Do not attempt to run tests or build artifacts that could be destructive.
- For private/secret matches, do not print values; only list file paths and line numbers.
- If a manifest is absent, note that explicitly in the corresponding output.
