Implementation plan: repository analysis for E2E Test Repo (id: 17d2f7cf-8afd-4c9e-a631-3c349f50207d)

Overview
- This plan defines a set of discrete tasks that, when executed in order, will produce the reports specified in the design spec. Each task is self-contained and includes exact shell commands, expected inputs and outputs, and file paths where reports should be written inside the repository working directory.

Assumptions
- The executor has a checked-out copy of the repository at the filesystem root (referred to as REPO_ROOT).
- Common CLI tools are available: bash, find, sed, jq, wc, du, sort, head. Optional tools recommended: cloc or scc, gitleaks, git (read-only), python3 for small parsing helpers.

Artifacts (paths relative to REPO_ROOT)
- reports/structure_summary.md
- reports/structure_details.json
- reports/top_files_by_size.txt
- reports/dependency_manifests.md
- reports/ci_and_workflows.md
- reports/tests_overview.md
- reports/gitleaks_report.json (optional)

Tasks

Task 1 — Prepare output directory
- Purpose: create reports/ directory and ensure clean state.
- Commands:
  mkdir -p reports

Task 2 — Top-level tree and counts
- Purpose: list top-level files and directories, file counts, and total size.
- Commands:
  echo "Top-level files and dirs:" > reports/structure_summary.md
  ls -la >> reports/structure_summary.md
  find . -maxdepth 2 -type d -print | sed 's|\./||' | sort > reports/top_level_dirs.txt
  find . -type f | wc -l > reports/total_files_count.txt
  du -sh . | cut -f1 > reports/repo_size.txt
- Outputs appended into reports/structure_summary.md and several small files.

Task 3 — Language detection and LOC
- Purpose: detect languages and size by LOC.
- Preferred tool: cloc. If cloc not available, use simple fallback using file extensions.
- Commands (preferred):
  cloc --json --quiet --out=reports/cloc.json .
  jq '.' reports/cloc.json > reports/structure_details.json # will include language breakdown
- Fallback (no cloc):
  find . -type f \( -name "*.py" -o -name "*.js" -o -name "*.ts" -o -name "*.go" -o -name "*.java" -o -name "*.rs" -o -name "*.php" \) > reports/code_files_list.txt
  # Count lines per extension
  awk -F. '{ext=$NF; print ext}' reports/code_files_list.txt | sort | uniq -c > reports/file_extension_counts.txt

Task 4 — Largest files (by size)
- Purpose: surface large blobs (binaries, datasets)
- Commands:
  find . -type f -printf "%s	%p\n" | sort -rn | head -n 100 | awk '{printf "%s\t%s\n", $1, $2}' > reports/top_files_by_size.txt

Task 5 — Identify package manifests and parse dependencies
- Purpose: find and capture contents of common dependency manifests.
- Files to look for (top-level and recursively): package.json, package-lock.json, yarn.lock, pnpm-lock.yaml, pyproject.toml, requirements.txt, setup.py, setup.cfg, Pipfile, go.mod, Cargo.toml, pom.xml, build.gradle
- Commands:
  for f in package.json package-lock.json yarn.lock pnpm-lock.yaml pyproject.toml requirements.txt setup.py setup.cfg Pipfile go.mod Cargo.toml pom.xml build.gradle; do
    find . -name "$f" -maxdepth 3 -print >> reports/manifest_paths.txt || true
  done
- For each manifest found, copy its contents into reports/dependency_manifests.md under a heading with the file path. Example (bash snippet):
  echo "### $path" >> reports/dependency_manifests.md
  sed 's/^/    /' "$path" >> reports/dependency_manifests.md
- Optionally, run parsers:
  # Node: extract dependencies
  jq '.dependencies, .devDependencies' path/to/package.json > reports/package_deps.json
  # Python: pip-tools or parse pyproject toml with python to extract [project] dependencies
  python3 - <<'PY'
import toml, sys
p='pyproject.toml'
obj=toml.load(p)
print(obj.get('project',{}) )
PY

Task 6 — CI / workflows / Docker
- Purpose: find CI config and Docker artifacts.
- Paths to scan: .github/workflows/**, .gitlab-ci.yml, Jenkinsfile, Dockerfile, docker-compose*.yml
- Commands:
  find . -path './.github/workflows/*' -type f -print > reports/ci_workflow_files.txt || true
  test -f .gitlab-ci.yml && echo '.gitlab-ci.yml' >> reports/ci_workflow_files.txt || true
  test -f Jenkinsfile && echo 'Jenkinsfile' >> reports/ci_workflow_files.txt || true
  find . -type f -name Dockerfile -print >> reports/dockerfiles.txt || true
- Summarize into reports/ci_and_workflows.md by including the file names and short excerpts (first 200 lines) for each.

Task 7 — Tests and coverage
- Purpose: locate tests directories and coverage config.
- Commands:
  find . -type d \( -name test -o -name tests -o -name spec -o -name __tests__ \) -print > reports/tests_dirs.txt || true
  grep -R --line-number "coverage" . | head -n 200 > reports/coverage_greps.txt || true
- Produce reports/tests_overview.md summarizing test frameworks detected and test file counts (e.g., pytest, jest).

Task 8 — Notable repository files
- Purpose: detect README, LICENSE, CONTRIBUTING, CODE_OF_CONDUCT, CODEOWNERS, .editorconfig
- Commands:
  for f in README.md LICENSE CONTRIBUTING.md CODE_OF_CONDUCT.md CODEOWNERS .editorconfig; do
    if [ -f "$f" ]; then
      echo "$f" >> reports/notable_files.txt
      sed -n '1,200p' "$f" > reports/snippet_$f || true
    fi
  done
- Summarize presence/absence and top snippets into reports/structure_summary.md.

Task 9 — Secrets scan (optional)
- Purpose: scan for high-confidence secrets using gitleaks or simple regex.
- Commands (gitleaks available):
  gitleaks detect --source . --report-format json --report-path reports/gitleaks_report.json || true
- Fallback lightweight scan (regex):
  grep -R --line-number -I -E "(AKIA|AIza|BEGIN RSA|PRIVATE KEY|password\s*=)" . | head -n 500 > reports/possible_secrets.txt || true

Task 10 — Produce structured JSON summary
- Purpose: consolidate outputs into reports/structure_details.json (if cloc produced JSON, augment it).
- Minimal structure (example):
  {
    "top_level": ["src","README.md",...],
    "languages": {"Python": 1200, "JavaScript": 300},
    "manifests": ["package.json", "pyproject.toml"],
    "ci": [".github/workflows/ci.yml"],
    "tests": {"dirs": ["tests"], "count": 42},
    "largest_files": [{"path":"path/to.bin","size": 12345678}],
    "possible_secrets": 5
  }
- Command example (jq composition):
  jq -n --arg top "$(ls -1 | jq -R -s -c 'split("\n")[:-1]')" '{top_level: $top}' > reports/structure_details.json
  # In practice build the JSON programmatically using Python or jq to merge earlier outputs.

Task 11 — Write human summary
- Purpose: synthesize findings into an executive summary (reports/structure_summary.md) with key bullets:
  - Primary languages and LOC
  - Presence/absence of CI
  - Tests and coverage status
  - Notable large files or binary artifacts
  - Missing maintenance files (license, contributing)
  - Any high-confidence secret findings
- Command: build by concatenating earlier report pieces into the md file.

Task 12 — (Optional) Create a small dependency graph
- Purpose: build a dependency graph for Node/Python/Go if manifests present.
- Example (Node): use npm ls --json (requires installing deps). Safer: parse package.json dependencies only and list names/versions into JSON.
- Commands (safe parse):
  jq '{name: .name, version: .version, dependencies: .dependencies}' package.json > reports/node_package_snapshot.json || true

Deliverables and Acceptance Criteria
- reports/structure_summary.md exists and contains a one-paragraph executive summary and a bullets section.
- reports/structure_details.json exists and is valid JSON with at least top_level, languages, manifests, ci, tests keys.
- reports/top_files_by_size.txt includes the 100 largest files.
- reports/dependency_manifests.md contains copies of any dependency manifest files found.
- If gitleaks is run, reports/gitleaks_report.json is produced; if not, reports/possible_secrets.txt should exist if fallback used.
