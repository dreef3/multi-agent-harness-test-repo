# Implementation Plan: Codebase Structure Analysis

Repository
- Name: E2E Test Repo
- URL: https://github.com/dreef3/multi-agent-harness-test-repo.git
- Repository ID: 4ba2fc47-8587-4fd9-bac1-c445c1666f66

Overview
- Create an automated, reproducible inspection that produces a Markdown report at analysis/codebase-structure-report.md in the target repository. The inspection will be split into small, self-contained tasks so sub-agents can run them independently.

Prerequisites (for any sub-agent running tasks)
- Linux/macOS environment (bash)
- git access to the repository (read-only)
- Standard CLI tools: ls, find, sed, awk, grep/rg
- Optional language tooling if deeper checks are enabled: node/npm or pnpm, python3 + pip, go, rust/cargo

Tasks (task-by-task)

Task 1 — Clone repository and create workspace
- Purpose: get a local copy and create a place to store analysis outputs
- Commands:
  - git clone https://github.com/dreef3/multi-agent-harness-test-repo.git repo && cd repo
  - mkdir -p analysis/inspection-output
- Expected output: repository checked out locally, analysis/inspection-output exists
- Time estimate: 1–2 minutes

Task 2 — Top-level file tree snapshot
- Purpose: provide a concise view of repository layout
- Commands:
  - find . -maxdepth 3 -mindepth 1 -type d -print -o -type f -name "*" -printf "%p\n" | sed 's/^\.\///g' > analysis/inspection-output/tree-level3.txt
  - head -n 200 analysis/inspection-output/tree-level3.txt
- Output: analysis/inspection-output/tree-level3.txt
- Notes: if repo is very large, increase depth or filter by sizes

Task 3 — Detect primary languages and manifests
- Purpose: detect language mix and manifests
- Commands (in order):
  - rg --hidden --glob '!node_modules' --files --type-add 'manifest:*.{json,toml,txt,mod,gemspec,csproj}' -g '!*node_modules/**' package.json pyproject.toml requirements.txt setup.cfg setup.py Pipfile Pipfile.lock go.mod Cargo.toml Gemfile Gemfile.lock build.gradle settings.gradle pom.xml package-lock.json yarn.lock pnpm-lock.yaml composer.json > analysis/inspection-output/manifests-found.txt || true
  - cat analysis/inspection-output/manifests-found.txt
  - optionally run: cloc --json . > analysis/inspection-output/cloc.json (if cloc is installed)
- Output: manifests-found.txt, optional cloc.json

Task 4 — Inspect README and contributing docs
- Purpose: identify project description and usage instructions
- Commands:
  - for f in README* CONTRIBUTING* docs/*; do echo "---- $f"; sed -n '1,200p' "$f"; done > analysis/inspection-output/readme-snippets.txt || true
- Output: readme-snippets.txt

Task 5 — CI and workflow discovery
- Purpose: find CI configs and summarize build matrix
- Commands:
  - rg -n "^" .github workflows .gitlab-ci.yml circleci .circleci Dockerfile -S --hidden || true
  - find .github -maxdepth 2 -type f -print > analysis/inspection-output/github-workflows.txt || true
  - find . -name "*Dockerfile*" -print > analysis/inspection-output/dockerfiles.txt || true
- Output: github-workflows.txt, dockerfiles.txt
- Notes: parse YAML files to extract jobs/matrices (next task)

Task 6 — Parse workflows for build/test steps (optional deeper parse)
- Purpose: extract job names, triggers, and matrix axes
- Commands (example using yq or python):
  - if command -v yq >/dev/null; then for f in $(cat analysis/inspection-output/github-workflows.txt); do echo "--- $f"; yq e '.name, .on, .jobs' "$f"; done > analysis/inspection-output/workflows-summary.txt; fi
- Output: workflows-summary.txt

Task 7 — Test discovery
- Purpose: find test directories and typical test commands
- Commands:
  - rg --hidden "pytest|unittest|jest|mocha|tape|ava|go test|cargo test|dotnet test|pytest.ini|tox.ini|jest.config|pytest" -S || true
  - find . -type d -name "test*" -o -name "tests" -print > analysis/inspection-output/test-dirs.txt || true
  - inspect package.json scripts: jq -r '.scripts // {}' package.json || true > analysis/inspection-output/npm-scripts.txt
- Output: test-dirs.txt, npm-scripts.txt

Task 8 — Dependency listing and quick audit
- Purpose: list declared dependencies and run quick audit commands
- Commands (examples by language):
  - Node/npm: if [ -f package.json ]; then jq -r '.dependencies,.devDependencies' package.json > analysis/inspection-output/npm-deps.txt || true; npm audit --json > analysis/inspection-output/npm-audit.json || true; fi
  - Python: if [ -f pyproject.toml ]; then pip-compile --output-file - pyproject.toml > /dev/null || true; fi; if [ -f requirements.txt ]; then cp requirements.txt analysis/inspection-output/python-requirements.txt; fi
  - Go: if [ -f go.mod ]; then go list -m all > analysis/inspection-output/go-mod-list.txt || true; fi
- Notes: Explicit install steps are optional and gated by a flag; audits that require network may be limited by environment

Task 9 — Static checks and linters (optional)
- Purpose: run linters or static analyzers to identify code health markers
- Commands (examples):
  - Node: npm run lint --if-present
  - Python: pytest -q --maxfail=1 --disable-warnings (but running tests is separate)
  - Run flake8, mypy if configured
- Output: analysis/inspection-output/lint-results.txt
- Notes: Mark these optional in the task so they aren't run unless requested

Task 10 — Run unit tests (explicit opt-in)
- Purpose: verify test suite runs and capture coverage
- Commands (examples):
  - Node: npm test --if-present
  - Python: pytest -q --maxfail=1 --disable-warnings --junitxml=analysis/inspection-output/junit.xml
  - Go: go test ./... -json > analysis/inspection-output/go-test.json
- Output: junit.xml or equivalent test result files
- Safety: tests may be slow or require services; require opt-in flag and environment configuration

Task 11 — Synthesize findings into final report
- Purpose: produce analysis/codebase-structure-report.md
- Inputs: all files under analysis/inspection-output
- Key sections in report (template):
  - Executive summary (languages, top-level structure, primary build/test flows)
  - File tree snapshot (top 3 levels)
  - Manifest and dependency summary
  - CI/workflow summary and notable build matrix items
  - Tests: discovered frameworks, commands, and failures (if run)
  - Docker and containerization notes
  - Recommended next actions (low/med/high effort)
  - Appendix: raw outputs (paths to files under analysis/inspection-output)
- Commands to generate the report: use a small helper script or assemble with cat/awk to create analysis/codebase-structure-report.md

Task 12 — Commit the report and open a PR (optional, follow repo policies)
- Purpose: make the report available in repository
- Commands (if allowed):
  - git checkout -b analysis/codebase-structure
  - git add analysis/codebase-structure-report.md analysis/inspection-output/*
  - git commit -m "Add codebase structure analysis report"
  - git push origin HEAD
- Note: This step must respect repository contribution policies; if not permitted, return the report to requester instead of pushing.

Deliverables
- analysis/codebase-structure-report.md (Markdown report)
- analysis/inspection-output/ (directory with raw outputs used to generate the report)
- Optional: a PR URL if the report was committed to a branch

Acceptance criteria
- The report contains the required sections from the spec and points to the inspection outputs.
- Manifest detection lists at least the top-level manifest files.
- CI/workflow section lists files under .github/workflows and their job names or a note if parsing failed.

Notes and recommendations
- Keep heavy operations optional (tests, dependency installs) behind an explicit opt-in flag.
- If the repo is large/monorepo, consider per-package scanning and an index page summarizing each package.
- Consider running GitHub language statistics (linguist) or cloc for more precise language breakdowns.

Timeline
- Quick scan (Tasks 1–6 + 11): ~15–30 minutes
- Full audit with tests & linters (Tasks 7–10 + 11): 1–3 hours depending on test suite size and environment
