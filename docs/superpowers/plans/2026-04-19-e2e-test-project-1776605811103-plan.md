Implementation Plan: Task-by-task runbook to analyze the codebase structure

Goal
- Run the plan against the repository to produce a detailed analysis report describing the repository tree, languages/frameworks, dependencies, tests, CI, and recommendations.

Overview
- The plan is a sequence of discrete tasks. Each task is self-contained and includes exact file paths to read or shell commands to run. A single agent can run tasks in order; tasks 1–3 are discovery steps and must run first.

Assumptions
- You have a clone of the repository on your machine or agent working directory.
- Common CLI tools are available: sh/bash, rg (ripgrep) or grep, jq, sed, awk, tree (optional), python or node (optional) for parsing manifest files.

Tasks

Task 1 — Top-level inventory
- Purpose: List top-level files and directories to get an initial map of the repo.
- Commands:
  - ls -la
  - tree -L 2 || (find . -maxdepth 2 -print)
- Files to note at root: README.md, LICENSE, CHANGELOG.md, CONTRIBUTING.md, CODE_OF_CONDUCT.md, package.json, pyproject.toml, setup.py, requirements.txt, Pipfile, Gemfile, go.mod, Cargo.toml, Dockerfile, Makefile, .github/
- Output files to produce: top-level-tree.txt (a plain text listing)

Task 2 — Identify primary languages and manifests
- Purpose: Find language-specific manifests to determine primary languages and frameworks.
- Commands and file reads (in order):
  - [Node.js] if package.json exists: cat package.json | jq '.' > package.json.parsed.json
  - [Python] if pyproject.toml exists: cat pyproject.toml
  - [Python] if requirements.txt exists: cat requirements.txt
  - [Go] if go.mod exists: cat go.mod
  - [Rust] if Cargo.toml exists: cat Cargo.toml
  - [Ruby] if Gemfile exists: cat Gemfile
  - [Docker] if Dockerfile exists: cat Dockerfile
- Expected deliverable: manifest-summary.md listing detected manifests and the primary language(s).

Task 3 — Source layout
- Purpose: Discover source code directories and their entry points.
- Commands:
  - rg --hidden --glob '!node_modules' --files --max-depth 3 --type f "package.json|pyproject.toml|setup.py|__init__.py|main.py|index.js|app.py|server.js|src/"  || find . -maxdepth 3 -type d -print
  - Look for directories: src/, lib/, packages/, apps/, services/, cmd/, internal/
- Files to read (examples): src/**/main.* , src/**/index.* , src/**/app.*
- Deliverable: source-layout.md mapping top-level components and entrypoints.

Task 4 — Tests and E2E test detection
- Purpose: Locate test suites, test frameworks, and E2E test harnesses.
- Commands:
  - rg --hidden --glob '!node_modules' "describe\(|test\(|it\(|pytest\.|from pytest|unittest" || find tests -type f
  - Look for directories: tests/, e2e/, integration-tests/, spec/
  - Read pytest.ini, tox.ini, jest.config.js, cypress.json, playwright.config.*
- Deliverable: tests-summary.md with test frameworks and how to run (commands like pytest, npm test, yarn test, playwright test, cypress run).

Task 5 — CI and GitHub workflows
- Purpose: Summarize CI pipelines and steps relevant to E2E testing and releases.
- Files to inspect:
  - .github/workflows/*.yml
  - .gitlab-ci.yml
  - circleci/config.yml
- Commands: cat .github/workflows/*.yml | sed -n '1,200p'
- Extract: matrix configuration, OSes, versions, caching, steps for install/test/build/publish.
- Deliverable: ci-summary.md with a step-by-step of each workflow and quick notes on long-running steps (e.g., docker build, E2E jobs).

Task 6 — Docker, containers, and deployment
- Purpose: Determine whether the project is containerized and how to build/run images.
- Files: Dockerfile, docker-compose.yml, k8s manifests (deploy/, manifests/)
- Commands: cat Dockerfile; cat docker-compose.yml
- Deliverable: docker-summary.md with build/run instructions and multi-stage build notes.

Task 7 — Dependency and security surface
- Purpose: Extract dependency lists and flag anything outdated or risky.
- Commands:
  - [Node] jq -r '.dependencies, .devDependencies' package.json
  - [Python] pip-compile or parsing requirements.txt
  - [Go] go list -m all
  - [Rust] cargo tree --depth 1
- Note: If cannot run language-specific tools, just parse manifest files and list top-level deps and versions.
- Deliverable: deps-summary.md listing direct dependencies, dev deps, and notable large/old transitive deps.

Task 8 — Linting, formatting, and developer tooling
- Purpose: Find configs for linters, formatters, and pre-commit hooks that affect contributions.
- Files: .eslintrc*, .prettierrc*, .flake8, .pre-commit-config.yaml, .husky/
- Commands: rg --hidden --files --glob '.pre*' --glob '.eslintrc*' . || ls -la
- Deliverable: tooling-summary.md

Task 9 — Tests: run and verify (optional, gated)
- Purpose: To validate the findings, run the tests in a reproducible environment. This may require installing language runtimes.
- Commands (examples):
  - npm ci && npm test
  - pip install -r requirements.txt && pytest -q
  - docker build -t repo-name:local . && docker run --rm repo-name:local
- Produce: test-run-output.txt and a test-run-summary.md
- Note: This task is optional and should be run in CI or sandbox to avoid side effects.

Task 10 — Produce final analysis report
- Combine all earlier deliverables into analysis-report.md. Sections:
  - Top-level tree (top-level-tree.txt)
  - Primary languages & manifests (manifest-summary.md)
  - Source layout (source-layout.md)
  - Tests (tests-summary.md)
  - CI (ci-summary.md)
  - Docker (docker-summary.md)
  - Dependencies (deps-summary.md)
  - Linting/tooling (tooling-summary.md)
  - Risk summary & recommendations (1–2 pages)
- Example recommendation bullets:
  - Add a single top-level README with quickstart and test commands (if missing).
  - If no CI for E2E, propose adding a dedicated E2E workflow matrix.
  - If dependencies are old, recommend a routine upgrade flow pinned in CI.
