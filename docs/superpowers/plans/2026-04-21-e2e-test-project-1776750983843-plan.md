Implementation Plan: Repository Structure Analysis (task-by-task)

Overview

This plan gives step-by-step tasks an implementing agent (or developer) should run in the checked-out repository workspace to produce the analysis outputs. Each task is self-contained and lists exact commands, target file paths, expected outputs, and time estimates.

Preparation

- Working directory: assume the repository is checked out at the workspace root (./). If the harness checks out repositories under a different path, adjust commands accordingly.
- Output artifacts (written by the implementing agent): ANALYSIS.md, analysis.json, commands.txt

Task 1 — Top-level listing

- Purpose: capture full top-level directory and files to identify obvious layout.
- Commands to run (in repo root):
  - ls -la > commands.txt
  - tree -a -L 2 | sed 's/$/\n/' >> commands.txt  # (if tree exists; otherwise use find)
  - find . -maxdepth 2 -type d -print | sed 's/^\.\///' >> commands.txt
- Files to inspect: README*, LICENSE*, CONTRIBUTING*, CODE_OF_CONDUCT*
- Time estimate: 1 minute
- Expected outcome: commands.txt contains top-level listing used in ANALYSIS.md

Task 2 — Detect common manifest files and package managers

- Purpose: find language indicators and dependency managers.
- Files to check (exact file paths to try to read if present):
  - package.json
  - package-lock.json, yarn.lock, pnpm-lock.yaml
  - pnpm-workspace.yaml, lerna.json, turbo.json, nx.json
  - pyproject.toml, setup.py, requirements.txt, Pipfile, Pipfile.lock
  - go.mod
  - Cargo.toml
  - Gemfile, Gemfile.lock
  - composer.json
  - setup.cfg
  - build.gradle, settings.gradle, gradle.properties
  - Makefile
- Commands (run as conditional reads):
  - for f in package.json pyproject.toml go.mod Cargo.toml Gemfile composer.json requirements.txt Pipfile package-lock.json yarn.lock pnpm-lock.yaml pnpm-workspace.yaml lerna.json turbo.json nx.json; do [ -f "$f" ] && echo "== $f ==" >> commands.txt && sed -n '1,200p' "$f" >> commands.txt; done
- Time estimate: 2-4 minutes
- Expected outcome: list of present manifests; contents help identify primary languages and frameworks

Task 3 — Language breakdown and LOC estimation

- Purpose: produce a rough language breakdown and lines-of-code (LOC) summary.
- Preferred tool: cloc (if available). Fallback: simple file extension counts.
- Commands:
  - if command -v cloc >/dev/null 2>&1; then cloc --by-file --json . > analysis_cloc.json; else (find . -type f -name "*.js" | wc -l > js_count.txt; find . -type f -name "*.py" | wc -l > py_count.txt; fi)
  - Use cloc output (analysis_cloc.json) to populate analysis.json.languages
- Time estimate: 1-5 minutes
- Expected outcome: analysis.json contains language: {"JavaScript": n, "TypeScript": m, "Python": k, ...}

Task 4 — Source directories and patterns

- Purpose: find the main source directories and common patterns (src/, lib/, packages/, services/, apps/, cmd/, internal/)
- Commands:
  - find . -maxdepth 3 -type d \( -name src -o -name lib -o -name packages -o -name services -o -name apps -o -name cmd -o -name internal -o -name pkg \) -print > commands.txt
  - Also check for monorepo package folders: cat pnpm-workspace.yaml or lerna.json or package.json.workspaces
- Output paths to capture in analysis.json.src_dirs and ANALYSIS.md
- Time estimate: 2 minutes

Task 5 — Test directories and frameworks

- Purpose: locate tests and identify CI/test frameworks (jest, pytest, go test, cargo test, mocha, rspec)
- Commands:
  - find . -type d -name test -o -name tests -o -name __tests__ -o -name spec -o -name e2e -o -name cypress -o -name integration -print > commands.txt
  - grep -R --line-number "jest\|mocha\|pytest\|go test\|cargo test\|rspec\|playwright\|cypress" -n package.json 2>/dev/null || true
- Expected outcome: list of test directories and test frameworks in ANALYSIS.md
- Time estimate: 3 minutes

Task 6 — CI and automation

- Purpose: find CI workflows, GitHub Actions, GitLab CI, Circle, etc.
- Files to check:
  - .github/workflows/*
  - .gitlab-ci.yml
  - .circleci/config.yml
  - azure-pipelines.yml
- Commands:
  - if [ -d .github/workflows ]; then ls -la .github/workflows >> commands.txt && for f in .github/workflows/*; do echo "== $f ==" >> commands.txt && sed -n '1,200p' "$f" >> commands.txt; done; fi
  - [ -f .gitlab-ci.yml ] && sed -n '1,200p' .gitlab-ci.yml >> commands.txt
- Time estimate: 2-3 minutes
- Expected outcome: list of CI files and brief summary of steps and triggers

Task 7 — Docker, container and infra files

- Purpose: locate Dockerfiles and infra scripts
- Files to check: Dockerfile, docker/*.Dockerfile, k8s/, helm/, charts/
- Commands:
  - find . -type f -iname "dockerfile*" -o -iname "*Dockerfile*" -o -path "./docker/*" -print > commands.txt
  - find . -maxdepth 3 -type d -name k8s -o -name helm -o -name charts -print >> commands.txt
- Time estimate: 2 minutes

Task 8 — Linting, formatting, and tooling

- Purpose: identify linters and formatters configuration files
- Files to check: .eslintrc*, .prettierrc*, .editorconfig, .flake8, tox.ini, stylua.toml
- Commands:
  - for f in .eslintrc* .prettierrc* .editorconfig .flake8 tox.ini; do [ -f "$f" ] && echo "== $f ==" >> commands.txt && sed -n '1,200p' "$f" >> commands.txt; done
- Time estimate: 1 minute

Task 9 — Security and sensitive files

- Purpose: detect possible secrets or sample env files to note but not print secrets
- Files to check: .env, .env.example, .env.sample, .npmrc
- Commands:
  - for f in .env .env.example .env.sample .npmrc; do [ -f "$f" ] && echo "FOUND:$f" >> commands.txt; done
- Note: do not include file contents in the output report
- Time estimate: 1 minute

Task 10 — Compose the final reports

- Purpose: build ANALYSIS.md and analysis.json from collected pieces.
- ANALYSIS.md structure (example):
  - Title and brief repo summary
  - Table of contents
  - Language breakdown (from cloc or extension heuristics)
  - Top-level files and directories
  - Source layout (src directories and notable packages)
  - Tests and CI summary
  - Docker/infra summary
  - Linters/formatters
  - Notable files (README, CONTRIBUTING, license)
  - Recommendations and next steps
- analysis.json fields example:
  {
    "repo": "multi-agent-harness-test-repo",
    "languages": { "JavaScript": 12345, "Python": 234 },
    "top_level_dirs": ["src","packages",".github","docs"],
    "manifests": ["package.json","pyproject.toml"],
    "ci_files": [".github/workflows/ci.yml"],
    "dockerfiles": ["Dockerfile"],
    "test_dirs": ["tests","e2e"],
    "src_dirs": ["src","packages/service-a"],
    "notes": "No secrets found; monorepo using pnpm"
  }
- Commands to write outputs (example):
  - echo "# Analysis for $(basename $(pwd))" > ANALYSIS.md
  - jq . analysis_cloc.json > analysis.json  # (compose as needed)
  - cp commands.txt ANALYSIS/commands.txt
- Time estimate: 5-10 minutes

Task 11 — Deliverables and QA

- Purpose: ensure outputs are present and concise.
- Verify files created at repo root: ANALYSIS.md, analysis.json, commands.txt
- Optionally open a PR or create an issue with the report; that step is outside this plan.
- Time estimate: 2 minutes

Total estimated time: 15-30 minutes (normal repo)

Notes about execution environment and permissions

- The implementing agent should run commands from repo root and must have read access to all files. The plan avoids writing or printing secrets.
- If cloc or tree are not available, the agent should fallback to find/wc and simple heuristics.

Example snippets to include in ANALYSIS.md (template)

- A short example header and language summary:

  # Repository Structure Analysis: multi-agent-harness-test-repo

  Summary: This repository appears to be a (language) project with the following top-level directories: ...

  Language breakdown (rough):

  - JavaScript/TypeScript: 12,345 LOC
  - Python: 234 LOC
  - Shell: 56 LOC

  Top-level files:

  - README.md, package.json, .github/workflows/ci.yml, Dockerfile

  Source layout:

  - packages/ - monorepo packages (n packages)  
  - services/ - microservices (m services)

  Tests:

  - tests/  (pytest)  
  - packages/*/__tests__ (jest)

  CI:

  - GitHub Actions workflows detected at .github/workflows/*.yml

  Recommendations:

  - Run npm ci && npm test to exercise test suite.  
  - Add code coverage in CI if missing.  
  - Consider standardizing on a single lockfile (pnpm/yarn/npm).

Closing

- After generating the artifacts, the implementing agent should attach them to the job output or commit them to a branch and open a PR as per the harness workflow.
