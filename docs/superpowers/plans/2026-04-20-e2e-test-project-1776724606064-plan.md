Implementation Plan: Task-by-task instructions to analyze codebase structure

Repository
- Name: E2E Test Repo 1
- ID: 8a57706d-7e6a-464a-bd9e-1b182af603da
- URL: https://github.com/dreef3/multi-agent-harness-test-repo.git

Objective
- Produce a reproducible repo structure analysis and summary artifacts placed in analysis/ at the repository root.

Output files (to be created under repository root)
- analysis/report.md           (human-readable summary)
- analysis/file_tree.txt       (text listing, tree -a -L 3)
- analysis/cloc.json           (language breakdown from cloc)
- analysis/dependencies.json   (collected dependency manifest contents and summaries)
- analysis/ci_files.txt        (list and small excerpts of CI/workflow files)
- analysis/tests_summary.md    (where tests are, how to run them, quick results if executed)
- analysis/scan_commands.txt   (commands used to produce artifacts)

High-level tasks

Task 1 — Prepare workspace (manual or agent)
- Clone repo: git clone https://github.com/dreef3/multi-agent-harness-test-repo.git
- Checkout main branch (or default): git checkout main
- Create analysis directory: mkdir -p analysis
- Record exact git commit hash: git rev-parse --verify HEAD > analysis/commit.txt

Task 2 — Top-level inventory and file tree
- Commands:
  - tree -a -L 3 > analysis/file_tree.txt  # if tree not available, use: find . -maxdepth 2 -print | sed 's|^\./||' > analysis/file_tree.txt
  - head -n 200 file_tree.txt (for quick review)
- Purpose: capture top-level directories and typical project layout.

Task 3 — Language breakdown
- Commands:
  - cloc --json . > analysis/cloc.json
  - If cloc not installed: use `npx cloc . --json` for JS environment, or `gocloc`/`scc` alternatives; save equivalent JSON to analysis/cloc.json
- Purpose: identify dominant languages and approximate code size.

Task 4 — Find important manifest files and package managers
- Search for common dependency files and copy contents to analysis/dependencies.json (structured):
  - package.json (npm/yarn)
  - yarn.lock / package-lock.json
  - requirements.txt, Pipfile, pyproject.toml, setup.py (Python)
  - go.mod (Go), Cargo.toml (Rust), Gemfile (Ruby)
  - pom.xml (Maven), build.gradle (Gradle)
  - composer.json (PHP)
- Commands (example):
  - for f in package.json pyproject.toml requirements.txt go.mod Cargo.toml pom.xml; do if [ -f "$f" ]; then echo "--- $f" >> analysis/dependencies.json; sed -n '1,400p' "$f" >> analysis/dependencies.json; fi; done
- Purpose: build a single file containing relevant manifests for quick review.

Task 5 — CI / workflows / automation
- Locate CI configs and workflows:
  - .github/workflows/*.yml
  - .gitlab-ci.yml, .circleci/config.yml, .travis.yml, azure-pipelines.yml
- Commands:
  - rg --hidden "^name: |runs-on: |jobs:|workflow:|on:" .github -S || true; ls -la .github/workflows 2>/dev/null | sed -n '1,200p' > analysis/ci_files.txt
  - For each workflow file found, append a header and the first ~200 lines to analysis/ci_files.txt
- Purpose: document CI, triggers, and important jobs (build/test/deploy).

Task 6 — Docker, containers, and infra
- Look for Dockerfile(s), docker-compose.yml, helm charts, k8s manifests, Terraform, and Makefiles.
- Commands:
  - rg --hidden "^FROM |docker-compose|helm|terraform|provider" -S --glob "**/Dockerfile" || true
  - cp any Dockerfile excerpts into analysis/ci_files.txt or a new analysis/docker_files.txt

Task 7 — Tests discovery and quick-run instructions
- Identify test directories and common test frameworks:
  - pytest, unittest, jest, mocha, go test, cargo test
- Commands:
  - rg --hidden "pytest\(|describe\(|it\(|go test|cargo test|mocha|jest" -S || true
  - List tests: find . -name "test*" -o -name "*_test.*" | sed 's|^\./||' > analysis/tests_list.txt
- If allowed and quick (< 10 minutes) and environment supports it, run tests and capture results:
  - npm ci && npm test  (or npm install)
  - pip install -r requirements.txt && pytest -q
  - go test ./... -json > analysis/go_test.json
- Record commands used in analysis/scan_commands.txt
- Place quick run results in analysis/tests_summary.md summarizing pass/fail and commands to reproduce.

Task 8 — Static metadata extraction
- README.md: summarize first 200 lines into analysis/report.md
- LICENSE: detect and name license
- CONTRIBUTORS, CODEOWNERS
- Commands:
  - head -n 200 README.md > analysis/readme_excerpt.md
  - sed -n '1,100p' LICENSE > analysis/license.txt

Task 9 — Produce human-readable report (analysis/report.md)
- Sections to include:
  - Repo metadata (URL, commit hash, top-level dirs)
  - Language summary (from cloc)
  - Package managers & key manifests
  - CI/workflows and Docker info
  - Tests (where they live and how to run)
  - Quick recommendations (missing CI? large vendor dirs? monorepo notes?)
  - Action items and next steps (optional vulnerability scan, test execution, coverage)
- Use snippets from collected files to illustrate (short excerpts only).

Task 10 — Quality checks and PR
- Verify all analysis/* files are present and non-empty.
- Create a branch: git checkout -b analysis/report/auto
- Add analysis/ files and commit: git add analysis && git commit -m "Add repository structure analysis"
- Push branch and open PR (note: the harness will open PR automatically per process if configured)

Optional Task 11 — Dependency vulnerability scans (explicit opt-in)
- Tools: npm audit, pip-audit, safety, snyk, trivy (for container images)
- Produce a separate artifact analysis/vulnerabilities.md with results and recommended fixes.

- rg --hidden "^name: |runs-on: |jobs:|workflow:|on:" .github -S || true; ls -la .github/workflows 2>/dev/null | sed -n '1,200p' > analysis/ci_files.txt
- find . -name "test*" -o -name "*_test.*" | sed 's|^\./||' > analysis/tests_list.txt

- (optional) run tests and capture results into analysis/tests_summary.md

Deliverables
- All analysis artifacts under analysis/ in the repository root.
- A clear human-readable summary at analysis/report.md.
- A short checklist of next actions and optional deeper scans in the report.

Who should run this
- A repository-analyzer automation agent (preferred) or a developer with git access.
- If run by a human, provide guidance to avoid modifying live code and to run commands in a fresh checkout.

Notes and assumptions
- The plan assumes a POSIX-like environment (Linux/macOS) with standard CLI tools. If running on Windows, adapt commands accordingly (Git Bash, WSL).
- For very large repositories (>100k files), some commands (cloc, tree) may be slow; adjust limits or use sampling.
