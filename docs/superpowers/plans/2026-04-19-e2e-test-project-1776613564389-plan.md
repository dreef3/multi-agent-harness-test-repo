Implementation plan: Task-by-task instructions to analyze repository structure

Repository
- URL: https://github.com/dreef3/multi-agent-harness-test-repo.git
- Repo id (for the harness): e720f506-f75a-4409-9393-2e6b7a72a300
- Working directory for commands: /workspace/multi-agent-harness-test-repo (adjust if different)

General notes
- Create a reports/ directory at repository root to store outputs. Use a single run script or a sequence of commands and save outputs explicitly.
- All commands are written for a POSIX shell. Where commands may modify the environment, run them inside an ephemeral container or virtualenv.

Task 1 — Clone and prepare (manual/automation)
- Purpose: obtain the repo and set a clean workspace.
- Commands:
  - git clone https://github.com/dreef3/multi-agent-harness-test-repo.git /workspace/multi-agent-harness-test-repo
  - cd /workspace/multi-agent-harness-test-repo
  - mkdir -p reports
- Outputs: repository available locally, reports/ directory created
- Estimated time: 30s

Task 2 — Full file inventory
- Purpose: get an authoritative list of files and top-level layout
- Commands:
  - find . -path './.git' -prune -o -type f -print | sed 's|^./||' > reports/file_inventory.txt
  - tree -a -I '.git' -L 3 > reports/tree_level3.txt  # optional if tree is installed
- Files to check explicitly (look for existence): README.md, CONTRIBUTING.md, LICENSE, CODE_OF_CONDUCT.md
- Estimated time: 1m

Task 3 — Detect languages and code size
- Purpose: quantify languages and LOC
- Commands:
  - cloc --json --out=reports/cloc.json . || (docker run --rm -v "$(pwd)":/src cytopia/cloc cloc --json --out=/src/reports/cloc.json /src)
  - jq . reports/cloc.json > reports/cloc_pretty.json  # pretty-printed
- Output: reports/cloc.json
- Estimated time: 1–2m

Task 4 — Detect package manifests and package managers
- Purpose: locate package manifest files to decide dependency analysis steps
- Commands (run from repo root):
  - rg "^" -g '!node_modules' -S "package.json|package-lock.json|yarn.lock|pnpm-lock.yaml|pyproject.toml|requirements.txt|setup.py|Pipfile|poetry.lock|go.mod|Cargo.toml|Gemfile|Gemfile.lock|composer.json|composer.lock|Dockerfile|Makefile" --hidden --files | sed 's|^|./|' | sort > reports/package_manifests.txt
  - cat reports/package_manifests.txt
- Output: reports/package_manifests.txt
- Estimated time: 1m

Task 5 — Per-ecosystem dependency listing and audit
- Purpose: enumerate and run built-in audit where available
- For Node.js (if package.json present in any directory):
  - For each directory containing package.json:
    - cd <dir>
    - npm ci --no-audit --no-fund --prefer-offline  # use CI install to avoid changing package-lock
    - npm ls --all --json > ../../reports/npm_deps_<sanitized-dir>.json
    - npm audit --json > ../../reports/npm_audit_<sanitized-dir>.json || true
  - Notes: if yarn or pnpm is used, swap commands accordingly (yarn install --frozen-lockfile; pnpm install --frozen-lockfile)
- For Python (requirements.txt, pyproject.toml, Pipfile):
  - python3 -m venv .venv_analysis && source .venv_analysis/bin/activate
  - pip install -U pip
  - if requirements.txt: pip install -r requirements.txt --no-deps -t /tmp/pip-install || true
  - pip-audit -f json > reports/pip_audit.json || pip-audit --output reports/pip_audit.json || true
  - Deactivate venv after
- For Go (go.mod present):
  - GO111MODULE=on go list -m all > reports/go_list_modules.txt
  - go mod graph > reports/go_mod_graph.txt
- For Rust (Cargo.toml present):
  - cargo metadata --format-version 1 --no-deps > reports/cargo_metadata.json
  - cargo tree --all-features --prefix none > reports/cargo_tree.txt
- For Ruby (Gemfile):
  - bundle install --path vendor/bundle --quiet || true
  - bundle audit --update ; bundle audit check --verbose > reports/bundle_audit.txt || true
- For PHP (composer.json):
  - composer install --no-interaction --no-dev --prefer-dist || true
  - composer show --format=json > reports/composer_show.json
- Notes: many dependency commands may require internet; document failures in reports/dependency_errors.txt
- Estimated time: 5–20m depending on size and network

Task 6 — CI / workflow inspection
- Purpose: enumerate CI files and surface key workflows
- Files to inspect:
  - .github/workflows/*.yml
  - .gitlab-ci.yml
  - Jenkinsfile
  - .circleci/config.yml
  - azure-pipelines.yml
- Commands:
  - mkdir -p reports/ci
  - for f in .github/workflows/*.yml .gitlab-ci.yml Jenkinsfile .circleci/config.yml azure-pipelines.yml; do [ -f "$f" ] && echo "--- $f ---" >> reports/ci/ci_workflows.md && sed -n '1,400p' "$f" >> reports/ci/ci_workflows.md; done
  - rg "runs-on:|uses:|steps:|script:|docker build" .github -n --hidden || true
- Output: reports/ci/ci_workflows.md
- Estimated time: 2–5m

Task 7 — Test discovery and run summary
- Purpose: find tests, run test commands where safe, and summarize test status
- Files/dirs to inspect: tests/, test/, spec/, src/**/test, package.json scripts.test
- Commands (examples):
  - rg "\b(pytest|unittest|nose|mocha|jest|ava|go test|cargo test)\b" -S || true
  - For Node.js: if package.json has test script, run npm test --silent and capture output to reports/npm_test_<dir>.txt (note: ensure a clean environment)
  - For Python: pytest -q --maxfail=1 > reports/pytest_summary.txt || true
  - For Go: go test ./... -json > reports/go_test.json || true
  - For Rust: cargo test --workspace --message-format=json > reports/cargo_test.json || true
- If running tests is risky (e.g., integration tests that require credentials), only perform test discovery and do not run them; list commands in reports/test_run_instructions.txt
- Output: reports/test_summary.md and any raw outputs
- Estimated time: discovery 1–2m; running tests variable

Task 8 — Docker and container artifacts
- Purpose: find Dockerfile(s) and note build contexts and targets
- Commands:
  - rg "^FROM\b" -n --hidden --glob '!**/node_modules/**' || true
  - for df in $(rg --files -g '!**/node_modules/**' -S -g 'Dockerfile' || true); do echo "--- $df" >> reports/docker_files.txt; sed -n '1,200p' "$df" >> reports/docker_files.txt; done
  - Optionally: docker build -f <Dockerfile> -t repo-analysis-temp:latest .  # only in CI-safe environment
- Output: reports/docker_files.txt, reports/docker_summary.md
- Estimated time: 2–10m

Task 9 — Static analysis / linters (optional)
- Purpose: run available linters to surface style and obvious issues
- Commands (examples):
  - JavaScript: npx eslint . --format json -o reports/eslint.json || true
  - TypeScript: tsc --noEmit || true
  - Python: ruff --format json . > reports/ruff.json || flake8 --format=default > reports/flake8.txt || true
  - Go: go vet ./...; staticcheck ./... > reports/staticcheck.txt || true
- Output: reports/*
- Estimated time: 5–20m

Task 10 — Vulnerability summary and triage
- Purpose: consolidate audit outputs (npm audit, pip-audit, snyk if available) into a single human-readable file
- Commands:
  - collate audits: jq -s '.' reports/npm_audit_*.json > reports/npm_audits_all.json || true
  - create reports/vulnerability_report.md summarizing counts and high/critical items for each ecosystem
- Output: reports/vulnerability_report.md
- Estimated time: 5–10m

Task 11 — Final structure report and recommendations
- Purpose: produce reports/structure_report.md summarizing findings, risk areas, quick wins, and recommended next steps
- Content outline for reports/structure_report.md:
  - Executive summary (one paragraph)
  - Repo snapshot (size, languages top 3, top-level directories)
  - Package managers and dependency counts
  - Tests: coverage of tests found and ability to run them
  - CI: workflows and trust boundaries
  - Docker: images and build complexity
  - Vulnerabilities: counts and critical items
  - Recommendations and prioritized remediation steps
- Commands:
  - Use simple concatenation and templating: generate markdown by combining small templates and data extracted from reports/*
  - Example: echo "# Structure report" > reports/structure_report.md; jq . reports/cloc.json >> reports/structure_report.md
- Output: reports/structure_report.md
- Estimated time: 10–30m depending on repo size

Task 12 — Commit artifacts (optional, if desired)
- Purpose: persist reports in a branch for review
- Commands:
  - git checkout -b analysis/repo-structure
  - git add reports && git commit -m "Add repo structure analysis reports" || true
  - git push --set-upstream origin analysis/repo-structure || true
- Note: committing reports is optional and may be policy-dependent; if running in ephemeral CI, prefer uploading artifacts instead of committing.

Risks and mitigation
