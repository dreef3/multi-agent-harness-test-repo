Implementation Plan: Task-by-Task Codebase Structure Analysis

Overview
This plan breaks the static analysis into discrete, self-contained tasks that can be executed by implementation agents. Each task lists exact commands to run, expected output files to create, and exact paths to write results. Agents must not modify source files except to add the two report files at repo root: CODEBASE_ANALYSIS.md and analysis_summary.json. The analysis is read-only otherwise.

Repository
- URL: https://github.com/dreef3/multi-agent-harness-test-repo.git
- Repository ID: 4cb446ae-9b19-4348-999f-730272ee9ae3
- Default branch: main

Preconditions
- Agent has network access and git installed. The agent will clone the repository into a working directory.
- Agent will not build or run project code.

Tasks

Task 1 — Clone repository (single-step, safe)
- Purpose: obtain a local copy for static inspection
- Commands:
  - git clone https://github.com/dreef3/multi-agent-harness-test-repo.git repo
  - cd repo
  - git checkout main
- Outputs: local copy at ./repo

Task 2 — Top-level inventory
- Purpose: record top-level files and directories
- Commands:
  - cd repo
  - printf "\n--- TOP-LEVEL FILES ---\n" > analysis_raw.txt
  - ls -la >> analysis_raw.txt
  - printf "\n--- TREE (depth=2) ---\n" >> analysis_raw.txt
  - (if tree installed) tree -L 2 >> analysis_raw.txt || (find . -maxdepth 2 -print | sed 's/^/ /') >> analysis_raw.txt
- Output files: analysis_raw.txt (intermediate)
- Notes: analysis_raw.txt will be parsed by later tasks

Task 3 — Language and package manager detection
- Purpose: detect language(s) and dependency files
- Files to look for (root and typical locations):
  - package.json, yarn.lock, pnpm-lock.yaml
  - pyproject.toml, requirements.txt, Pipfile, Pipfile.lock
  - go.mod
  - Cargo.toml
  - Gemfile
  - composer.json
  - build.gradle, gradlew, settings.gradle
- Commands:
  - grep -Ei "package.json|pyproject.toml|requirements.txt|Pipfile|go.mod|Cargo.toml|Gemfile|composer.json|build.gradle|pom.xml|Dockerfile" -n || true
  - For each found file, record first 200 lines: sed -n '1,200p' <file> >> analysis_raw.txt
- Outputs: language_candidates.txt (list of matches), snippets in analysis_raw.txt

Task 4 — Source layout discovery
- Purpose: identify src directories, packages, CLI entrypoints, and library layout
- Typical paths to check: src/, lib/, pkg/, cmd/, internal/, apps/, services/, packages/, frontend/, backend/
- Commands:
  - for d in src lib pkg cmd internal apps services packages frontend backend; do [ -d "$d" ] && echo "$d" >> source_dirs.txt; done
  - For each dir found, run: find <dir> -maxdepth 2 -type f -print | sed 's/^/  /' >> analysis_raw.txt
- Output: source_dirs.txt, snippets in analysis_raw.txt

Task 5 — Tests, E2E, and CI detection
- Purpose: locate tests and CI workflows
- Paths to inspect:
  - tests/, test/, e2e/, integration/, spec/
  - .github/workflows/*.yml
  - circleci/config.yml, .gitlab-ci.yml, jenkinsfiles, azure-pipelines.yml
- Commands:
  - find . -type f \( -iname "*test*.py" -o -iname "*_test.go" -o -iname "*Test.java" -o -iname "*.spec.js" -o -iname "*.cy.js" -o -iname "*.spec.ts" \) -print > test_files.txt || true
  - ls -la .github/workflows || true; if [ -d .github/workflows ]; then for f in .github/workflows/*.yml .github/workflows/*.yaml; do echo "--- $f ---" >> analysis_raw.txt; sed -n '1,200p' $f >> analysis_raw.txt; done; fi
- Outputs: test_files.txt, CI snippets in analysis_raw.txt
- Notes: record test runners discovered (jest, pytest, mocha, go test, etc.) by examining workflow files and package configs.

Task 6 — Linting, formatting, and tooling
- Purpose: find linting and formatting configs
- Files: .eslintrc*, .prettierrc*, .flake8, pyproject.toml `[tool.black]`, .editorconfig, .golangci.yml
- Commands:
  - grep -Il "eslint|prettier|flake8|black|golangci|editorconfig" -R . || true
  - Record matches in analysis_raw.txt
- Outputs: tooling_list.txt

Task 7 — Docker and deployment artifacts
- Purpose: detect containerization and deployment manifests
- Files: Dockerfile, docker-compose.yml, helm/, charts/, k8s/ or deployment/*.yaml
- Commands:
  - grep -I "FROM \|apiVersion: apps/\|kind: Deployment\|helm" -R --line-number . || true
  - Record found files and snippets
- Outputs: deploy_files.txt

Task 8 — Security and dependency automation
- Purpose: find Dependabot, Snyk, code scanning, or manifest-cves
- Files: .github/dependabot.yml, snyk.*, .github/codeql.yml, .github/workflows (code scanning)
- Commands:
  - [ -f .github/dependabot.yml ] && echo ".github/dependabot.yml" >> security_files.txt || true
  - grep -RIl "dependabot|snyk|codeql|security" . || true
- Outputs: security_files.txt

Task 9 — Documentation and contribution files
- Purpose: collect README, docs/, CONTRIBUTING.md, CHANGELOG.md
- Commands:
  - for f in README.md README.rst docs CONTRIBUTING.md CHANGELOG.md; do [ -e "$f" ] && echo "$f" >> docs_list.txt; done
  - For each, capture first 200 lines into analysis_raw.txt
- Outputs: docs_list.txt

Task 10 — Compose structured findings and recommendations
- Purpose: synthesize the raw data into two artifacts
  1) CODEBASE_ANALYSIS.md — detailed human readable report
  2) analysis_summary.json — small JSON summary for automation
- Exact content and structure (examples below)

CODEBASE_ANALYSIS.md structure (exact path: ./repo/CODEBASE_ANALYSIS.md)
- Title and repo metadata (URL, branch, analysis date, agent id)
- Top-level file/directory summary (short tree)
- Languages and package managers detected (with evidence lines)
- Source layout with key entrypoints and test locations
- CI/workflow summary (which workflows run which commands)
- Tests summary and how to run tests locally (commands found in package.json or CI)
- Linting/formatting and recommended developer commands
- Docker/Kubernetes/Deployment artifacts
- Security and dependency automation
- Recommendations and prioritized next steps
- Appendices: analysis_raw.txt excerpts (or pointers)

analysis_summary.json structure (exact path: ./repo/analysis_summary.json)
- Example JSON schema:
  {
    "repo": "https://github.com/dreef3/multi-agent-harness-test-repo.git",
    "branch": "main",
    "languages": ["JavaScript", "Python"],
    "package_managers": ["npm"],
    "ci": ["github-actions"],
    "tests": { "unit": 12, "e2e": 2, "frameworks": ["jest"] },
    "has_dockerfile": true,
    "has_docs": true
  }
- The agent should populate observed values; use counts where applicable.

Task 11 — Commit and open PR
- Purpose: create a change that adds the two report files
- Commands (agent should use git as usual):
  - git checkout -b analysis/codebase-structure
  - git add CODEBASE_ANALYSIS.md analysis_summary.json
  - git commit -m "Add static codebase analysis report"
  - git push origin analysis/codebase-structure
- Note: Do not create GitHub PR via scripts; the harness will open the PR automatically after planning is saved.

Task 12 — Deliverables and checklist
- Deliverables written to repo root:
  - CODEBASE_ANALYSIS.md
  - analysis_summary.json
- Additional artifacts retained locally: analysis_raw.txt, test_files.txt, source_dirs.txt
- Checklist before committing:
  - Verify reports exist and are under 200KB each
  - Confirm no source files other than new reports were modified

Safety and Constraints
- Do not run or build project code
- Do not store any secrets
- Keep network access minimal (only git clone and optionally any metadata retrieval needed)

Estimated effort
- 20–45 minutes for a single implementation agent depending on repo size (most tasks are simple file scans)

Example command snippets to include in Task descriptions (exact commands for implementer):
- git clone https://github.com/dreef3/multi-agent-harness-test-repo.git repo
- cd repo
- git checkout main
- ls -la
- (if installed) tree -L 2
- find . -maxdepth 3 -type f -name "*test*" -print
- grep -I "scripts\|jest\|pytest\|go test" -R package.json pyproject.toml .github/workflows || true
- sed -n '1,200p' .github/workflows/*.yml || true

Notes for implementers
- If the repo is monorepo (multiple packages), enumerate each package and repeat detection for each package folder (look for package.json in subfolders, workspaces in root config).
- If large files (>200KB) are present, only include excerpts and mention that large artifacts were truncated.

Outputs to the planning system
- After the implementation PR is created and merged, dispatch tasks can be created for follow-up dynamic checks (run tests, collect coverage, security scans). This planning document does not dispatch tasks now.
