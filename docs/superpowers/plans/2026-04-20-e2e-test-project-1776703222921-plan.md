# Implementation Plan: Codebase Structure Analysis for E2E Test Repo 1

Repository: E2E Test Repo 1
Repository ID: 90485bf7-784e-4769-bffe-823dfcbffe0f
Remote URL: https://github.com/dreef3/multi-agent-harness-test-repo.git

This plan lists concrete tasks and exact file locations to add to the repository. Each task is self-contained and can be executed by a sub-agent or a human contributor. No implementation code is included here; instead, tasks describe what must be added and how they should behave.

High-level tasks
----------------
1) Add analyzer scaffolding
2) Add analyzer README and run instructions
3) Add CI workflow to run analyzer and upload artifacts
4) Run analyzer and produce initial report (performed by executor)
5) Review and handoff (human review + PR)

Tasks (detailed)
----------------
Task 1: Add analyzer scaffolding
- Files to add (create):
  - tools/codebase-analyzer/README.md
  - tools/codebase-analyzer/manifest-schema.json  (JSON schema for analyzer output)
  - tools/codebase-analyzer/run_analysis.sh  (executable wrapper to invoke the analyzer)
  - tools/codebase-analyzer/analyzer.sh or analyzer.py (the analyzer implementation) -- NOT PROVIDED HERE
  - tools/codebase-analyzer/output/ (directory where output will be written; should be in .gitignore if large)

- Purpose: Create a self-contained folder with a clear entry point to run the analysis and produce both JSON and Markdown outputs.

- Expected behavior (for implementer to implement):
  - When executed, the analyzer should:
    - Walk the repository root and collect file/directory tree (top ~5 levels)
    - Detect common manifest files (package.json, yarn.lock, pnpm-lock.yaml, pyproject.toml, requirements.txt, Pipfile, Gemfile, go.mod, Cargo.toml, build.gradle, pom.xml)
    - Detect CI config files (.github/workflows/*.yml, .gitlab-ci.yml)
    - Run language statistics (cloc/tokei) if available; otherwise, compute a simple language breakdown by extension
    - Detect Dockerfiles and Docker Compose files
    - Search for tests and common test directories (tests/, spec/) and test runners (jest, pytest, go test, cargo test)
    - Search for TODO/FIXME/XXX comments using ripgrep or a fallback
    - Output a machine-readable JSON report to tools/codebase-analyzer/output/report.json
    - Output a human-readable Markdown summary to docs/codebase-analysis.md

- Output examples (described; implementer to produce):
  - JSON keys: repo, scanned_at, file_tree (sample), languages (map), top_files (list), ci (list), tests (list), dockerfiles (list), notices (list of TODO lines), metrics (counts), recommended_actions (list)

Task 2: Add README and docs
- Files to add or update:
  - tools/codebase-analyzer/README.md (how to run, prerequisites, sample outputs)
  - docs/codebase-analysis.md (initial template with headings for Findings, Languages, Dependency files, CI, Tests, Recommendations)

- Purpose: Provide maintainers and reviewers with clear instructions and a place to store the final analysis.

Task 3: Add CI workflow
- Files to add:
  - .github/workflows/codebase-analysis.yml

- Purpose: Run the analyzer on push and/or on-demand (workflow_dispatch). The workflow should:
  - Set up an environment with commonly used utilities (e.g., install cloc, ripgrep, jq) or use a container image that already contains them
  - Run tools/codebase-analyzer/run_analysis.sh
  - Upload artifacts: tools/codebase-analyzer/output/report.json and docs/codebase-analysis.md

- Workflow specifics for implementer:
  - Trigger: workflow_dispatch and push on default branch
  - Artifact names: codebase-report-json, codebase-report-md

Task 4: Execution step (performed by implementer or CI)
- Steps to execute (these will be run by the sub-agent or maintainer):
  - Clone repository: git clone <repo-url>
  - Checkout default branch
  - Run: tools/codebase-analyzer/run_analysis.sh
  - Observe output in tools/codebase-analyzer/output/report.json and docs/codebase-analysis.md
  - Open PR with the generated docs or attach artifacts to an issue

Task 5: Review and next steps
- After the initial run, a human or sub-agent should:
  - Review docs/codebase-analysis.md and the JSON report
  - Open issues for high-priority findings (e.g., outdated dependencies, missing CI, license mismatch, absence of tests)
  - Optionally, convert the analyzer into a re-usable GitHub App or action

Execution notes and tools
- Preferred tooling: cloc or tokei for language stats, ripgrep for text search, jq for JSON processing, Python 3 for portable scripts.
- Fallbacks: pure-Python implementations to compute basic language breakdowns, file tree, and TODO extraction if the environment lacks above tools.

Security and privacy
- The analyzer is read-only; it must not attempt to modify any files in-place.
- Sensitive files (keys, secrets) should not be output in artifacts. Analyzer should redact obvious secrets (lines that match common secret patterns). Implementers must add basic redaction rules.
