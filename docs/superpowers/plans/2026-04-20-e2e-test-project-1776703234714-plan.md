# Implementation Plan: Codebase Structure Analysis for E2E Test Repo 0

Repository: https://github.com/dreef3/multi-agent-harness-test-repo.git
Repository ID: e93d4511-5b8d-42f4-b6e9-8cdf11e053a4

Overview
- The following tasks will be executed by an implementation agent (or local operator) to inspect the repository and produce both a Markdown report and a JSON metadata file. All steps are read-only.

Outputs (exact paths)
- docs/codebase-structure.md
- reports/codebase-structure.json

Tools and environment
- Unix-like shell (bash)
- ripgrep (rg) recommended, fallback to find/grep
- wc, du, awk, sort, head, sed, jq

Task 1 — Clone repository
- Command:
  git clone https://github.com/dreef3/multi-agent-harness-test-repo.git repo
  cd repo
- Purpose: Create a local copy for inspection. (If CI executes inside an existing checkout, skip clone and cd into repo root.)

Task 2 — Top-level inspection
- Commands:
  pwd
  ls -la
  du -sh .
  git rev-parse --show-toplevel || true
- Purpose: Report repo size and top-level entries. Record top-level directories and files.

Task 3 — List top-level files and directories (save to report)
- Commands (produce both human-friendly and machine-friendly outputs):
  find . -maxdepth 1 -printf "%P\n" | sed '1d' > top-level.txt
  cat top-level.txt
- Purpose: Identify likely project layout (src/, docs/, .github/ etc.).

Task 4 — Language and extension analysis
- Commands:
  # List all files and extensions
  rg --files | sed -n '1,2000p' > all_files.txt
  # Count by extension
  rg --files | sed -E 's/.*\.([^.]+)$/\1/' | sort | uniq -c | sort -rn > lang_by_ext.txt
  # Fallback if rg missing:
  # find . -type f | sed -n '1,2000p' > all_files.txt
- Purpose: Determine predominant languages by file extensions and count files and bytes per extension.

- Additional: bytes per extension
  awk '{ext=sub(/.*\./,"",$0); print}' all_files.txt | xargs -I{} bash -lc 'du -b "{}" 2>/dev/null || true' # (or use a more robust script)

Task 5 — Detect package managers and manifest files
- Files to search for (check existence):
  - package.json, yarn.lock, pnpm-lock.yaml
  - pyproject.toml, requirements.txt, Pipfile
  - go.mod, Gopkg.lock
  - Cargo.toml
  - Gemfile
  - composer.json
  - build.gradle, gradle.properties, pom.xml
  - composer.json
- Commands:
  for f in package.json pyproject.toml requirements.txt go.mod Cargo.toml Gemfile pom.xml build.gradle composer.json Pipfile yarn.lock pnpm-lock.yaml; do [ -f "$f" ] && echo "FOUND: $f"; done > detected_manifests.txt

Task 6 — CI and workflows
- Paths to check:
  - .github/workflows/*.yml
  - .gitlab-ci.yml
  - .circleci/config.yml
  - .travis.yml
- Commands:
  ls -1 .github/workflows 2>/dev/null || true
  [ -f .gitlab-ci.yml ] && echo gitlab
  [ -f .circleci/config.yml ] && echo circleci
  [ -f .travis.yml ] && echo travis
- Purpose: Identify continuous integration configuration.

Task 7 — Docker and containerization
- Check for Dockerfile(s) and compose files
  rg -g '!node_modules' --glob "**/Dockerfile" --files || true
  rg -g '!node_modules' --glob "**/docker-compose*.yml" --files || true
- Purpose: Detect containerization/infra.

Task 8 — Tests and test frameworks
- Look for test directories and common test files:
  - directories: test, tests, spec
  - files: *_test.go, test_*.py, *.spec.js, jest.config.js
- Commands:
  rg --hidden --glob '!node_modules' "(^|/)tests?(/|$)|_test\.py$|_test\.go$|\.spec\.js$" --files || true
- Purpose: Report presence and size of test suite, frameworks used.

Task 9 — Documentation and onboarding files
- Check for README, CONTRIBUTING, CHANGELOG, LICENSE
  for f in README.md README.rst CONTRIBUTING.md CHANGELOG.md LICENSE; do [ -f "$f" ] && echo "FOUND: $f"; done
- Purpose: Assess project documentation and licensing status.

Task 10 — Static code metrics
- File count and total lines:
  total_files=$(rg --files | wc -l)
  total_lines=$(rg --files | xargs -r wc -l | tail -n1)
- Top-10 largest files by bytes:
  rg --files -0 | xargs -0 du -b | sort -rn | head -n 10 > largest_files.txt

Task 11 — Produce reports (Markdown + JSON)
- Generate JSON metadata structure (reports/codebase-structure.json):
  {
    "repo": "https://github.com/dreef3/multi-agent-harness-test-repo.git",
    "size": "...",
    "top_level": [ ... ],
    "languages_by_extension": {"js": 120, "py": 40},
    "manifests": ["package.json"],
    "ci": ["github-actions"],
    "docker": ["Dockerfile"],
    "tests": {"present": true, "paths": ["tests/"]},
    "license": "MIT",
    "suggestions": ["Add CI", "Add LICENSE"]
  }
- Use jq or a small Python script to assemble metadata from intermediate files produced above.

- Generate Markdown report at docs/codebase-structure.md with sections:
  - Quick summary (1–3 bullets)
  - Detailed findings (top-level dirs, languages, manifests, CI, Docker, tests, docs, license)
  - Metrics (total files, lines, repo size, top file types, largest files)
  - Recommendations and next steps

Task 12 — Quality checks and examples
- Verify that both outputs exist and are non-empty:
  [ -s docs/codebase-structure.md ] && echo OK || echo MISSING
  [ -s reports/codebase-structure.json ] && echo OK || echo MISSING

Task 13 — Commit results (if desired)
- If changes should be committed back to repo (follow org policy), steps:
  git checkout -b docs/codebase-analysis
  git add docs/codebase-structure.md reports/codebase-structure.json
  git commit -m "Add codebase structure analysis report"
  git push --set-upstream origin docs/codebase-analysis
- Note: The planning agent will not push or create PRs; this step is optional and must follow repo contributor rules.

Notes and recommendations for implementer
- Prefer ripgrep (rg) for speed and handling of gitignore.
- If running inside CI with shallow clone, ensure workflow checks out full repository to compute accurate sizes.
- Be careful with binary files when computing line counts; use file extension heuristics.
- If repo is large, limit scanning depth or skip vendor/third_party directories.

Estimated effort
- Automated run: ~2–5 minutes for small repositories (<100MB), longer for large ones.
