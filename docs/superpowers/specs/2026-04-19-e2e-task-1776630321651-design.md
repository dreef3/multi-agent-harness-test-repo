# Design Spec: Add repository marker file

Goal
- Create a simple, clearly-named marker file at the repository root to indicate the planning/automation step has been performed. This is a minimal, non-invasive change that can be used by CI or humans to detect that the planning PR was created.

Approach
- Add a single file named REPO_MARKER.md at the repository root.
- File will contain a one-paragraph human-readable message and minimal metadata (date and repository id from the planning context).
- Make a single commit on a short-lived branch and open a PR with a concise title and description.

Key decisions
- File name: REPO_MARKER.md — chosen for visibility and to avoid hidden-dotfiles that might be ignored by some tools.
- Location: repository root — easiest for consumers and for automated checks.
- Content: short text with date and repository id so the file is auditable and unique per run.
- Branch naming: planning/add-repo-marker to make purpose clear.
- Commit message and PR title: explicit and brief (see implementation plan).

Risks and mitigations
- Risk: accidental overwrite of an existing file. Mitigation: implementation task will first check for existence and fail-fast if the file exists, letting the operator choose whether to overwrite.
- Risk: content leaking sensitive info. Mitigation: only include non-sensitive metadata (repository id from plan, date) and an explanatory line.

Non-goals
- Do not modify any code, CI config, or tests. This is strictly a marker file addition.

Approval
- This spec is minimal and can be merged without further design review if the repository owners accept the marker.
