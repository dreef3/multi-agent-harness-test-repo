Design Spec: Codebase Structure Analysis for E2E Test Repo 1

Goal

Produce a concise, accurate analysis of the codebase structure for the repository
identified as "E2E Test Repo 1" (UUID: d4d1a0d6-7f00-4a1b-b2f4-67cdaa66afde,
URL: https://github.com/dreef3/multi-agent-harness-test-repo.git). The analysis
should identify top-level directories and files, primary languages, package and
dependency manifests, CI/configuration files, test locations, build scripts,
and any notable patterns (monorepo layout, microservices, examples, docs).

Constraints and Key Decisions

- I will not clone or inspect the repository in this planning phase (per
  instructions). The spec therefore focuses on approach and verification tasks
  that downstream implementation agents will execute.
- The final analysis will be produced by automated, repeatable tasks run by
  implementation sub-agents. Each task is self-contained and includes exact
  commands to run, expected outputs, and acceptance criteria.
- The implementation will create a human-readable report at path
  docs/codebase-analysis.md in the repository and a machine-readable JSON at
  diagnostics/codebase-summary.json.

Approach

1) Create a small suite of reproducible tasks for a sub-agent to run against the
   repository. Tasks will (a) fetch the repository, (b) produce a file tree, (c)
   detect languages and dependency manifests, (d) locate CI and test assets,
   (e) summarize build/test scripts, and (f) write the analysis artifacts.

2) Use standard tooling where available: tree/find for layout, cloc for language
   breakdown, and manifest-specific parsing for package.json / pyproject.toml
   / go.mod / Gemfile / Cargo.toml etc. If cloc is not present, fall back to
   counting file extensions.

3) Produce two artifacts in the repository:
   - docs/codebase-analysis.md — readable narrative with sections for each
     finding (top-level layout, languages, packages, CI, tests, build, notes).
   - diagnostics/codebase-summary.json — structured data with fields for tools
     to consume (top-level dirs list, language percentages, manifests found,
     CI files, test dirs, suggested next steps).

Success Criteria

- The report docs/codebase-analysis.md exists and contains the required
  sections and concise summary.
- The diagnostics JSON exists and is parseable; top-level dirs, languages, and
  manifest filenames are populated.
- The tasks and commands are deterministic and can be executed by a CI agent.

Next step: implementation plan (task-by-task instructions) will be written and
saved as the planning document of type "plan".
