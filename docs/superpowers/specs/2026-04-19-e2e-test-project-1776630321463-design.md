Design spec: Analyze codebase structure for repository "E2E Test Repo 1" (id: 80d2851d-a196-47e0-b79f-45256dbd1745)

Goal
- Produce a concise, machine- and human-readable inventory of the repository: top-level layout, languages/frameworks, build tools, dependency files, CI and test configuration, Docker/container artifacts, important scripts, and documentation. Deliverables: a Markdown summary (docs/codebase-structure.md) and a JSON metadata file (reports/codebase-structure.json).

Approach
- Non-destructive repository inspection using filesystem and text commands. The analysis will:
  - Enumerate top-level files/directories and classify them.
  - Detect primary languages and package/dependency managers (package.json, pyproject.toml, go.mod, Gemfile, Cargo.toml, etc.).
  - Locate CI and workflow files (.github/workflows/, .gitlab-ci.yml, github actions, azure-pipelines.yml).
  - Find Dockerfiles and container-related configuration.
  - Identify test frameworks and test directories (tests/, spec/, e2e/).
  - Summarize any build/test scripts (Makefile, scripts/, bin/).
  - Capture README, CONTRIBUTING, LICENSE and other docs.
  - Produce an overall assessment and recommended next steps (e.g., add missing docs, unify CI, fix failing tests).

Key decisions and constraints
- The plan assumes the executor has a checked-out copy of the repository (the harness will run tasks inside the repo workspace). The plan will use standard commands (ls, find, rg, sed, head) and read small key files to prevent heavy operations.
- Output artifacts and their paths:
  - docs/codebase-structure.md (human-readable report)
  - reports/codebase-structure.json (structured metadata: languages, build tools, CI, tests, size estimates)
- The analysis will avoid making code changes; it is read-only.

Acceptance criteria
- The Markdown report exists and contains: top-level tree, per-language summary, CI listing, Dockerfiles, tests summary, and a short recommendations section.
- The JSON file contains consistent structured metadata for programmatic consumption.

Assumptions
- Repository is available to agents that will execute the plan; this planning step does not itself access the repository.
- Standard location conventions are used (common filenames listed above).
