Design Spec: Analyze repository structure for E2E Test Repo (id: 8054c555-f5da-4b82-bc63-ea61f8776c2f)

Goal
- Produce a concise, actionable analysis of the codebase structure for the repository https://github.com/dreef3/multi-agent-harness-test-repo.git. The output will identify top-level layout, primary languages and package managers, CI configuration, test and e2e structure, build tools, and any notable issues or maintenance/review priorities.

Approach
- Clone the repository locally (or, in the harness, have a worker clone it) and perform an automated, reproducible set of inspections to enumerate files, directories, and key configuration.
- Commands to run (examples to be executed by implementation tasks): git clone, ls -la, tree -a -L 2, rg/grep for common files, cat/ read important files (README.md, package.json, pyproject.toml, go.mod, Cargo.toml, Gemfile, Makefile, Dockerfile, .github/workflows/*), cloc for language breakdown, and run language-specific commands to list dependencies (npm/yarn/pnpm list, pip freeze from virtualenv, go list -m all, cargo metadata).
- Build a structured report containing: Top-level file and directory list, primary languages and LOC, package managers and dependency manifests, CI/test configuration and commands to run tests, recommended dev and run commands, potential red flags (missing README, pinned deps, no CI, failing tests), and recommended next steps.

Key decisions and assumptions
- We will not perform invasive changes; the analysis is read-only.
- Where multiple dependency files exist, we'll report all and prioritize the one used by CI or documented in README.
- If tests exist, the plan will include commands to run them but will not automatically run them in this planning step.
- The plan will be organized as a set of discrete implementation tasks that can be assigned to sub-agents (clone & inventory, languages/deps, CI & workflows, tests & coverage, final report).

Deliverables
- A written analysis summarizing the repository layout, languages, package managers, CI, tests, and potential issues.
- A set of commands and file references that an implementer can run to reproduce the analysis.
- Recommendations for next steps (fixes, automation, docs) and a checklist for maintainers.

Success criteria
- The produced report lists the top-level directories and files and gives a clear set of next steps and run commands.
- The report includes an inventory of dependency manifests, test entry points, CI workflows, and a list of actionable recommendations.

Constraints
- Do not modify the repository in any way.
- The planning document is for orchestration; execution tasks will be created after this spec is merged.
