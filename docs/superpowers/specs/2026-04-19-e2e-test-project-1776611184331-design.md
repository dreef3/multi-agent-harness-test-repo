Design Spec: Analyze codebase structure for E2E Test Repo

Goal

The analysis should identify languages, package managers, top-level directories, CI configuration, test harnesses, Docker/containers, important scripts, and potential risks or smells.

Approach
- Because I am not allowed to clone or inspect the repository files directly in this phase, this spec defines the methodology and checks that a follow-up implementation task will run to produce a concrete analysis. The methodology focuses on an automated, reproducible checklist that a sub-agent or developer can run locally or in CI to collect the repository structure and metadata.

Deliverables

- CI/CD configuration summary (.github/workflows, .gitlab-ci.yml, etc.)
- Docker/Dockerfile and devcontainer configurations
- Test harness and E2E test locations and frameworks (e.g., tests/, e2e/, cypress/, playwright/)
- Notable scripts (Makefile, scripts/, bin/) and developer tooling

Key Decisions and Constraints
- Do not clone or inspect the repository in this planning stage. The plan will list exact commands and file paths that must be run by an implementer.
- Focus on automatable, reproducible checks that create a concise human-readable summary and a machine-readable artifact (JSON) containing the metadata.
- Keep the scope limited to structure and metadata (no code quality analysis beyond obvious signs such as missing tests or CI).
