# Design Spec: Add a simple marker file to repository

Goal

- Create a one-file marker in the repository to indicate the harness/agent has run a planning step. The marker file should be minimal, human-readable, and placed at the repository root.

Approach

- Add a new file at the repository root named MARKER.md (plain text/Markdown to be visible in GitHub UI).
- The file will contain a short message, the repository identifier from the provided context, and a placeholder for the creation timestamp and author so the CI/harness or implementation agent can fill in exact values when applying the change.

Key decisions

- File name: MARKER.md — chosen because it is visible in GitHub UI and unlikely to be ignored. Alternative names considered: .pi-marker (hidden) or MARKER (no extension); MARKER.md chosen for clarity.
- Location: repository root — simplest and most discoverable location.
- Content: short, static lines plus placeholders for dynamic data (timestamp, actor) so the committing agent can populate them.
- Single-task change: keep the change minimal (one file) to reduce code review friction and risk.

Constraints

- Do not modify any existing files.
- Do not run git or create commits here; the harness will create the branch/PR from the planning document.
- Implementation must be self-contained and include exact file content to be written.

Success criteria

- A new file MARKER.md exists at the repository root containing the expected marker text and repository id.
- A PR is opened on the repository default branch with a clear title and commit message.
