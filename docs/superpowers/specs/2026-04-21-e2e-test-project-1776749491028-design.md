Design Spec: Analyze codebase structure for E2E Test Repo

Goal
- Produce a clear, actionable analysis of the repository https://github.com/dreef3/multi-agent-harness-test-repo.git (repo id: 6ed2cd10-da8f-48a5-9543-b070a7487fea). The analysis must describe the project's layout, primary languages, dependency manifests, build and test infra, CI, notable files, and surface any risks or maintenance hotspots (large files, generated code, missing tests, unclear structure).

Approach
- Perform an automated, reproducible repository inspection and manual review steps (to be executed by an implementation agent). The inspection will collect: top-level tree, language breakdown, dependency files (package.json / pyproject.toml / Cargo.toml / go.mod / Gemfile / build.gradle etc.), build/test commands, CI workflows (.github/workflows, GitLab CI), Dockerfiles, source directories (src, lib, pkg, cmd), tests and coverage reports, and notable docs (README, CONTRIBUTING, LICENSE).
- Produce a human-readable report including a file tree snapshot, metrics (file counts, lines of code by language), dependency summary, CI status summary (workflows present and what they do), and recommendations (e.g., add README sections, consolidate manifests, add tests, remove generated files, add .gitignore entries).

Key decisions and scope
- We will not modify repository contents. Analysis is read-only.
- The analysis will prioritize clear, actionable outputs: (1) Summary page, (2) File tree and notable files, (3) Language and size metrics, (4) Dependency and build/test mapping, (5) CI/workflow mapping, (6) Risk and improvement suggestions.
- Tools and commands used by implementers are explicitly documented (git clone, tree, find, rg/grep, cloc or github-linguist, parsing dependency files). Where a tool might not be installed, alternatives are provided.

Deliverables
- A short executive-summary markdown (README-style) with key findings.
- A detailed report.md including commands run, raw outputs (or file references), and recommendations.
- A small JSON metadata file with computed metrics (language lines, file counts, test files count, presence of CI, presence of dockerfiles).

Constraints
- Implementation must not push changes to the repo. The analysis artifacts (report and metadata) should be written to a new directory named analysis/<repo-name>-analysis in the local workspace or in the planning PR branch as instructed by the harness.

Success criteria
- The report contains: top-level file tree, primary languages with LOC, list of dependency manifests and their contents summary, list of CI workflow files with purpose, list of tests and coverage instructions (if any), and 5 prioritized recommendations.
