Design Spec: Codebase Structural Analysis for E2E Test Repo 1

Goal
- Produce a thorough, reproducible analysis of the repository "multi-agent-harness-test-repo" (id: 0e3d8ac6-333a-4b53-b987-30dd94883eae) describing its codebase structure, languages, dependency surfaces, test and CI setup, and notable files/dirs. Deliverables: a human-readable Markdown report and a machine-readable JSON summary placed under a dedicated artifacts directory in the repository.

Approach
- Automated repository inspection performed by a small sequence of non-destructive tasks that list files, detect languages, read key manifest files (README, package manifests, build files, CI workflows), enumerate top-level packages/modules, and gather test/CI commands. The inspection will collect metadata (file counts, LOC approximations per language, dependency manifests, test frameworks, scripts), and produce a prioritized list of findings and potential risks (missing tests, outdated lockfiles, or monorepo quirks).
- Use standard CLI tools for deterministic results: find/ls/tree for structure, rg (ripgrep) for quick pattern searches, cloc or simple wc-based heuristics for line counts, and JSON extraction from known manifest files (package.json, pyproject.toml, go.mod, Cargo.toml, requirements*.txt). The plan includes exact commands to run so sub-agents can execute them.

Key Decisions
- Non-destructive: the analysis will not modify repository files. All outputs will be written to a new artifacts/analysis/ directory.
- Output formats: Markdown for human readers (artifacts/codebase_analysis/report.md) and JSON for tooling (artifacts/codebase_analysis/summary.json). Optionally include CSV/TSV for tabular data (artifacts/codebase_analysis/file_list.csv).
- Scope: Top-level and src/test directories, CI workflows (.github/workflows), package manifests, docs, scripts, and Dockerfiles. Do not attempt to run tests or install dependencies (no network side-effects) — only static analysis.
- Reproducibility: include the exact commands used and environment assumptions (Linux shell, coreutils, ripgrep, jq, cloc if available). If optional tools are missing, fallback commands are supplied.

Acceptance Criteria
- Generated artifacts contain: (1) repository tree snapshot, (2) language breakdown with LOC estimates, (3) list of package/manifest files and parsed dependency entries, (4) CI and test configuration summary, (5) notable files and directories with short descriptions, (6) recommended next steps and risks.

Security and Privacy
- Do not expose secrets. If any files matching secret patterns are discovered (e.g., *.pem, AWS keys), record their path but do not dump contents in the report. Flag them as sensitive findings.
