Design Spec: Codebase Structure Analysis

Goal
- Produce a clear, automated analysis of the E2E Test Repo (multi-agent-harness-test-repo) that summarizes repository layout, languages, dependency manifests, CI, test harnesses, important files (Docker, Makefiles), and actionable observations (missing tests, key modules, potential maintenance risks). Deliver machine-readable and human-readable reports saved into the repository under /reports.

Approach
- Create an automated analysis workflow (shell script + lightweight tools) to be run by an implementation agent that clones the repo into a sandbox, enumerates files and directories, detects languages and line counts (cloc), parses package manifests (package.json, pyproject.toml, go.mod, Cargo.toml, pom.xml, etc.), inspects CI config (.github/workflows/, .gitlab-ci.yml, azure-pipelines.yml), locates tests and test commands, and gathers Docker/Makefiles. The workflow will not modify production code; outputs are written under reports/ and commit them on a feature branch.
- Produce two output artifacts: reports/codebase-structure.md (human-readable summary) and reports/codebase-structure.json (structured data for further automation). Optionally include additional artifacts: reports/dependencies.txt and reports/ci-summary.md.
- Keep operations safe: prefer static inspection, avoid network-heavy operations or executing arbitrary code. If running tests is required, do so in an isolated CI runner and only with explicit approval.

Key Decisions
- Language detection: use cloc for line counts; detect manifests by filename patterns rather than heuristics alone.
- Dependency parsing: extract declared dependencies from manifest files without performing installs by default. Optionally run package manager commands (npm ci, pip install -r requirements.txt) behind an opt-in flag and in an isolated environment if the user wants runtime dependency resolution.
- Test execution: identify test commands and test frameworks; do not run tests by default to avoid side effects. Provide instructions for running tests locally or in CI.
- Output formats: Markdown (human) and JSON (machine) stored under /reports. Provide a small summary top-level README snippet in reports/README.md.
- Safety and reproducibility: scripts must check for existence of tools (cloc, jq, rg) and fall back to POSIX tools where possible.

Acceptance criteria
- Reports generated at repo path: reports/codebase-structure.md and reports/codebase-structure.json
- The human-readable report contains: top-level file/directory list, language breakdown, list of manifest files and extracted dependencies, CI workflows found, test frameworks and suggested test commands, Docker/Makefile presence, license file, README summary, and recommendations.
- The JSON output contains structured fields for all sections above for programmatic consumption.
