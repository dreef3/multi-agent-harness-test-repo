Design Spec: Codebase Structure Analysis for E2E Test Repo 1

Goal

Produce a clear, repeatable analysis of the codebase structure for the repository "E2E Test Repo 1" (id: 8a57706d-7e6a-464a-bd9e-1b182af603da, URL: https://github.com/dreef3/multi-agent-harness-test-repo.git). The final deliverable will be a short, readable report describing the repository layout, primary languages, important files (CI, Docker, package manifests), test locations and status, and actionable notes / next steps for maintainers.

Approach

- Run a deterministic set of static inspection steps on a checked-out copy of the repository to collect filesystem layout, language breakdown, dependency manifests, CI/workflow files, Docker and build artifacts, and tests.
- Produce machine-readable artifacts (JSON) for programmatic consumption and a human-friendly Markdown report summarizing findings and recommendations.
- Use standard CLIs where available (tree, find, rg/grep, cloc) and fall back to shell commands for environments lacking them.

Key decisions

- Non-intrusive read-only analysis: no code changes, no network requests to external package managers beyond optional dependency vulnerability scans which must be explicit tasks.
- Output artifacts placed under analysis/ in the repository root so they can be reviewed in a PR or uploaded to a results store.
- Use both a concise human summary (report.md) and structured outputs (cloc.json, file_tree.txt, dependencies.json) to serve both maintainers and automation.
- The plan below is written so it can be executed by an automation agent (or developer) with git access; the planning repo will track the spec and the plan via the planning branch/PR.

Primary success criteria

- A Markdown report at analysis/report.md summarizing the repository structure and listing high-value objects (language summary, top-level directories, CI files, dockerfiles, dependency files, test locations).
- Machine-readable outputs: analysis/file_tree.txt, analysis/cloc.json, analysis/dependencies.json, analysis/ci_files.txt, analysis/tests_summary.md.

Security and constraints

- Do not modify repository files other than adding the analysis/ artifacts.
- If dependency vulnerability scanning is included, it should be an optional follow-up task and require explicit permission.


Brief timeline

- Immediate: create planning documents (this spec + implementation plan) and open planning PR.
- Execution (by a repo-analyzer agent / developer): 30–60 minutes to run and produce artifacts for a mid-sized repo (<1000 files).
