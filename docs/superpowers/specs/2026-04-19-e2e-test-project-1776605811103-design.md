Design Spec: Codebase Structure Analysis for E2E Test Repo

Goal
- Produce a clear, actionable analysis of the repository "multi-agent-harness-test-repo" (E2E Test Repo, id: 38166595-410c-4949-81c4-ef911bacb24f). The analysis should identify the repository's high-level structure, primary languages/frameworks, entry points, test layout, CI configuration, packaging/installation files, and any obvious risks or maintenance concerns.

Scope and constraints
- I will not clone or open the repository locally (per instructions). This design spec and the subsequent implementation plan describe a reproducible, step-by-step approach that an agent or human can run against the repository to produce the analysis.
- The output of the plan will be a written report (README-style) containing: repository tree, language/libraries summary, dependency list, test layout and how to run tests, CI workflows summary, containerization/Docker usage, common issues, and recommendations.

Approach
- Use standard repository inspection commands and file reads to extract structure and metadata. Key artifacts to inspect include: README.md, LICENSE, package manifests (package.json, pyproject.toml, setup.py, requirements.txt, Pipfile, go.mod, Cargo.toml), src/ or lib/ folders, tests/ or spec/ folders, examples/, .github/workflows/, Dockerfile(s), Makefile, scripts/, and CI configuration files.
- Derive primary languages and frameworks from manifest files and the layout of source directories.
- Extract dependency information from package manifests and summarize top-level dependencies and their versions.
- Examine test runner configs (pytest.ini, tox.ini, jest.config.js, etc.) and tests/ to map test coverage and E2E test entry points.
- Collect CI workflows in .github/workflows and summarize steps, matrixes, and coverage/upload steps.
- Note any Linting/formatting configs (.eslintrc, .prettierrc, .flake8) and developer tooling (husky, pre-commit).

Key decisions
- Focus first on high-value files: root manifests, CI workflows, test directories, and Dockerfiles. These give the fastest signal about language/stack and intended usage.
- Produce machine-friendly artifacts: a tree listing, a dependency summary, and a mapping of tests/commands to run. These will support automation and future tasks.
- Emphasize reproducible commands to run in the repo so another agent can pick up and execute them.

Deliverables
- A single analysis report (Markdown) summarizing findings and recommendations.
- A set of reproducible shell commands and file reads to re-create the analysis.

Acceptance criteria
- The report contains: repository tree (top 2–3 levels), language and framework identification, dependency summary, test layout and run instructions, CI summary, list of potential maintenance risks, and recommended first follow-up actions.

Constraints and assumptions
- The repository follows common conventions (manifest filenames and directory names). If the repo uses uncommon structures, the plan includes exploratory commands to discover them.

Next step
- Execute the implementation plan (a detailed, task-by-task runbook) to collect files and produce the final analysis report.
