Design spec: Analyze codebase structure for repository 'E2E Test Repo 1' (id: 12ad4237-eee1-4a87-83d0-858bd7518e3b, https://github.com/dreef3/multi-agent-harness-test-repo.git)

Goal
- Produce a concise, actionable analysis of the repository's structure: languages, package managers, top-level layout (src/, pkg/, cmd/, tests/, e2e/), CI, Docker, important manifests, test frameworks, and any monorepo or workspace patterns.

Constraints
- Per instructions, do not clone or explore the repository in this planning step. All analysis steps must be planned but not executed here.
- The planning document must be saved to the planning branch via the harness API.

Approach
- Define a sequence of reproducible tasks and shell commands to be executed by an implementer or a sub-agent that will clone and inspect the repository. Tasks will collect structured evidence: file listings, manifest contents, CI workflow files, test directories, and language detection.
- Provide a template script and commands that produce a machine-readable summary (JSON) and a human-readable report.
- Identify exact file paths to check, common filenames to look for, and heuristics to detect monorepo/workspace setups and frameworks.

Key decisions
- The plan avoids making any repository-specific assumptions beyond the general patterns above; it will detect specifics by examining the actual files after cloning.
- The output will include both a short executive summary and a detailed inventory (file-level) so reviewers can quickly understand the repo and also dive deeper.
- Provide commands that are safe (read-only) and do not run tests or modify the repo.

Success criteria
- A completed analysis will include: (1) language(s) and versions where available; (2) dependency manifest files and a short list of top-level dependencies; (3) CI configuration and workflows; (4) test layout and recommended test/run commands; (5) Docker and containerization files; (6) any monorepo/workspace hints; (7) license and contribution files; (8) structured JSON and a human-readable report saved to repository-analysis/<timestamp>/report.{json,md} or printed to stdout.

Next step
