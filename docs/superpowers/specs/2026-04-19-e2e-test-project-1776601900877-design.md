Design spec: Analyze codebase structure for repository c4473d5e-1135-4e4c-9828-3a97d3036e45 (https://github.com/dreef3/multi-agent-harness-test-repo.git)

Goal
- Produce a clear, actionable analysis of the repository's codebase structure: top-level layout, languages and package managers in use, build/test/CI tooling, key entry points, library vs. application separation, test coverage locations (unit/e2e), and potential maintenance issues (big files, outdated configs, missing CI, etc.).

Approach
- Static repository inspection (no code execution): list and categorize files and folders, read key manifest/configuration files, and summarize findings.
- Focus areas: top-level files (README, LICENSE), language manifests (package.json/pyproject.toml/go.mod/etc.), CI workflows (.github/workflows), Dockerfiles, build scripts (Makefile, scripts/), source code layout (src/, cmd/, pkg/, lib/), tests/ or e2e/ directories, and docs/.
- Produce a summary report with: tree of top-level directories and their purpose, detected languages and package systems, CI and deployment config, test locations, potential technical debt hotspots, and suggested next steps.

Key decisions
- Prioritize automated, repeatable inspection steps that can be performed by CI or sub-agents.
- Provide concrete file paths and shell commands that an implementer can run without additional context.
- Output a single Markdown analysis file in the repo (docs/codebase-structure-analysis.md) and an accompanying machine-readable JSON summary (reports/codebase-structure.json).

Deliverables
- docs/codebase-structure-analysis.md — human-readable analysis and recommendations.
- reports/codebase-structure.json — structured summary (top-level directories, languages, manifests found, CI files, tests present).

Constraints
- Do not run code from the repository or install dependencies; only static file reads.
- Keep the analysis neutral and factual; avoid making unverifiable claims.

Acceptance criteria
- The analysis file exists at docs/codebase-structure-analysis.md and contains a top-level inventory and concise recommendations.
- The JSON report contains keys: top_level, languages, manifests, ci, tests, dockerfiles, notable_files.
