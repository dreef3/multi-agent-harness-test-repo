Design spec: Codebase Structure Analysis for E2E Test Repo

Goal
- Produce a concise, actionable analysis of the repository https://github.com/dreef3/multi-agent-harness-test-repo.git (repo id: 8d4da9fc-ff5d-448f-a5fe-9373bfeaa4ca). The deliverables are: a human-readable report (analysis/REPORT.md) and a machine-readable summary (analysis/summary.json) describing languages, top-level directories, dependency manifests, CI configuration, test layout, build/run commands, containerization, docs, licensing, and potential red flags (missing tests, large binary blobs, no CI, outdated deps, common security issues).

Approach
- Non-invasive, read-only static analysis executed by CI/sub-agents that inspect repository files. The analysis will use simple shell commands and small inspection scripts to detect language ecosystems, important manifests, CI workflows, test frameworks, and basic metrics (file counts, lines of code by language, size of large files). The analysis produces both granular outputs (lists of files, excerpts from manifest files) and high-level conclusions.

Key decisions
- Do not modify repository contents. All tasks are read-only.
- Favor broad coverage: detect common package managers for JS/TS, Python, Go, Rust, Ruby, Java, .NET, and container artifacts (Dockerfile, docker-compose). Look for CI configs in .github/workflows and popular CI files (.gitlab-ci.yml, circleci/config.yml, .github/workflows/*).
- Produce outputs in analysis/ to make them easy to review in a PR.
- Provide both human guidance and automated outputs so maintainers can act on findings.

Outputs
- analysis/REPORT.md — narrative summary of structure, key files, recommended next steps.
- analysis/summary.json — structured description with arrays for top-level dirs, languages, package/manifests found, CI files, test files, Dockerfiles, large files (>5MB), license file, and problems flagged.
- scripts/analyze_repo.sh — orchestration script (idempotent, read-only) that generates both outputs when run at repo root. The plan will not commit code — the planning PR will add the spec and plan only; implementation tasks will be dispatched after planning PR is merged.

Acceptance criteria
- The analysis script runs without network access and produces the two outputs above when executed in the repository root.
- The REPORT.md contains the high-level summary and actionable recommendations.
