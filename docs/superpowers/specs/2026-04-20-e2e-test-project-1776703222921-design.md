# Design Spec: Codebase Structure Analysis for E2E Test Repo 1

## Goal
Perform a comprehensive static analysis of the codebase structure for the repository:

- Repository: E2E Test Repo 1
- Repository ID: 90485bf7-784e-4769-bffe-823dfcbffe0f
- Remote URL: https://github.com/dreef3/multi-agent-harness-test-repo.git

Deliver a clear, reproducible report that describes the repository layout, primary languages and frameworks, dependency manifests, test and CI configuration, notable risks, and recommended next steps for maintainers.

## Approach
Because this planning step must not clone or inspect the repo directly, the plan is to create a reproducible analysis workflow (scripts + CI job) that an implementation agent or human operator can run against the target repository. The analysis will be automated and non-invasive (read-only). The analyzer will gather both high-level and detailed artifacts:

- Top-level file and directory tree
- Language detection (by extension and by token counting)
- Dependency manifests and versions (package.json, requirements.txt, pyproject.toml, Cargo.toml, go.mod, Gemfile, etc.)
- CI configuration (GitHub Actions, .gitlab-ci.yml)
- Test frameworks and test commands
- Build scripts and Dockerfiles
- Monorepo/workspace indicators (workspaces in package.json, Nx, Lerna, turborepo, multiple language roots)
- Size metrics: repository size, counts of files per language
- TODOs, FIXMEs, and conspicuous TODO-like annotations
- License and CODEOWNERS

Key outputs will be a human-readable Markdown report and a machine-readable JSON report.

## Key Decisions
- Non-invasive and reproducible: generate scripts that run locally or in CI, and produce both JSON + Markdown outputs.
- Use common, well-known CLI tools where available (cloc/tokei for language stats, ripgrep for TODOs, jq for JSON processing). If those tools are not available in the execution environment, provide pure-Python fallbacks.
- Report must include the commands used and a summary so maintainers can reproduce results.
- The analysis will be implemented as a self-contained tool inside the repo (tools/codebase-analyzer/) to facilitate running and maintenance by repository owners.

## Success Criteria
- A Markdown report at docs/codebase-analysis.md summarizing the repository structure and findings.
- A JSON report at tools/codebase-analyzer/output/report.json containing structured data produced by the analyzer.
- A GitHub Action (or instructions) that runs the analyzer and uploads the JSON and Markdown reports as CI artifacts.


## Constraints
- This spec does not clone or inspect the repository (per the planning constraints).
- Implementation tasks will be authored so they can be executed by a follow-up sub-agent which will clone the repository and run the analyzer.
