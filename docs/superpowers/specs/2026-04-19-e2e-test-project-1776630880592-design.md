Design spec: Analyze codebase structure for repository 'E2E Test Repo 0' (id: d16f98b0-7188-46b4-9599-74fd18f33525)

Goal
- Produce a concise, actionable summary of the repository's structure, primary languages, packages and modules, test layout, CI/configuration, and notable files (README, LICENSE, Dockerfile, etc.).

Approach
- Automatic inspection steps will be performed in the repository working tree to extract:
  - Top-level directory and file tree (1-3 levels)
  - Language breakdown (cloc or linguist)
  - Dependency manifests (package.json, requirements.txt, go.mod, pyproject.toml, etc.)
  - CI and automation (GitHub Actions, CircleCI, GitLab CI files)
  - Test locations and test commands
  - Dockerfiles and infrastructure IaC (Terraform, helm, k8s manifests)
  - Notable scripts (Makefile, build scripts)
- Produce machine-readable artifacts and human-friendly reports in /reports:
  - file-tree.txt
  - language-stats.json
  - dependency-list.json
  - test-summary.md
  - ci-summary.md
  - CODEBASE_SUMMARY.md (final readable analysis)

Key decisions and assumptions
- Focus on static file inspection and manifest parsing; no runtime or network operations.
- Exclude secrets and binary blobs from reports.
- Prioritize README/CONTRIBUTING/LICENSE for human context.
- If multiple language ecosystems are present, surface the primary ones by lines-of-code and presence of package managers.
- Use CLI tools available in the agent environment (ls, find, rg, jq, cloc) or provide fallback parsing instructions if tools are unavailable.

Success criteria
- A CODEBASE_SUMMARY.md that answers: What is this repo for? What languages and frameworks are used? Where are the services/packages? Where are tests and how to run them? What CI automation exists?
- JSON/flat artifacts that can be consumed by other automation.

Notes
- This is a planning/spec document only; implementation tasks are defined in the companion implementation plan.
