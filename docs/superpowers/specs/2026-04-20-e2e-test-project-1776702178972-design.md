Design Spec: Analyze codebase structure for E2E Test Repo

Goal
- Produce a clear, actionable analysis of the repository "multi-agent-harness-test-repo" (E2E Test Repo, id: 029162f8-e68f-4b75-a119-4d8d0eee8e88, URL: https://github.com/dreef3/multi-agent-harness-test-repo.git). The output will summarize the repository layout, primary languages and frameworks, build/test/CI configuration, notable files (README, LICENSE, Dockerfile, manifest/package files), test coverage surface, and any apparent maintenance risks (large binary blobs, outdated dependencies, missing CI, TODOs, high churn files).

Approach
- Perform an automated, reproducible repository inspection using standard filesystem and language-detection tools executed against a fresh clone of the repository. The inspection will include a top-level directory listing, recursive tree, language summary (cloc or GitHub linguist), dependency manifest detection (package.json, pyproject.toml, go.mod, Gemfile, Cargo.toml, etc.), CI/workflow detection (.github/workflows, .gitlab-ci.yml), containerization (Dockerfile), build scripts (Makefile, build.gradle, mvnw), test suites (pytest, jest, go test, etc.), and linting/config files (.eslintrc, .prettierrc, editorconfig).
- Produce a human-readable report with: (1) high-level repo summary, (2) language & size breakdown, (3) key files and where they live, (4) CI and test status and how to run tests locally, (5) recommended next steps (fixes, risk mitigation, suggestions for contributor onboarding), and (6) an artifacts section listing exact commands used and outputs collected.

Key decisions
- Use standard CLI tools (ls, find, rg, cloc, du, git log) for deterministic outputs that sub-agents can reproduce. Avoid destructive operations.
- Focus on static analysis only (no running of arbitrary untrusted code). Tests may be listed and described but will not be executed by the analysis unless specifically authorized.
- Produce a modular task plan so sub-agents can run tasks in parallel: repository clone & metadata, filesystem inventory, language and dependency detection, CI/workflow analysis, tests and coverage discovery, and final reporting.
- Deliverables: a concise summary (for humans), a machine-readable inventory (JSON), and a guided checklist for maintainers.

Constraints
- This planning document does not access the repository contents itself; it's a design for how to analyze the repo. Implementation tasks in the plan will reference the repository UUID provided by the user for sub-agent execution.
