Design Spec: Codebase Structure Analysis for "E2E Test Repo 0" (multi-agent-harness-test-repo)

Goal
- Produce a concise, actionable analysis of the repository structure for the repo identified as E2E Test Repo 0 (id: 8f897ac1-8fb1-44a3-a525-98ec7ec02959). Deliverables: a high-level summary, a file-by-file inventory, dependency overview, CI/test coverage surface, documentation/README assessment, and prioritized recommendations.

Approach
- Automated inventory: run file-system scans to collect a file tree, detect language(s), locate manifest files (package.json, pyproject.toml, go.mod, Cargo.toml, etc.), find Dockerfiles and CI workflows, and list tests.
- Content sampling: open README, CONTRIBUTING, LICENSE, and top-level manifests to extract project purpose, install/run/test instructions, and license.
- Dependency analysis: parse manifest files to list direct dependencies and tooling (formatters, linters, test frameworks). Produce a short risk/maintenance note (outdated packages, pinned versions, etc.).
- CI & testing scan: locate .github/workflows, GitLab CI, Travis, or other CI config and summarize steps (build/test/lint/deploy). Check for test folders and run commands noted in manifests/README.
- Entrypoints & architecture: identify main source directories, executable entrypoints (bin/, cmd/, main.py, index.js), and any microservices or package modules.
- Maintainability & security checks: enumerate any missing/weak areas—lack of tests, no lockfile, missing license, unclear README, secrets in repo—flag high-priority issues.

Key Decisions and Scope
- This spec focuses on structural analysis and actionable recommendations, not code-level static analysis or test execution.
- The initial analysis will rely on repository files available in the default branch (main). It will not modify repo contents.
- Deliver intermediate outputs as a small report and a prioritized checklist for remediation.

Deliverables
- Summary report (1 page) describing languages, key files, CI/tests, and maintainability score.
- Full file inventory (tree) limited to top 4 levels.
- Dependency list and suggested commands to generate a dependency graph.
- Prioritized recommendations and quick fixes (e.g., add LICENSE, add CI tests, create README sections).

Success Criteria
- The produced report should allow a developer unfamiliar with the project to understand how to build, test, and contribute within 15 minutes, and it should list 5–10 concrete next steps for improvement.

Notes
- This is a planning/spec phase only. The next step (implementation) will list discrete tasks that agents can run to collect the data described above.
