# Design Spec: Codebase Structure Analysis — E2E Test Repo 0

Goal

- Produce a concise, actionable static analysis of the repository `multi-agent-harness-test-repo` (id: cd7b7fc8-a8a9-48a0-956b-eec84f94d280) to identify its structure, primary languages, dependency manifests, CI, tests, Dockerization, and potential maintenance or security concerns.

Approach

- Use non-invasive, repository-level inspection (file-listing and manifest parsing). The analysis will focus on top-level files and common language/build/CI manifests to determine how the project is organized and how to run or test it.
- The work will be split into targeted checks: top-level tree, language detection, dependency manifests, CI/automation, tests, Docker, docs & licensing, size and code metrics, and housekeeping (TODOs, insecure patterns).
- Produce a human-readable summary report and a compact checklist of recommended next actions (e.g., add CI, fix license, modernize manifests).

Key decisions

- Prioritize common manifests and locations: README*, LICENSE, package.json, pyproject.toml, requirements.txt, go.mod, Cargo.toml, pom.xml, build.gradle, Dockerfile, docker-compose.yml, .github/workflows, tests/, src/ or pkg/.
- Do not run tests or build steps during analysis; analysis is static only. Any execution steps will be explicit and optional tasks for follow-up agents.
- Output artifacts: a markdown report saved at reports/multi-agent-harness-test-repo-structure-analysis.md and a JSON summary at reports/multi-agent-harness-test-repo-structure-summary.json for machine consumption.

Deliverables

- reports/multi-agent-harness-test-repo-structure-analysis.md — human readable findings and recommendations.
- reports/multi-agent-harness-test-repo-structure-summary.json — structured metadata: languages, manifests found, CI, Docker, tests, top-level dirs, files of interest.

Estimated effort

- Quick scan (top-level, manifests): 10–20 minutes.
- Full manifest parsing and metrics (cloc, TODOs, dependency counts): 30–60 minutes.

Constraints

- This spec assumes ability to clone or read the repository contents. The planned tasks will include cloning; if cloning is not permitted by the harness, the implementer should instead use the harness-provided repo checkout.

QA and verification

- The report will include command-lines used for discovery and the exact file paths that were inspected so reviewers can reproduce the analysis.
