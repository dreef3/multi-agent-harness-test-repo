Design Spec: Codebase Structure Analysis for E2E Test Repo

Goal
- Produce a comprehensive high-level analysis of the codebase structure for the repository
  - Repository: E2E Test Repo
  - ID: 6dbe2ce2-afd8-4ae6-94cf-41f00f1c7184
  - URL: https://github.com/dreef3/multi-agent-harness-test-repo.git

Scope
- Identify top-level layout, languages and frameworks, package/manifest files, build/test tooling, CI configuration, documentation, and likely areas of technical debt or maintenance risk.
- Produce a concise written report summarizing findings and actionable next steps (fixes, improvements, documentation gaps, tests to add).

Approach
- Static repository inspection only (no running of project code or tests). Steps will include obtaining a copy of the repository (clone or download), enumerating top-level files and directories, detecting language(s) and package managers, locating CI config, reading README and contributing docs, enumerating source and test directories, listing build/test scripts, searching for common markers (TODO/FIXME), and producing a summary report.

Key decisions
- The analysis will not execute any project code, nor run builds/tests.
- The report will focus on structure, toolchain, and maintainability signals rather than runtime correctness.
- The plan will be written as a set of discrete, reproducible tasks so it can be executed by any engineer or automated agent.

Deliverables
- A final analysis report (README-format) with:
  - Top-level inventory (files/dirs)
  - Languages and package manifests
  - CI and automation
  - Source vs tests mapping
  - Notable findings (missing docs, inconsistent structure, TODOs)
  - Suggested next actions and prioritized remediation list

Constraints & Assumptions
- The repository URL and ID from the request are accurate and accessible.
- The executor of the plan has network access and permissions to clone the repository.
- No runtime or build steps will be performed; only file inspection and static analysis.

Success criteria
- A readable report that answers: what is in the repo, how it is organized, what tooling it uses, where tests/CI live, and what to fix next.
