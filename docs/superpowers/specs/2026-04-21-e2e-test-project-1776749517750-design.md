Design spec: Analyze codebase structure for repository d585b833-b17d-4df6-ad8b-52dfc9d49758 (https://github.com/dreef3/multi-agent-harness-test-repo.git)

Goal
- Produce a concise, actionable analysis of the repository structure: languages, packages/manifests, top-level directories, build/test tooling, CI, entry points, docs, and gaps or recommended next steps.

Approach
- Perform a systematic filesystem-level inspection of the repository root and key files without modifying the repo. Steps:
  1. Produce a complete top-level file tree (depth 2–3) to understand layout.
  2. Detect primary languages and count lines per language (cloc or linguist-like summary).
  3. Identify package manifests and dependency files (package.json, pyproject.toml, go.mod, Gemfile, Cargo.toml, etc.) and extract top-level metadata (name, version, main/entry points, scripts).
  4. Inspect CI pipelines (.github/workflows, .gitlab-ci.yml) and Dockerfile/Makefile for build/test flow.
  5. Locate tests and test runner configuration (jest, pytest, go test, etc.), and run coverage/test commands where possible (or document how to run them).
  6. Review documentation (README.md, docs/) and CONTRIBUTING.md for developer setup.
  7. Summarize findings, highlight risks (e.g., missing CI, no tests, locked dependencies), and propose recommendations.

Key decisions
- Prioritize manifests and CI to determine how the project is built and tested.
- Provide both automatic commands (cloc, tree, jq) and manual file checks so the implementer can reproduce results in different environments.
- Deliver outputs as a single self-contained report (REPORT.md) plus supporting artifacts (file-tree.txt, dependencies.json, ci-summary.md).

Success criteria / deliverables
- REPORT.md: high-level summary with recommendations.
- artifacts/file-tree.txt: repository tree (depth 3).
- artifacts/language-summary.json: language breakdown (cloc output or equivalent).
- artifacts/dependency-manifests.md: extracted metadata from manifests.
- artifacts/ci-summary.md: list and short description of CI workflows.

Constraints
- The planning agent will not modify the repository. Implementation steps assume a normal git checkout of the given repo URL.
- Avoid running privileged or networked build steps in the analysis stage; prefer to document commands to run locally or in CI.

Owners and timeline
- Single analyst can complete this in 2–4 hours depending on repository size.
