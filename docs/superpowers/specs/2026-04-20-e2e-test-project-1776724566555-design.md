Design spec: Codebase Structure Analysis for E2E Test Repo (be3bb149-3d93-4669-b161-b04b0624a88b)

Goal
- Produce a thorough, human-readable summary and machine-readable report describing the repository's structure, technologies, key entry points, build/test/CI tooling, and potential areas that need follow-up (missing docs, unclear layout, large binary blobs).

Approach
- Non-invasive repository scan that identifies: top-level directories and files, language(s) used, package and dependency manifests, build scripts, test suites, CI/CD configuration, container artifacts (Dockerfiles), license and contributing files, and any monorepo layout (packages/, services/, apps/).
- Collect artifacts into two outputs committed to the repository: docs/codebase-structure.md (narrative summary) and analysis/codebase-structure.json (structured metadata). Also include a short checklist of recommended next steps.
- Use common detection heuristics (file names and extensions) plus tooling where available (cloc for language breakdown, git ls-tree or tree for file layout, and parsing of common manifest files). For CI detection look in .github/workflows/, .gitlab-ci.yml, circleci/ etc.

Key decisions
- Non-destructive read-only analysis: Plan will instruct sub-agents to clone and inspect the repository but not to modify source files except to add the generated analysis docs in a new branch/PR.
- Broad compatibility: The plan assumes repo could contain any popular language (JavaScript/TypeScript, Python, Go, Java, Rust, C#, etc.) and will look for standard manifest files (package.json, pyproject.toml, requirements.txt, go.mod, Cargo.toml, pom.xml, build.gradle, etc.).
- Outputs: Produce both a human-friendly Markdown report and a JSON metadata file so downstream automation can consume results.

Acceptance criteria
- docs/codebase-structure.md committed in a branch with a PR opened, summarizing: top-level layout, language breakdown, key manifests and entry points, test frameworks, CI files found, Docker/container details, and any notable files (large assets, generated folders, vendor, node_modules, dist).
- analysis/codebase-structure.json produced with structured fields (repo id, scan timestamp, top-level directories, file counts, languages, manifests found, CI configs found, dockerfiles found, tests found, recommended next steps).
- A short checklist of recommended actions (add README if missing, add license, enable CI, split monorepo, reduce large binaries, etc.).

Constraints
- Do not assume any particular tool availability; plan includes fallbacks (basic shell commands) and optional tools (cloc, github-linguist) if present.
- The planning agent will not itself run any repository operations; the generated plan is for implementers/sub-agents to follow.
