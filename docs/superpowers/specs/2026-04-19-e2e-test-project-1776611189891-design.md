Design spec: Codebase structure analysis for repository multi-agent-harness-test-repo

Goal
- Produce a concise, actionable analysis of the repository's codebase structure: top-level layout, languages and size, dependency manifests, build/test/CI surfaces, notable scripts/tools, major packages/modules, and gaps (missing CI, tests, docs).

Approach
- Do a read-only, non-destructive analysis using repository metadata and file inspection. The analysis will be automated where possible (directory listing, language line counts, manifest parsing) and supplemented by targeted manual inspections (README, CI workflows, top-level packages).
- Produce two deliverables: a human-readable report (SUMMARY.md) and machine-readable outputs (analysis.json, dependency-list.txt, file-tree.txt). Also include recommended next steps and low-effort fixes.

Key decisions
- Only read repository files; do not modify code or run tests. All commands are exploratory (ls, rg, cloc, tokei/cloc fallback, parsing with jq/python).
- Focus on artifacts that convey structure and maintenance effort: language breakdown, dependency manifests, CI workflows (.github/workflows), Dockerfiles, top-level scripts (scripts/, Makefile, package.json), tests (tests/, spec/, __tests__), and README.
- Produce a prioritized list of recommendations (high, medium, low effort) for improving discoverability and developer experience.

Deliverables
- SUMMARY.md: 1–2 page human-readable summary with top-level tree, language breakdown, CI status, test coverage indicators, important files, and recommendations.
- analysis.json: structured machine-readable data (directories, files, language stats, manifests found, CI files, dockerfiles, test paths).
- file-tree.txt: recursive tree of repository up to configurable depth.
- dependency-list.txt: flattened list of direct dependency files and key dependencies extracted from manifests.

Success criteria
- The reports allow a developer to quickly understand where to look for main application code, how to build and test, and where to start making changes.
- The machine-readable outputs enable follow-up automation (dependency scanning, license checks, test-run tasks).

Notes/Constraints
- This spec assumes repository contents are available to the analysis agent. The plan that follows lists exact commands and file paths to inspect; the agent executing the plan must run these inside the repository.
