# Design Spec: Codebase Structure Analysis

Goal
- Produce a clear, actionable analysis of the codebase structure for the E2E Test Repo (https://github.com/dreef3/multi-agent-harness-test-repo.git, repo id: 4ba2fc47-8587-4fd9-bac1-c445c1666f66). The deliverable will be a written report that summarizes the repository layout, primary languages and frameworks, test structure, CI configuration, build and run entrypoints, dependency surfaces, and recommendations for improvements.

Approach
- Since I am not to clone or explore the repository directly, this spec defines a repeatable, automated investigation workflow that a sub-agent or developer can run against the repository. The workflow consists of lightweight repository inspection steps (file tree, language detection, manifest and config file inspection, CI and workflow analysis, test discovery, dependency listing), deeper checks (static analysis, linter and test runs, dependency audit), and a final synthesis step to produce the analysis report.
- The plan focuses on non-destructive commands and deterministic file reads. All tasks will produce machine-readable intermediate outputs (JSON or plain text) and a final human-readable Markdown report.

Key decisions
- Non-destructive inspection: prefer read-only commands (ls, rg/grep, find, cat/README parsing, package manager commands that print but do not modify state). Where package-manager operations may change state (e.g., installing dependencies), make them optional and gated behind a flag.
- Language-agnostic baseline: look for common manifest files (package.json, pyproject.toml/requirements.txt, go.mod, Gemfile, Cargo.toml, .NET csproj, etc.) so the plan works across polyglot repos.
- CI-first: inspect .github/workflows, GitLab CI, CircleCI, and Docker-related files early to identify build and test entrypoints.
- Report structure: include tree snapshot, language breakdown, test coverage of repository layout, CI matrix summary, dependency manifest list, and prioritized recommendations.

Constraints
- This spec does not perform the analysis itself; it prescribes the investigation steps. The investigation must be executed by a sub-agent or developer with repository access.

Success criteria
- A Markdown report at repo-root/analysis/codebase-structure-report.md containing:
  - file tree snapshot (top 3 levels) and notable files
  - detected languages and manifest files
  - CI/workflow summary
  - test and coverage setup summary
  - build/run instructions discovered
  - dependency summary and notable vulnerabilities or unusual patterns (if audited)
  - recommended next steps and quick wins
