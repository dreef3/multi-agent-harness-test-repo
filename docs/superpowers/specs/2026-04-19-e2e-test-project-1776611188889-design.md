Design Spec: Codebase Structure Analysis for "E2E Test Repo 0"

Goal
- Produce a concise, actionable analysis of the codebase structure for the repository "E2E Test Repo 0" (id: e4bcb16b-6482-4b43-9535-42a6f784b649).
- Deliverables: human-readable summary (README-style), machine-readable outputs (JSON), a prioritized list of follow-up recommendations (tests, CI, security), and exact commands used to derive the results.

Approach
- Perform non-invasive, static inspection of the repository using standard filesystem and text-parsing commands. No code execution (beyond safe metadata commands) will be required.
- Rely on conventional repository indicators: README*, LICENSE, change logs, top-level manifests (package.json, pyproject.toml, go.mod, Cargo.toml, composer.json), language-specific directories (src, lib, pkg, app), test directories (tests, test, __tests__, e2e), CI configs (.github/workflows, .gitlab-ci.yml, .circleci), Dockerfile(s), Makefile, and scripts/.
- Parse each manifest to extract the language, dependency manager, key dependencies, and test commands. Inspect CI workflows and Dockerfiles to infer build/test/deploy pipeline steps.
- Produce both a short summary and structured artifacts (JSON) that tools or other agents can consume.

Key Decisions
- Use heuristic rules based on presence of standard files to determine primary languages and frameworks (e.g., package.json => Node, pyproject/requirements => Python, go.mod => Go, Cargo.toml => Rust).
- When multiple language manifests exist, report all and mark a primary language if one dominates in file counts or top-level project conventions.
- Report discovered test commands but do not run them.
- Flag security-sensitive files (private keys, secret patterns) but do not exfiltrate their content; only report their presence and paths.

Success Criteria
- A short human-friendly report summarizing: languages, top-level structure, test and CI setups, notable files, and any red flags.
- A machine-readable JSON containing a file tree summary, language breakdown, dependency manifests and parsed dependencies, CI/workflow summaries, and recommended next steps.

Constraints & Assumptions
- This spec assumes the repository will be available to the execution environment for filesystem inspection by subsequent implementation tasks.
- No dynamic code execution or running test suites will be performed.
