Design Spec: Analyze repository "E2E Test Repo 1" (id: f6503b47-5c27-4789-bad7-af9154233752)

Goal
- Produce a clear, repeatable analysis of the codebase structure for the repository at https://github.com/dreef3/multi-agent-harness-test-repo.git (default branch: main). The analysis should identify primary languages, package managers, top-level layout, key files (README, LICENSE, CI), build/test targets, dependency manifests, source directories, and any CI/CD, Docker, or reproducibility artifacts.

Approach
- Use non-invasive, read-only repository inspection steps executed in a clean checkout of the repository. The inspection will be automated (shell commands and small parsing utilities) to produce a structured report file (CODEBASE_STRUCTURE.md) in the repository’s planning branch and a short summary for humans.
- Steps will include listing top-level files, walking the tree to a configurable depth, detecting language and tooling files (package.json, pyproject.toml, go.mod, Cargo.toml, Gemfile, Dockerfile, Makefile), reading CI workflow definitions (.github/workflows/*), and collecting test/coverage configuration and scripts.
- No tests or build steps will be executed. The plan focuses on static analysis only (file and content inspection).

Key decisions
- Conservative, read-only approach: only list and read files to avoid side effects.
- Produce both machine-readable outputs (JSON summary) and human-readable report (CODEBASE_STRUCTURE.md) so downstream agents or maintainers can quickly consume results.
- Cover a broad set of common ecosystems (Node, Python, Go, Rust, Java, Ruby, Docker/containers) to maximize detection accuracy.

Deliverables
- CODEBASE_STRUCTURE.md placed in the repository planning branch (and included in the PR) containing: top-level summary, detected languages and package managers, CI and Docker artifacts, source layout, tests and e2e directories, and recommended next steps.
- A JSON summary file (codebase_summary.json) with structured fields for automated consumption: languages, manifests, CI workflows, Dockerfiles, top-level directories, test frameworks, license, README presence.

Risks and mitigations
- Repository may contain large binary files or submodules; the analysis will skip large binary blobs and submodule recursion by default.
- Private repositories or rate-limited API access: assume the harness has repository checkout access (the plan is designed to run inside the harness workspace where the repo is checked out).

Success criteria
- A CODEBASE_STRUCTURE.md that accurately lists top-level layout and key artifacts.
- A codebase_summary.json with structured detection entries that align with the README and manifest contents.

Notes
- This spec does not implement the analysis; it defines the plan to run in the harness after the planning PR is created.
