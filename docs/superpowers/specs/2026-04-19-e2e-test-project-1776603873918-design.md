Design Spec: Codebase Structure Analysis for E2E Test Repo (04a1c36c-59d9-4748-8a45-ab40b8ff7d29)

Goal
- Produce a clear, actionable analysis of the repository's structure, key components, technologies, tests, and CI configuration. The analysis will identify major directories and files, dependency manifests, build/test commands, and potential maintenance or quality concerns.

Approach
- Because I was instructed not to clone or open the repository in this planning phase, this spec defines how the analysis should be performed by an implementation agent (or CI job). The analysis will be automated where possible and also include manual inspection steps.
- The plan will enumerate precise commands to run, exact file paths to read, and expected output snippets to capture. The analysis will produce a short summary, a detailed file/directory inventory, dependency manifests and major dependencies, test and CI status, and suggested next actions.

Key decisions and assumptions
- Assume the repository follows common layout conventions for one or more of these ecosystems: Node.js (package.json), Python (pyproject.toml / setup.py / requirements.txt), Go (go.mod), Rust (Cargo.toml), Java (pom.xml / build.gradle), or mixed polyrepo. Plan will include checks for each common ecosystem but only run ecosystem-specific steps when the relevant manifest is present.
- The analysis will prioritize these files and directories: README.md, LICENSE, package.json, pyproject.toml, setup.py, requirements.txt, go.mod, Cargo.toml, src/ or lib/, tests/ or spec/, .github/workflows/, Dockerfile, Makefile, scripts/, docs/. Also look for CODE_OF_CONDUCT, CONTRIBUTING, and .editorconfig/.eslintrc/.prettierrc.
- The analysis must be reproducible: all commands and parsing steps are explicitly listed.

Success criteria
- A summary (one-paragraph) describing the repository purpose and primary language(s)/frameworks.
- A precise file/directory tree listing (top 2–3 levels) and detection of the main entry points/build files.
- A dependency list (top-level) with versions for the primary manifest(s).
- Test discovery and instructions to run tests, plus reported results (pass/fail/coverage if available).
- CI inspection (list of GitHub Actions or other CI configs and a short description).
- A brief list of improvement suggestions and any blockers.

Deliverables
- summary.txt — short paragraph summary
- tree.txt — trimmed tree output (top 3 levels)
- manifests/* — copies or parsed JSON of manifests found (package.json, pyproject.toml, go.mod, Cargo.toml, etc.)
- tests/report.txt — test run output and commands used
- ci/report.txt — CI workflow list and short descriptions
- final-report.md — consolidated human-readable analysis

Privacy and safety
- The analysis will not modify the repository. It only reads files and runs read-only inspection commands.

"Done" criteria
- final-report.md present and contains all Success criteria entries.
