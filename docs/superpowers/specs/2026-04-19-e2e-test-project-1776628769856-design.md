Design Spec: Analyze codebase structure for the E2E Test Repo (d8a593e2-8cdc-40c5-9a6a-e50f1d497095)

Goal
- Produce a concise, actionable analysis of the repository located at https://github.com/dreef3/multi-agent-harness-test-repo.git that answers: primary languages, top-level layout, build/runtime/dependency manifests, CI and test setup, Docker/containers, notable scripts and entry points, and potential technical risks and quick wins.

Approach
- Non-invasive, read-only analysis performed by automated steps and manual inspection. The analysis will produce two outputs: (A) a machine-readable inventory (JSON) listing languages, files, and key manifests, and (B) a human-readable Markdown report summarizing findings and recommendations.
- Steps to generate both outputs:
  1. Clone the repository (shallow) to a working directory.
  2. Build a directory tree snapshot and identify top-level files and directories.
  3. Detect languages and counts using cloc (or tokei) and basic file-glob heuristics.
  4. Locate and parse common manifest files (package.json, pyproject.toml, requirements.txt, go.mod, Cargo.toml, pom.xml, build.gradle, etc.) and lockfiles (package-lock.json, yarn.lock, Pipfile.lock, poetry.lock). Extract declared dependencies, scripts, and engines.
  5. Inspect CI configuration (.github/workflows, .gitlab-ci.yml, circleci, etc.) to extract build/test steps and matrix.
  6. Search for Dockerfile(s), docker-compose.yml, and infra-deployment manifests.
  7. Locate tests, test runner config, and measure test counts (by files and basic heuristics) and presence of coverage tools.
  8. Identify common tooling scripts (Makefile, scripts/ directory, entrypoints in package.json, setup.py console_scripts, etc.).
  9. Run quick static checks: lint config presence, formatter config, and license file.
  10. Produce reports with findings, risk markers (outdated deps, missing CI, monorepo indicators, unclear entrypoint), and prioritized recommendations.

Key decisions and constraints
- Read-only analysis: do not modify the repository.
- Use well-known CLI tools (git, find, rg, cloc, jq, python) that are available in most CI environments. If a tool is missing, fall back to portable scripts (find + wc + file extension heuristics).
- Provide both machine-friendly and human-friendly outputs so follow-up automation or manual reviewers can consume the results.
- Deliverables: inventory.json, report.md, and a short summary (1-2 paragraphs) suitable for a PR description or issue.

Success criteria
- Inventory JSON lists top-level tree, detected languages and line counts, manifests found (with parsed metadata), CI files and parsed steps, Docker artifacts, test locations and counts, and list of recommended next actions.
- Markdown report clearly summarizes the above and gives prioritized recommendations and estimated effort for the top 3 fixes.
