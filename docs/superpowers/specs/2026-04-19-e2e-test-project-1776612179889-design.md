Design spec: Analyze codebase structure for repository "E2E Test Repo 0" (id: 76e60581-1b04-49e5-bc09-13fc1138917b)

Goal
- Produce a clear, reproducible analysis of the repository structure: top-level layout, languages, build systems, dependency manifests, CI configurations, test locations, documentation, and notable files (README, LICENSE, CONTRIBUTING, Dockerfiles, etc.).

Approach
- Because I do not have the repository contents locally in this planning phase, the analysis will be executed by automated implementation tasks (sub-agents) that will clone the repository and collect structured information. The spec documents what to collect and how. Outputs will be machine- and human-readable artifacts (MD report, JSON metadata, raw directory tree) saved into the repo under a dedicated analysis/ directory and proposed as a PR.

Key decisions
- Keep the analysis non-destructive: tasks will only read files, run static inspection commands, and produce reports under analysis/.
- Use simple, widely-available CLI tools to collect data: git, ls/tree, rg (ripgrep) or grep, file, wc, and language-specific manifest readers (npm/yarn/pnpm, pip/poetry, go, cargo). Where a manifest is present, capture the manifest content and a parsed dependency list (JSON).
- Standardize outputs: analysis/tree.txt (raw tree), analysis/structure.md (human-readable summary), analysis/metadata.json (machine-readable summary), analysis/deps/* (dependency manifests and parsed JSON), analysis/ci/* (CI workflow copies and summary).
- Produce a small plan of follow-up tasks (lint, test, dependency scan) if requested.

Acceptance criteria
- A PR with analysis/ containing: tree.txt, structure.md, metadata.json, and copied important files (README, LICENSE, package manifests, CI workflows).
- structure.md should list languages, frameworks, build tools, test locations, CI status, and suggested next steps.

Constraints
- This spec assumes the repository is accessible over HTTPS and contains at least one recognizable language manifest (package.json, pyproject.toml, go.mod, Cargo.toml, etc.). If none are found, the analysis will still produce a directory tree and a best-effort file inventory.

Estimated effort
- Automated run: ~5–10 minutes to clone and collect artifacts (depending on repo size). Human review: 15–30 minutes.

Next step
- Run the implementation plan to collect the repository data and create the analysis PR.
