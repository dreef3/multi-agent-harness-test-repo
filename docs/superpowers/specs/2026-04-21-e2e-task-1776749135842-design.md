# Design Spec: Add simple repository marker file

Goal

Create a simple, human-readable marker file at the repository root to indicate that the repository has been processed by the planning agent. The marker should be minimal, easy to spot in the repo root, and contain enough metadata to be useful (who added it, when, and which planning run).

Approach

- Add a new file named `MARKER.txt` at the repository root. This filename was chosen for visibility and low risk (no hidden-file semantics).
- The file will contain a small set of metadata lines:
  - A human-friendly title/description
  - Created-by (planning agent)
  - Timestamp (ISO-8601)
  - Repository identifier (the provided repo name and UUID)
- Create a short-lived feature branch (suggested name `add/marker-file`), commit the new file, and open a PR against the repository's default branch (`main`).
- If the file already exists, the implementer should NOT blindly overwrite it; instead either skip creating the file or update it by appending a new "Updated-by" line. The implementation plan includes behavior guidance.

Key decisions

- File name: `MARKER.txt` (visible at repo root, does not interfere with dotfile rules).
- Location: repository root for discoverability.
- Minimal content: small set of metadata lines for traceability.
- Branch and PR: use `add/marker-file` and PR title `Add repository marker file`.
- Non-destructive: prefer not overwriting an existing marker file; document update behavior.

Constraints

- Do not modify other files in the repository.
- Keep the change isolated to adding (or safely updating) a single file.

Owners

- Planning agent: produces this spec and implementation plan.
- Implementation task: a sub-agent or human contributor will create the branch, add the file, and open the PR.
