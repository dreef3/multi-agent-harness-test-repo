# Design Spec: Codebase Structure Analysis

Goal
- Produce a concise, actionable analysis of the repository "multi-agent-harness-test-repo" (id: c77a5bf7-3b2b-40fd-8ba3-ee07c43e35b1). Deliverables: a written report describing the codebase structure, detected languages, package managers, CI, test layout, build artifacts, dependency highlights, and recommended next steps for onboarding, testing, and maintenance.

Approach
- Non-invasive static inspection only (no network cloning, no changes). Use repository-local reads and standard command-line tools to enumerate files and read key manifest files.
- Focus on high-value artifacts: README(s), top-level manifest files (package.json, pyproject.toml, go.mod, Cargo.toml, setup.py), monorepo manifests (pnpm-workspace.yaml, lerna.json, turbo.json), CI configs (.github/workflows/*), Dockerfile(s), tests folders (tests/, __tests__, spec/), src/ or lib/ directories, and tooling config (.eslintrc, .prettierrc, Makefile, tox.ini).
- Produce a single report file docs/codebase-structure.md summarizing findings with: repository tree snapshot, language and framework breakdown, key files reviewed with short summaries, test & CI status (what exists, not run), technical debt & risks, and recommended follow-ups.

Key Decisions
- Do not attempt to run tests or builds automatically. Instead include explicit commands and expected outputs to be run by implementers or CI.
- Keep the initial analysis high-level and reproducible: commands are included so anyone can re-run and verify.
- Save the spec now and then save a detailed implementation plan that lays out the exact inspection tasks, file paths to check, commands to run, and the format of the final report.

Success Criteria
- A saved planning PR containing the design spec and the implementation plan.
- A follow-up implementation plan that, when executed, produces docs/codebase-structure.md containing the requested analysis.

Notes
- This plan assumes repository access by the implementer; the planning documents include exact commands and file paths to inspect so the implementer can run them without further guidance.
