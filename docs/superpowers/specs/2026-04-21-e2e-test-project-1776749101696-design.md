Design spec: Analyze codebase structure for E2E Test Repo 1 (id: 45bc0ff1-05dc-4210-904a-f2c288471d83, https://github.com/dreef3/multi-agent-harness-test-repo.git, default branch: main)

Goal
- Produce a clear, actionable static analysis of the repository's structure: top-level layout, language(s), package managers, build/test tooling, CI configuration, Docker/containers, tests, docs, license, dependency manifest(s), and potential areas of attention (missing tests, large artifacts, binary blobs, licensing gaps).

Approach
- Because the planning agent is not allowed to clone or inspect the repository directly, this spec defines an automated, repeatable static analysis plan that will be executed by implementation sub-agents. The plan will: clone the repository on a worker, enumerate files and directories, detect languages and code metrics, locate configuration and CI files, parse dependency manifests, generate an SBOM, surface large files and binaries, and produce human-readable summaries and machine-readable artifacts (JSON). Outputs will be stored under a reports/ directory alongside the repository clone.

Key decisions and constraints
- Non-invasive static analysis only: do not run untrusted code or tests, do not build Docker images, do not execute binaries. All operations are read-only on the checked-out files except creating the reports directory.
- Use standard CLI tools available on CI/agents: git, tree, rg/grep, cloc/tokei (or fallback to wc/ls if not present), jq, du, file, identify Dockerfiles and GitHub Actions workflow files.
- Produce two primary artifacts: reports/structure.md (human-friendly narrative) and reports/structure.json (structured metadata). Also produce reports/dependencies.(json|txt), reports/sbom.json if SBOM tool available (e.g., syft).
- Repository-specific outputs will be named using the repository id to avoid collisions.
- The plan is modular: each task is self-contained and can be run independently or recomposed in different worker environments.

Success criteria
- A complete reports/ directory containing at minimum: structure.md (overview), structure.json (machine-readable summary), top_files.txt (top-level files), biggest_files.txt (top 50 by size), languages.txt (language breakdown by LOC), detected_CI.txt (list of CI/config files), and dependency_manifests.txt (paths and types).
- No code executed; all analysis is file-system and manifest parsing only.

Deliverables
- reports/{repo-id}/structure.md
- reports/{repo-id}/structure.json
- reports/{repo-id}/dependencies.*
- A short implementation plan that the harness can dispatch to worker agents to perform the actual analysis.

Notes for implementers
- The default branch is main; the worker should clone the repository with --depth=1 and checkout main. If main is absent, fall back to default branch reported by remote refs.
- If required tools (cloc, tokei, syft) are not present, produce best-effort outputs with available commands and record which tools were missing in reports/tooling.txt.
