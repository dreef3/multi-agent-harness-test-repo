# Design Spec: Codebase Structure Analysis for E2E Test Repo 0

Goal
- Produce a concise, actionable analysis of the repository `https://github.com/dreef3/multi-agent-harness-test-repo.git` (id: e93d4511-5b8d-42f4-b6e9-8cdf11e053a4). The output will summarize the project layout, primary languages, dependency managers, CI configuration, tests, dockerization, important files (README, LICENSE), and recommended next steps.

Approach
- Perform a static, read-only inspection of the repository filesystem (no builds or tests). Collect metadata and generate two deliverables:
  1. Human-readable Markdown report: docs/codebase-structure.md
  2. Machine-readable JSON metadata: reports/codebase-structure.json
- The inspection will be automated via shell commands and small parsing steps to determine languages (by file extensions and byte-count), package manager files, presence of CI workflows, Dockerfiles, and test directories.

Key decisions
- Non-invasive: do not run code, tests, or install dependencies.
- Use file-extension heuristics plus presence of language-specific manifest files (package.json, pyproject.toml, go.mod, Cargo.toml, Gemfile, etc.) to identify languages and package managers.
- Search for CI in .github/workflows, .gitlab-ci.yml, circleci/, etc.
- Detect Dockerization by searching for Dockerfile or docker-compose.yml.
- Inspect top-level directories (src, lib, cmd, pkg, test, tests, scripts, .github).
- Report will include: overall size, file counts, dominant languages, dependency files, CI, Docker, license, test presence, recommended next steps.

Outputs
- docs/codebase-structure.md — narrative report and checklist.
- reports/codebase-structure.json — structured metadata including counts and file lists.

Constraints
- The planning agent will not clone or inspect the repository itself. The plan will describe exact commands and file paths for a follow-up implementation agent to run.

Success criteria
- The plan enables any implementer to run the commands and produce the two deliverables.
- The report highlights potential maintenance, onboarding, or security concerns (missing license, no CI, no tests, multiple languages, etc.).
