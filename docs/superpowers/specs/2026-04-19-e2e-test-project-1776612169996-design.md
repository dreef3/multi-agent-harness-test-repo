Design Spec: Codebase Structure Analysis for "E2E Test Repo 1"

Goal
- Produce a concise, actionable analysis of the codebase structure for the repository:
  https://github.com/dreef3/multi-agent-harness-test-repo.git (id: 52d27700-93be-453e-8549-98c57498a9ca, default branch: main).
- Deliverables: a summary report describing languages, package managers, top-level layout, CI, tests, docs, and suggested next steps/risk areas.

Approach
- Since I must not clone or inspect the repository directly in this planning phase, this spec defines the analysis procedure that will be executed by downstream implementation tasks (or by an agent with repository access).
- The analysis will use a deterministic sequence of repository inspection steps (commands and file reads) to produce the report. Steps include listing top-level files, detecting language and tooling (package.json, pyproject.toml, go.mod, Cargo.toml, Gemfile, etc.), examining CI config (.github/workflows, .gitlab-ci.yml), reading README and CONTRIBUTING, listing tests and their frameworks, inspecting Dockerfiles and infra, and summarizing dependencies and version constraints.

Key decisions
- Focus first on automation and testability: identify CI, test suites, and test failures risk areas.
- Prioritize files that reliably indicate project type (README, package files, build files, .github/workflows/*, Dockerfile, src/, tests/, docs/).
- Produce a short human-readable summary and a longer, machine-readable checklist of findings.

Outputs
1) Short summary: top-level languages, primary package manager, whether CI exists, presence of tests, Docker, and notable missing docs.
2) Detailed findings: file tree snapshot, key files with short excerpts (e.g., package.json scripts, pyproject metadata), list of test frameworks and commands, CI workflows names and triggers.
3) Actionable recommendations: missing tests, suggested CI improvements, dependency update suggestions, security concerns (e.g., pinned versions), and next investigation steps.

Constraints and assumptions
- This plan assumes repository access will be provided to an executor agent. No repo exploration happens in this planning message.
- The analysis will avoid making changes to the repository; it is read-only.

Success criteria
- The report identifies languages, build system, test framework, CI presence, and lists the top 25 files by relevance.
- Recommendations are practical and low-effort (e.g., add a test workflow if none exists), and tasks are actionable for developers.
