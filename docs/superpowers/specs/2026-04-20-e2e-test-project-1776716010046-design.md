Design Spec: Analyze the codebase structure for the E2E Test Repo

Goal
- Produce an accurate, actionable analysis of the codebase structure for the repository at https://github.com/dreef3/multi-agent-harness-test-repo.git (repo id: 6c093857-2fd1-4758-8cbc-1a563b139721, default branch: main).
- Deliver a human-readable report and a machine-readable manifest describing top-level layout, languages, package/dependency files, CI/workflows, build tooling, test locations, documentation, and notable risks or maintenance concerns.

Approach
- Because this planning agent must not clone or explore the repository, the plan defines a set of deterministic inspection tasks for worker/sub-agent(s) to execute against the repository. Those tasks will clone the repo into a temporary workspace, enumerate files, detect languages and package managers, inspect common project files (README, LICENSE, package manifests, build files, CI config, Dockerfiles), and run minimal non-destructive commands (list, parse, static metadata extraction). The worker will not run tests or untrusted binaries by default; any dynamic execution (build/test) is optional and gated behind explicit acceptance.
- Produce two outputs: (1) analysis/structure-report.md — narrative summary and findings; (2) analysis/structure-manifest.json — structured JSON manifest with entries for top-level files, directories, languages, package managers, CI workflows, Dockerfiles, test frameworks, size metrics, and a short list of recommended next steps.

Key Decisions
- Non-destructive inspection only: tasks will avoid executing project binaries or test suites unless explicitly allowed. We will prefer static analysis of files (manifest parsing, file headers, CI config) and lightweight tools (cloc, git ls-files, tree) for enumeration.
- Use standard heuristics for language detection: presence of manifest files (package.json, pyproject.toml, go.mod, Cargo.toml, Gemfile), common file extensions, and cloc output.
- Target outputs and exact file paths will be created inside the repo under analysis/ so reviewers can find them easily (analysis/structure-report.md and analysis/structure-manifest.json).
- Provide exact commands that implementation agents will run, so tasks are deterministic and reproducible.

Acceptance criteria
- A narrative report (analysis/structure-report.md) that lists: top-level files/dirs, detected languages, package managers, CI/workflow files, Dockerfiles, test directories, docs, license, and a short risk/maintenance summary.
- A JSON manifest with structured fields for automated consumption.
- All steps, commands, and files referenced in the plan must be self-contained; tasks should produce the outputs and exit non-zero only if a fatal error occurs.

Risks and mitigations
- Private or large repositories: clone with --depth=1 to reduce footprint. If repo is private or inaccessible, tasks should record failure and stop.
- Malicious code: do not run project-supplied scripts or binaries. If deeper dynamic analysis is desired, have a separate, explicitly authorized task.

Next step
- Dispatch implementation tasks (detailed in the implementation plan) to clone and analyze the repository automatically, produce the two output files, and open an analysis PR referencing this planning document.
