Design Spec: Codebase Structure Analysis for E2E Test Repo (dreef3/multi-agent-harness-test-repo)

Goal
- Produce a clear, actionable analysis of the repository's structure and contents for the E2E Test Repo (https://github.com/dreef3/multi-agent-harness-test-repo.git).
- The analysis should identify languages, package managers, top-level directories and purpose, key entry points, CI and test configuration, dependency manifests, scripts, and any notable issues (missing README, monorepo layout, nested projects, large files, or vendor directories).

Approach
- Use a scripted, repeatable inspection checklist that records file-system layout and extracts metadata from common manifest files. The plan will avoid running tests or installing dependencies; it will only read files.
- Produce two artifacts: a human-readable Markdown report (summary + detailed tree) and a machine-readable JSON inventory (languages, package files, CI files, test frameworks, top-level directories, gitignore patterns, size estimates).
- The inspection steps will be executed by an implementation agent (not in this planning step). Steps include cloning the repo, listing files, reading key manifests (package.json, pyproject.toml, go.mod, Cargo.toml, etc.), scanning for CI config (.github/workflows, .gitlab-ci.yml), scanning for Dockerfiles, and searching for test files and README.

Key decisions and assumptions
- No code will be executed, built, or installed; only static file inspection will be performed to ensure safety and reproducibility.
- The analysis will target common language ecosystems (JavaScript/TypeScript, Python, Go, Rust, Java, .NET) and will fall back to a generic file inventory for other files.
- Output format: markdown summary at reports/<repo>-structure.md and JSON inventory at reports/<repo>-structure.json in the workspace root.
- If the repo is a monorepo (multiple package roots), the plan will attempt to discover workspace roots (e.g., package.json workspaces, pnpm-workspace.yaml, lerna.json, nx.json) and include a per-package summary.

Success criteria
- A Markdown report that provides a concise top-level summary (languages, primary package manager, CI, test presence), a detailed file tree (top 2-3 levels), and notable findings/recommendations.
- A JSON manifest listing the discovered files and metadata suitable for downstream automation.

Constraints
- Do not modify repository files.
- Do not run unit/integration tests or install dependencies.
- The plan must be executable by an implementation agent with network access to clone the repo.

Deliverables
- reports/multi-agent-harness-test-repo-structure.md
- reports/multi-agent-harness-test-repo-structure.json
- (Optional) attachments with raw outputs: file-list.txt, manifest-excerpts/.

Owners
- Planning agent: this document (saved to planning branch via write_planning_document).
- Implementation agent(s): execute the steps below and produce the deliverables.

Estimated effort
- Automated run: ~5–10 minutes (cloning + static scanning).
- Manual review / write-up: ~15–30 minutes.
