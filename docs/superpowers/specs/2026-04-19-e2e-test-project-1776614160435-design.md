Design Spec: Codebase Structure Analysis for multi-agent-harness-test-repo

Goal
- Produce a focused analysis of the repository "multi-agent-harness-test-repo" (E2E Test Repo id: eb462090-58cf-4e8e-8a34-8c16ce308297) to identify the codebase layout, critical entry points, test suites, CI, and areas requiring follow-up.

Approach
- Without cloning or reading files (per instructions), propose a repeatable checklist and heuristics that an implementation agent should follow to enumerate the repository structure and surface findings. The checklist includes top-level file inspection, language/runtime detection, package metadata, source and test directories, CI/workflows, containerization, scripts, docs, and example/e2e tests.
- Identify likely important files and directories to inspect (README.md, package.json, pyproject.toml, go.mod, Cargo.toml, src/, lib/, tests/, e2e/, .github/workflows/, Dockerfile, scripts/, tools/, docs/). For each item, state why it’s important and what to look for.
- Define concrete tasks for an implementation agent to run (shell commands and file reads), and the expected outputs/observations that will form the analysis.

Key decisions / assumptions
- We will not access the repository now; instead we create a deterministic plan based on common multi-agent harness projects and the repository metadata provided.
- The plan will produce a clear, actionable list of file paths and commands for a sub-agent to execute to gather the actual structure, along with acceptance criteria for each task.
- The deliverable produced by the implementation tasks will be a short report describing the actual structure and any issues (missing metadata, failing tests, unclear entry points, broken CI).

Success criteria
- A delivered repository structure report that lists: top-level files and directories, detected language(s) and package manager(s), test suites and how to run them, CI workflows and what they trigger, Docker/Container files and how to build, and a prioritized list of issues or follow-ups.

Risks and mitigations
- Risk: repository uses uncommon layout or multiple languages. Mitigation: the plan instructs to run language-detection heuristics (file globs, package manager files) and to include guidance for multi-language projects.
- Risk: large repository. Mitigation: use targeted finds (depth-limited) and inspect only key files first.

Next step
- Implementation phase (detailed plan) will contain task-by-task instructions, exact file paths to read, shell commands to run, and the expected outputs to gather the repository structure and produce the final report.
