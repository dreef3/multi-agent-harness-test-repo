Design spec: Analyze codebase structure for E2E Test Repo 1 (github.com/dreef3/multi-agent-harness-test-repo, default branch: main)

Goal
- Produce a clear, machine-readable and human-friendly analysis of the repository structure and contents. The analysis should identify primary languages, top-level files and directories, build / packaging metadata, CI configuration, test suites, dependency manifests, and any notable infra (Dockerfile, Makefile, GitHub Actions) and licensing.

Approach
- Because the planning agent is not allowed to clone or inspect the repository directly, the spec defines an automated, reproducible set of inspection tasks to be executed by sub-agents (or by a human) against the repository. Each task is self-contained, runnable from the repository root on the main branch, and will write its results to a small set of output files in a results/ directory inside the repository.
- The inspection tasks are minimal and non-destructive: they list files, read key manifests (README, package.json, pyproject.toml, setup.py, go.mod, cargo.toml, etc.), detect CI files (.github/workflows, .gitlab-ci.yml), find tests, and summarize language and dependency information.
- Outputs will include:
  - results/tree.txt: top-level tree (find output)
  - results/language-summary.md: languages and file counts
  - results/manifests/: copies of manifest files (package.json, pyproject.toml, go.mod, Cargo.toml, setup.py) when present
  - results/ci.md: CI workflows found and brief summary
  - results/tests.md: test files and suggested commands to run tests
  - results/summary.md: high-level human summary with recommended next steps

Key decisions & trade-offs
- Non-invasive read-only approach is chosen to avoid execution risk and to ensure safety in CI contexts.
- Use simple shell commands (find, wc, sed, jq when available) to keep tasks portable across Linux runners.
- Extracting dependency graphs will be limited to parsing manifest files and not running package managers. That avoids network calls and build steps.
- Tests will be enumerated but not executed by default. If the stakeholder wants tests run, that will be a separate, explicitly authorized task.

Acceptance criteria
- The analysis tasks produce the files listed above and a results/summary.md that answers: primary language(s), build system, whether CI is configured, where tests live and how to run them, and any potential blockers for local development.
- Tasks are runnable by sub-agents given repository id: 4f966f8d-2e5c-4f97-b396-ef7fe727452f and branch main.
