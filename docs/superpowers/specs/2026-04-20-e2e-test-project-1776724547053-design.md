Design spec: Codebase Structure Analysis for E2E Test Repo 1

Goal

- Provide a concise, actionable analysis of the codebase structure for the repository referenced: "E2E Test Repo 1" (id: 01f654cd-db55-449e-8d8e-56f0298a87eb). The output will identify top-level layout, languages, dependency manifests, test locations, CI, tooling, likely runtimes, and any immediate risks or gaps (missing README, tests, or CI). Produce a clear artifact that can be reviewed and used to drive deeper automated checks and remediation tasks.

Approach

- Because I must not clone or explore the repository in this planning phase, this design assumes an initial read-only analysis will be performed by implementation tasks that do clone and inspect the repo. The design focuses on a reproducible, automated analysis pipeline that a sub-agent (or CI job) can run on the repository to generate a report.

- The analysis pipeline will:
  1. Detect repository layout: top-level files and directories (README, LICENSE, src/ or app/, tests/, examples/, docs/).
  2. Detect languages and locate primary source dirs (package manifests like package.json, pyproject.toml, requirements.txt, Gemfile, go.mod, build.gradle, Cargo.toml, etc.).
  3. Collect dependency manifests and produce a dependency tree summary.
  4. Discover test suites and run unit tests (if configured/allowed) to determine coverage and status.
  5. Inspect CI/CD configuration (.github/workflows/, .gitlab-ci.yml, .circleci/) and Dockerfiles.
  6. Run static analysis (linters) and basic security checks (Snyk/OWASP dependency-check or pip-audit/npm audit) where feasible.
  7. Create a final structured report (markdown) that lists findings, recommended remediations, and risk level.

Key decisions

- Non-invasive default: the pipeline will first perform read-only operations (list files, read manifests). Running tests or audits will be optional and gated behind a flag in the plan so it can be enabled by the operator.

- Tooling: prefer standard, widely available CLIs: git, tree, rg (ripgrep), cloc or github-linguist, npm/pip/cargo commands for dependency inspection, pytest/npm test for tests. Use 'cloc' for language stats if available; fallback to simple heuristics otherwise.

- Deliverables: (1) machine- and human-readable report in repo root (CODEBASE_ANALYSIS.md), (2) JSON summary for programmatic use (codebase_summary.json), (3) optional artifacts: dependency graphs (DOT/SVG), test results, and linter output.

- Security & permissions: do not run arbitrary code from the repository in an untrusted environment. Tests and builds are optional and should be run in a sandboxed environment or after operator consent.

Success criteria

- The pipeline produces CODEBASE_ANALYSIS.md that clearly documents the repository structure and actionable next steps.
- The JSON summary lists files discovered, languages with line counts, dependency manifests found, test locations, CI config files, and a short risk assessment.

Repository context

- Target repository: E2E Test Repo 1 (id: 01f654cd-db55-449e-8d8e-56f0298a87eb) — default branch: main.

- The plan will include exact commands and file paths that a sub-agent should run to perform the analysis and produce artifacts.
