Design Spec: Analyze codebase structure for repository "E2E Test Repo 1" (id: 65e3f553-8dce-4693-90a0-e11ebf637ddf, URL: https://github.com/dreef3/multi-agent-harness-test-repo.git)

Goal
- Produce a clear, actionable analysis of the repository's structure and contents: top-level layout, primary languages, dependency manifests, CI and tooling, tests, entry points, and items that need attention.

Approach
- Because I cannot clone or inspect the repository from this planning agent, this spec outlines the approach and the exact commands, file paths, and artifacts that a worker/sub-agent should produce when performing the analysis.
- The analysis will be performed by running a deterministic sequence of filesystem and content-inspection commands after cloning the repo. The commands will collect a tree view, identify language and dependency manifests, enumerate CI/workflow files, list Docker/packaging/config files, extract README and other doc summaries, and summarize test suites and scripts.
- The output will be a structured report (Markdown) containing: a short summary, a hierarchical file tree (top 3 levels), language and dependency breakdown, CI and build system description, test coverage and runners, notable TODOs or failing tests, and recommended next steps.

Key decisions and assumptions
- The plan assumes a standard repository layout (README, LICENSE, src/ or pkg/ or app/, tests/, .github/workflows/, Dockerfile, package manifests). If the repo uses an uncommon layout, the analysis tasks will still discover relevant files via pattern matching (find, rg).
- The analysis will not run the project's test suite by default (optional task) unless requested, to avoid side effects. If requested, the plan includes instructions and commands to run tests and capture results.
- The sub-agent performing the analysis will produce raw command outputs (tree, find, cloc, package manager outputs) and distilled textual summaries.

Deliverables
- analysis.md: human-readable Markdown report with findings and recommendations
- tree.txt: textual file-tree of repo (top 3 levels)
- manifests.txt: list and excerpts of dependency manifests (package.json, pyproject.toml, go.mod, etc.)
- ci-list.txt: list of CI/workflow files and short descriptions
- tests-summary.txt: listing of test frameworks and test count where detectable
- optional: test-run-output.txt (if tests executed)

Metrics and success criteria
- Report includes clear identification of primary language(s), dependency manifests, CI presence, main entry points, and at least 5 actionable recommendations (e.g., add CI, fix lints, add docs).
- All commands used and their outputs are included in an artifacts directory for traceability.

Constraints
- This spec is a planning artifact only. It does not execute any commands or modify the repository. Implementation tasks will be written next and will detail the exact commands and file paths.
