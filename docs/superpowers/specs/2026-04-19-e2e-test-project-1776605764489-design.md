Design Spec: Codebase Structure Analysis for "E2E Test Repo" (dreef3/multi-agent-harness-test-repo)

Goal
- Produce a clear, reproducible analysis of the repository structure and contents for the E2E Test Repo (https://github.com/dreef3/multi-agent-harness-test-repo.git).
- Deliver a human-readable summary (Markdown) and machine-readable artifacts (JSON), plus optional visualizations (module graph) that capture languages, top-level directories, important files, CI configuration, test suites, dependency manifests, and basic metrics (file counts by language, line counts, test coverage pointers).

Approach
- Because this planning agent cannot clone or read the repository directly, the plan will specify an automated analysis workflow for an executor (sub-agent or developer) to run locally or in CI. The workflow will (a) fetch the repo, (b) enumerate files and directories, (c) detect languages and dependency manifests, (d) parse CI workflows and important config files, (e) discover tests and test runners, (f) produce summary artifacts and visualizations, and (g) open a PR (or create a report) with findings.
- The analysis will be non-invasive (read-only) and script-driven so it can be rerun. Scripts will be idempotent and output into a dedicated reports/analysis/ folder in the repo.

Key decisions
- Output artifacts: reports/analysis/summary.md (human summary), reports/analysis/summary.json (structured), reports/analysis/module-graph.svg (optional), and reports/analysis/raw-file-list.txt.
- Tools: use standard, broadly-available tools (git, tree/find, ripgrep/rg, cloc for language stats, jq for JSON processing, graphviz for visualization if available). Also include npm/pip/go manifest parsing where relevant.
- Scope: structural analysis only—no code modifications, no test execution by default, though one task may run tests if permitted and environment is available.
- Deliverables will include instructions and commands so the analysis can be reproduced by humans or sub-agents.

Constraints and assumptions
- The plan assumes the executor has network access to clone the repo and has common CLI tools installed (git, bash, find, rg, cloc, jq, graphviz optional). If not, the plan lists fallback commands.
- The analysis will not change repository files by default; writing into reports/analysis/ is optional and should be done in a feature branch if the executor will commit results.

Success criteria
- A concise Markdown summary that clearly lists top-level directories, languages, dependency manifests, CI workflows, test frameworks, and noteworthy files (e.g., README, LICENSE, CONTRIBUTING).
- A machine-readable summary.json with counts and lists for programmatic use.
- Reproducible scripts/commands to regenerate the result.
