Design Spec: Codebase Structure Analysis for E2E Test Repo

Goal
- Produce a clear, actionable analysis of the codebase structure for the repository "E2E Test Repo" (id: 809560c5-950e-4a81-8aa7-b4cc99523414, https://github.com/dreef3/multi-agent-harness-test-repo.git). The analysis should summarize languages, layout, important files (README, CI, Dockerfiles), dependency manifests, test arrangement, and any notable maintenance or security signals.

Approach
- Because this planning agent is not permitted to clone or inspect the repository directly, the plan delegates the concrete repository inspection to implementation tasks that will run repository-local commands and create a structured report inside the repository.
- The inspection will be performed with a short set of deterministic shell commands and language-specific introspection commands. Outputs will be captured into files in a well-known folder (e.g., docs/codebase-analysis.md and artifacts/structure.json) so that reviewers and automated checks can consume the results.
- The analysis will be both human-friendly (Markdown summary) and machine-friendly (structured JSON listing tree, detected languages, dependency files, CI pipelines, Dockerfiles, test frameworks and counts, LOC by language).

Key Decisions
- Focus only on repository static analysis (tree, manifest files, CI, tests, Docker, package managers). No dynamic runtime or security scanning in this phase.
- Capture raw command outputs to reproducible files, and derive a concise summary document (docs/codebase-analysis.md) with findings and recommendations.
- Use common OS tools (ls, find, rg, wc, du) combined with language-specific tools where available (npm, pip, go, mvn, cargo) — tasks will gracefully skip a language-specific step if the manifest is not present.
- Keep tasks self-contained: each task must produce artifacts committed in the repository so the planning PR can be reviewed and CI can re-run if needed.

Acceptance Criteria
- A Markdown report at docs/codebase-analysis.md summarizing: primary languages, top-level layout, important files, CI config, Dockerfiles, tests, key dependency files, and recommended follow-ups.
- A machine-readable JSON file at artifacts/structure.json listing discovered files and metadata (file tree snippets, counts, LOC by language, detected package manifests).
- Raw command outputs stored at artifacts/raw/* for reproducibility.

Risks and mitigations
- Missing language tools in the runner environment: steps will detect manifest presence and skip language-specific introspection if the toolchain is absent, but still record manifest files.
- Large repositories: commands will limit deep recursion depth and respect a size threshold; large outputs will be truncated with a manifest describing truncation.
