Design Spec: Analyze codebase structure for E2E Test Repo 1 (multi-agent-harness-test-repo)

Goal
- Produce a clear, actionable inventory and structural analysis of the repository identified as "E2E Test Repo 1" (repo id: ea078373-bcf2-4333-929b-e18ee8d78cb6). The output will identify languages, package managers, entry points, test layout, CI, notable config files, and potential maintenance or security hotspots.

Approach
- Because I must not clone or explore the repository in this planning phase, the design describes the exact analysis steps that implementation agents will run after this plan is accepted and merged.
- The analysis will be automated by a small set of inspection tasks that read the repository tree and key files, detect project type(s), list dependencies, locate tests and CI workflows, and surface obvious issues (large files, missing license, unpinned deps, Dockerfiles, or secret-like files).
- The output will be a concise human-readable report and a machine-readable summary (JSON) to enable downstream automation or dashboards.

Key decisions
- Non-destructive: all analysis steps will be read-only (listing and reading files) — no edits or network-side installs.
- Prioritized artifacts: README, license, language manifests (package.json, pyproject.toml, go.mod, Cargo.toml), lockfiles, Dockerfiles, CI workflows (.github/workflows), tests/e2e directories, top-level src/ or cmd/ directories.
- Produce both a high-level overview and detailed findings (file tree snapshot, dependency lists, test commands, recommended next actions).

Expected deliverables
- report/high-level.txt — plain-language summary
- report/structure.json — machine-readable structure with file types and locations
- report/details.md — annotated findings, notable concerns, and suggested follow-ups (e.g., run tests, pin deps, scan for secrets)

Acceptance criteria
- The plan must enumerate exact commands and file paths for each inspection step so implementation agents can run them without additional clarifications.
- The analysis should not modify repository files or run untrusted code from the repo.

Risks & mitigations
- Large repositories may time out: tasks will cap tree depth and file size when reading. Binary files will be skipped.
- Secrets exposure: reading files is read-only but may surface secrets; outputs will be redacted following simple heuristics (e.g., base64 strings, private key headers) and flagged for human review.
