Design Spec: Codebase Structure Analysis for E2E Test Repo 0

Goal
- Produce a clear, repeatable analysis of the repository at https://github.com/dreef3/multi-agent-harness-test-repo.git (repo id: 0b2d3c73-ba9d-4aad-9107-fbb86c7fa954). The output will be a human-readable report (docs/repo-analysis.md) and machine-readable artifacts (reports/structure.json) describing the repository layout, languages, dependency manifests, CI configuration, test suites, notable large files, and recommended next steps for contribution or automated checks.

Approach
- Use an automated script to generate the basic structural inventory, plus a small set of CLI tools (find, wc, file, cloc/scc if available) to collect metrics. Complement automated output with a short manual review checklist for items that need human interpretation (architecture descriptions in README, ambiguous or generated files, monorepo workspaces, and purpose of top-level directories).
- Produce two artifacts: (1) docs/repo-analysis.md — narrative summary and prioritized findings; (2) reports/structure.json — structured data that can be consumed by other tools or tests.

Key Decisions and Rationale
- Keep the analysis non-invasive: only read repository contents and create analysis artifacts in the repo. Do not modify existing source code.
- Target both human and machine consumers: a markdown report for maintainers and a JSON file for automation/CI checks.
- Make the analysis reproducible by storing the analyzer script in scripts/analyze_repo.sh and adding an optional GitHub Actions workflow (.github/workflows/repo-analysis.yml) that can run on-demand or on PRs.
- Focus on the most actionable items: languages and size by directory, dependency manifests and their package managers, CI workflows and test frameworks, presence of Dockerfiles and build scripts, and large or generated files.

Deliverables
- docs/repo-analysis.md — narrative report with sections: Overview, Languages & Size, Dependency Manifests, CI and Workflows, Tests & Coverage, Large Files & Generated Artifacts, Recommendations.
- reports/structure.json — structured inventory with keys: repo, top_level_dirs, files_by_type, languages, manifests, ci_files, tests, large_files.
- scripts/analyze_repo.sh — reusable script to produce both artifacts from a checked-out repository.
- .github/workflows/repo-analysis.yml — optional CI workflow to run the analysis automatically on push/PR or on-demand.

Constraints
- I did not access the repository contents while preparing this spec. The plan assumes a standard git-hosted repository layout and common manifest filenames (package.json, requirements.txt, go.mod, Cargo.toml, setup.py, pyproject.toml, pom.xml, etc.). The actual repo may vary and the analyzer will be written to detect these common patterns.

Success Criteria
- The PR containing the analysis script and report is created and passes the repository's CI checks (if any).
- The report surfaces the repository's top-level structure, languages, key manifests, and at least three actionable recommendations (e.g., add CI test run, split large files, add README sections).

Next step
- Implement the analyzer script, run it against the repository, produce the reports, and open a PR with the artifacts.

(End of spec)
