# Design Spec: Codebase Structure Analysis for E2E Test Repo

Repository: E2E Test Repo
Repository ID: 4f32690d-46cd-4399-a144-f98f1759d609
URL: https://github.com/dreef3/multi-agent-harness-test-repo.git

Goal

- Produce a concise, actionable analysis of the repository's codebase structure: top-level layout, primary languages and package managers, CI/CD configuration, test layout, notable scripts and tools, and potential points of technical debt or missing documentation. Deliver artifacts that support further automated and manual review (tree, language metrics, dependency lists, test surface, TODOs, and recommended next steps).

Approach

- Because this planning phase must not clone or inspect the repository directly, the plan below defines an automated set of tasks that an implementation agent (or CI job) will run inside a checked-out copy of the repository. Those tasks will produce a consistent set of report artifacts under a reports/ directory.

- The analysis is a mix of lightweight static discovery (file existence, directory layout, language detection, cloc for size metrics, tree for layout) and targeted manifest inspection (package.json, pyproject.toml, go.mod, Cargo.toml, pom.xml, Dockerfile, .github/workflows) plus test discovery and CI config identification. Optional deeper tasks (dependency graphs, license scan, container analysis) are included as additional steps.

Key decisions

- Focus on non-invasive, fast commands that require minimal environment setup; where language-specific commands are included they are optional and guarded (e.g., npm/pip/cargo commands run only if the corresponding manifest file exists).

- Produce machine-readable outputs (JSON where possible) and human-readable summaries (Markdown in reports/summary.md) so downstream agents or reviewers can consume results programmatically or visually.

- Include explicit list of artifacts, exact commands to run, and environment/tooling requirements so implementers can run the plan reproducibly.

Success criteria

- A reports/ directory containing at minimum: tree.txt, cloc.json, manifest-summary.md, deps-<lang>.json or txt when applicable, ci-files.md, test-summary.md, todos.md, and a top-level summary.md that synthesizes findings and recommendations.

Constraints and assumptions

- This spec assumes the implementer will run the plan in a Linux-like environment with git access to clone the repository.
- No code changes are made to the repository—analysis is read-only.

Outputs

- Primary artifacts: reports/tree.txt, reports/cloc.json, reports/manifest-summary.md, reports/dependency-<lang>.*, reports/ci-files.md, reports/test-summary.md, reports/todos.md, reports/summary.md
