Design Spec: Codebase Structure Analysis for E2E Test Repo

Goal
- Produce a concise, reproducible analysis of the repository "multi-agent-harness-test-repo" (id: 664a9911-d079-4727-8105-699a84fdfa9e) that describes its top-level layout, primary languages, build/test tooling, CI configuration, containerization, dependency manifests, notable scripts, and locations of tests and docs. Deliverables will be machine-readable and human-readable reports added to the repository.

Approach
- Implement an automated repository-scan workflow (bash + standard CLI tools) that can be run by sub-agents or CI to collect the file tree, language breakdown, dependency manifests, CI configs, Dockerfiles, test layouts, and known scripts.
- Produce outputs in a reports/ directory: reports/structure_report.md, reports/tree.txt, reports/language_summary.json, reports/dependencies.md, reports/ci_summary.md, and reports/metrics.json.
- Add lightweight analysis scripts under tools/analyze_repo.sh so maintainers can re-run the audit locally or in CI.
- Keep the analysis non-invasive (read-only): no changes to source code, only new report files and scripts.

Key Decisions
- Use only standard, widely-available CLI tools (tree, find, rg/grep, sed, awk, cloc) to maximize portability. Document alternative commands when tools are missing.
- Focus on static analysis (file inspection) rather than running builds or tests to avoid side effects.
- Store all outputs under reports/ with clear timestamps and a single summary Markdown file intended for maintainers and reviewers.
- All tasks will be self-contained and include exact commands and file paths so sub-agents can execute them without additional coordination.

Success Criteria
- A completed reports/structure_report.md describing: top-level folders, main languages with approximate LOC breakdown, package/dependency manifests, CI/Docker presence, test directories, and recommended follow-ups.
- Tools/scripts under tools/ that reproduce the reports.
- A plan and task list suitable for dispatch to sub-agents to implement the scripts and generate the reports.

Notes
- This design avoids cloning or inspecting the repository in this planning phase. The implementation plan lists explicit commands that will be run by sub-agents that do have repo access.
